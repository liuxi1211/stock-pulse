# Java 高级代码质量与设计原则

> 目标：代码不仅仅能跑，还要有设计、有扩展、有性能、有可读性，又不过度设计。
> 参考：Effective Java、阿里巴巴Java开发手册、代码整洁之道

---

## 一、核心设计原则

### 1.1 SOLID 💡 SHOULD

| 原则 | 说明 |
|-----|------|
| **SRP 单一职责** | 一个类/方法只做一件事，只有一个引起变化的原因 |
| **OCP 开闭原则** | 对扩展开放，对修改关闭（新增功能加代码，不改老代码） |
| **LSP 里氏替换** | 子类可以替换父类而不破坏程序行为 |
| **ISP 接口隔离** | 接口要小而专，客户端不应依赖它不需要的方法 |
| **DIP 依赖倒置** | 依赖抽象不依赖具体，高层模块不依赖低层模块 |

### 1.2 其他重要原则

- **DRY**：不要重复代码，相同逻辑出现 3 次以上必须抽象
- **KISS**：保持简单，能不用设计模式就不用
- **YAGNI**：你不会需要它的，不要为"将来可能"提前写代码
- **最小惊讶原则**：方法做它名字说的事情，不要有隐藏副作用

## 二、常见代码异味 ✅ MUST

遇到以下情况必须停下来思考：

| 异味 | 说明 | 解决方向 |
|-----|------|---------|
| **超长方法/类** | 方法 > 50 行，类 > 500 行 | 按职责拆分 |
| **深层嵌套** | if/for 嵌套 > 3 层 | 提前返回（卫语句）、策略模式 |
| **魔法值** | 代码中出现未定义的数字/字符串 | 定义常量或枚举 |
| **上帝类** | 一个类什么都干，几十个方法 | 按职责拆分类 |
| **过长参数** | 方法参数 > 4 个 | 用对象封装（DTO/Query） |
| **重复代码** | 同样的逻辑在多处出现 | 抽取公共方法/类 |
| **循环中 IO** | for 循环里查 DB / 调接口 / 读写文件 | 批量操作 + 内存组装 |
| **空 catch** | catch 了异常什么都不做 | 至少打日志，不能处理就上抛 |
| **注释掉的代码** | 大量注释代码留着 | 删掉，Git 会记录 |
| **滥用工具类** | 什么都往 Util 里塞 | 考虑是不是实体的实例方法，按领域拆分 |

## 三、Effective Java 精华

### 3.1 对象创建

- **多参数用 Builder**：参数 > 4 个优先用 Builder 模式（Lombok `@Builder`）
- **工具类私有构造**：防止被实例化
- **避免不必要的对象**：不在循环中创建对象，优先基本类型，正则预编译
- **try-with-resources**：IO 流、连接等资源必须用

### 3.2 类和接口

- **可访问性最小化**：字段尽量私有，内部类用 private
- **可变性最小化**：值对象（DTO/VO）尽量不可变
- **复合优先于继承**：能用组合就不要用继承，继承只在真正 is-a 时用
- **接口优于抽象类**：接口定义契约，抽象类提供骨架实现
- **用 enum 代替 int 常量**：状态、类型等必须用枚举
- **用实例域代替 ordinal**：永远不要用 `ordinal()` 获取枚举的 int 值

### 3.3 泛型与集合

- **不要用原生态类型**：`List` 必须写成 `List<String>`
- **列表优先于数组**：尽量用 List 而不是数组
- **Map 遍历用 entrySet**：比 keySet 再 get 高效
- **集合初始化指定容量**：避免频繁扩容
- **判断空用 isEmpty()**：不用 `size() == 0`

### 3.4 方法

- **检查参数有效性**：公共方法参数必须校验，失败快速
- **谨慎设计方法签名**：方法名清晰，参数 ≤ 4 个，布尔参数是坏味道
- **慎用重载**：重载是编译时决定的，容易出意外
- **返回空集合不用 null**：返回 `Collections.emptyList()` 等
- **Optional 用于返回值**：返回空是正常情况时用，容器类型不要包 Optional
- **局部变量作用域最小化**：第一次使用的地方声明

### 3.5 异常

- **异常不用来做流程控制**
- **优先使用标准异常**：`IllegalArgumentException`、`IllegalStateException` 等
- **异常信息要说清楚**：什么错了、为什么错
- **不要忽略异常**：空 catch 是犯罪

## 四、阿里巴巴 Java 开发手册精华

### 4.1 OOP 规约

- 避免通过对象引用访问静态变量/方法
- 外部方法参数不要用 null 做特殊含义
- `final` 可提高效率：类、方法、基本类型变量
- `toString` 要有实际内容，不要只打 hashCode
- 使用 `@Data` 时注意 equals/hashCode 是所有字段参与

### 4.2 集合处理

- `Arrays.asList()` 返回的内部类不能 add/remove
- 集合转数组用 `toArray(new T[0])`
- foreach 里不要 remove/add，用 Iterator 或 removeIf
- 不要在 foreach 里调用远程接口或数据库操作

### 4.3 并发处理

- 线程资源必须通过线程池提供，不允许手动 new Thread
- 线程池用 ThreadPoolExecutor，不用 Executors
- SimpleDateFormat 线程不安全，不要定义为 static
- 多资源加锁保持一致顺序，避免死锁

### 4.4 控制语句

- if/else/for/while/do 必须用大括号
- 复杂条件表达式提取变量，提高可读性
- 用卫语句减少嵌套

## 五、可扩展性设计 💡 SHOULD

### 5.1 什么时候该考虑扩展

**必须考虑：**
- 业务规则经常变化（如指标计算、策略选择）
- 第三方服务可能替换（如 Tushare → 其他数据源）
- 新增功能不应影响现有稳定功能

**不用考虑：**
- 一次性脚本/工具
- 稳定的核心领域模型
- 说不清楚什么时候会变的地方

> 💡 原则：在变化最可能发生的地方花心思设计，其他地方保持简单。

### 5.2 常用扩展手段

| 模式 | 适用场景 |
|-----|---------|
| **策略模式** | 一个行为有多种实现，需要动态切换 |
| **模板方法** | 流程固定，某几步具体实现不同 |
| **工厂模式** | 对象创建逻辑复杂，或根据参数决定类型 |
| **观察者/事件** | 一个状态变化需要通知多个对象 |

### 5.3 扩展点设计原则

1. 接口稳定，一旦确定不要轻易改
2. 入参出参完整清晰
3. 提供合理的默认实现
4. 文档完善

## 六、善用已有工具类 ✅ MUST

写代码前先想想：JDK / Spring / 已有依赖里有没有？

### 6.1 JDK 自带

| 工具类 | 常用方法 |
|-------|---------|
| `Objects` | equals, hash, requireNonNull, isNull, nonNull |
| `Collections` | emptyList, singletonList, unmodifiableList, sort |
| `Collectors` | toList, toMap, groupingBy, joining |
| `Optional` | ofNullable, orElse, orElseGet, map, flatMap |
| `java.time.*` | LocalDate, LocalDateTime, Duration, Period |

### 6.2 Spring 自带

| 工具类 | 用途 |
|-------|------|
| `StringUtils` | 字符串工具 |
| `CollectionUtils` | 集合工具 |
| `BeanUtils` | Bean 拷贝 |
| `Assert` | 参数校验 |

### 6.3 引入新依赖的原则

1. **必要性**：真的需要吗？JDK/Spring 已有功能就不要额外引入
2. **活跃度**：社区活跃吗？最近有更新吗？
3. **口碑**：Star 数、使用者多吗？有没有严重 bug
4. **体积**：依赖有多大？会不会带进来一堆传递依赖
5. **License**：许可证是否兼容项目

> 💡 一个依赖 = 一份风险。引入容易，移除难。

## 七、可读性进阶 💡 SHOULD

### 7.1 命名

- **变量**：名词短语，有意义，布尔用 is/has/can 开头
- **方法**：动词短语，准确描述做什么
- **类**：名词短语，描述职责，不描述实现
- 不要拼音，不要无意义缩写（HTTP/URL 等公认的除外）

### 7.2 函数

- 一个函数只做一件事
- 参数越少越好（≤ 4 个理想）
- 不要有副作用（要么做事要么返回值，不要二者兼有）

### 7.3 注释

**好注释：**
- 解释为什么这么做（业务背景、设计决策）
- 解释复杂算法原理
- 警示潜在的坑

**坏注释：**
- 复述代码做什么（`// 获取股票名称`）
- 注释掉的代码
- 过时的注释（比没注释更坑）

## 八、避免过度设计 💡 SHOULD

### 8.1 过度设计的信号 🚩

- 为"将来可能"写了大量代码，但说不清楚什么时候会用到
- 一个简单需求用了 3 种设计模式
- 抽象层次超过 3 层
- 新增一个简单功能要改 N 个类
- 新人需要一周才能看懂一个简单功能

### 8.2 三次原则（Rule of Three）

- 第一次：直接写
- 第二次：复制粘贴（但开始注意）
- 第三次：必须重构抽象

### 8.3 渐进式重构

- 不要一开始就想设计完美
- 代码是演化出来的，不是设计出来的
- 每一次改动都让代码比原来好一点
- 测试是重构的安全网

## 九、代码质量自检清单

提交前对照检查：

### 正确性 ✅ MUST
- [ ] 正常流程跑通了
- [ ] 边界情况处理了（空值、空集合、极值）
- [ ] 异常情况有处理

### 可读性 💡 SHOULD
- [ ] 命名清晰，不用猜
- [ ] 方法不长，职责单一
- [ ] 嵌套不超过 3 层
- [ ] 没有魔法值
- [ ] 复杂逻辑有注释说明

### 设计质量 💡 SHOULD
- [ ] 没有重复代码
- [ ] 依赖关系清晰
- [ ] 该用的模式用了，不该用的没用
- [ ] 没有过度设计

### 性能 ✅ MUST（大数据量/高频调用）
- [ ] 循环里没有 SQL/接口调用
- [ ] 批量操作真的用了批量
- [ ] 查询走索引了
- [ ] 热点数据考虑了缓存

### 安全性 ✅ MUST
- [ ] 用户输入做了校验
- [ ] 没有 SQL 注入风险
- [ ] 敏感信息没有泄露
