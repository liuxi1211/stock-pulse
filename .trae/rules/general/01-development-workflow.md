# 开发工作流规范

> 适用于 Stock 项目所有子系统的标准开发流程。

---

## 一、标准开发流程

```
需求分析 → 方案设计 → 编码实现 → 自测验证 → 代码提交 → Code Review → 合并部署
```

### 1.1 需求分析 ✅ MUST

- 明确需求背景、目标用户、验收标准
- 识别依赖关系和技术风险
- 确认工作量预估和优先级

### 1.2 方案设计 💡 SHOULD

- 复杂功能先设计后编码，输出设计思路（可选文档或注释）
- 设计需考虑：数据结构、接口定义、错误处理、性能影响
- 涉及数据库变更时，需准备 schema 变更脚本
- 涉及跨系统交互时（watcher ↔ engine），需明确接口契约

### 1.3 编码实现 ✅ MUST

- 遵循对应语言的代码风格规范
- 遵循项目的架构分层和模块划分
- 新增代码需与现有代码风格保持一致
- 敏感信息（token、密码、密钥）**不得**硬编码

### 1.4 自测验证 ✅ MUST

- 功能开发完成后必须自行验证
- 覆盖正常路径、异常路径、边界情况
- 验证数据正确性（如计算结果、数据库写入）
- 确认无控制台报错、无异常日志

### 1.5 代码提交 ✅ MUST

- 遵循 Git 提交规范（见 `02-git-commit.md`）
- 单次提交聚焦单一功能/修复
- 提交前检查：无 debug 代码、无敏感信息、注释完整

### 1.6 Code Review 💡 SHOULD

- CR 重点关注：逻辑正确性、安全性、性能、可维护性
- CR 反馈需具体，指明问题并给出改进建议
- 作者需认真回应每条评论

### 1.7 合并部署 📌 MAY

- 合并前确保所有检查通过
- 部署后验证线上功能正常

## 二、分支策略

### 2.1 分支类型 ✅ MUST

| 分支类型 | 命名格式 | 说明 |
|---------|---------|------|
| 主分支 | `main` / `master` | 稳定版本，随时可发布 |
| 功能分支 | `feature/<功能描述>` | 新功能开发 |
| 修复分支 | `fix/<问题描述>` | Bug 修复 |
| 重构分支 | `refactor/<描述>` | 代码重构，不含功能变更 |
| 文档分支 | `docs/<描述>` | 文档更新 |
| 杂项分支 | `chore/<描述>` | 构建工具、依赖升级等 |

### 2.2 分支生命周期 💡 SHOULD

- 功能分支从主分支创建
- 开发完成后提 PR/MR 合并回主分支
- 分支合并后及时删除
- 避免长期存在的功能分支

## 三、环境配置管理

### 3.1 环境划分 ✅ MUST

| 环境 | 说明 | 配置文件 |
|-----|------|---------|
| 开发环境 | 本地开发调试 | `application-dev.yml` / `.env` |
| 生产环境 | 正式运行 | `application-prod.yml` / `.env.prod` |

### 3.2 敏感信息管理 ✅ MUST

- **严禁**将敏感信息提交到版本库
- 敏感信息包括：Tushare token、数据库密码、API 密钥、TOTP secret
- 使用模板文件（如 `application-secret.properties.template`、`.env.example`）
- 真实配置文件加入 `.gitignore`

### 3.3 配置文件原则 💡 SHOULD

- 不同环境的配置通过 profile 或环境变量区分
- 默认配置偏向安全（如生产环境默认关闭调试）
- 配置项需有明确的注释说明用途

## 四、开发原则

### 4.1 KISS 原则 ✅ MUST

- 保持简单，避免过度设计
- 能用简单方案解决的，不用复杂方案
- 不引入不必要的依赖和抽象

### 4.2 YAGNI 原则 💡 SHOULD

- 不实现当前不需要的功能
- 不为"未来可能需要"预留扩展点
- 需要时再重构

### 4.3 最小改动原则 ✅ MUST

- 修复 Bug 时做最小必要改动
- 不进行无关的风格调整或重构
- 重构与功能变更分开提交

### 4.4 防御式编程 💡 SHOULD

- 对外部输入做校验（用户输入、API 调用、第三方数据）
- 对可能失败的操作做异常处理
- 关键路径加日志，便于排查问题

## 五、项目特定约束

### 5.1 双系统协作约束 ✅ MUST

- watcher（Java）是数据唯一来源，独占 SQLite 读写
- engine（Python）不直接读写数据库，所有数据通过 HTTP API 获取/返回
- 跨系统接口变更需同步更新双方代码和文档

### 5.2 数据准确性优先 ✅ MUST

- 量化系统中，数据准确性是第一位的
- 涉及计算的代码必须有数据校验
- 复权、聚合等操作需明确算法并加注释

### 5.3 性能与资源 💡 SHOULD

- 数据量大的操作考虑批量处理
- 避免 O(N²) 及以上复杂度的算法处理大量数据
- 合理使用缓存（Caffeine / 内存缓存）
