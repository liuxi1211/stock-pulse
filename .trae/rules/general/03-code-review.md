# 代码审查规范

> 适用于 Stock 项目所有子系统的代码审查（Code Review）流程与标准。

---

## 一、CR 流程

### 1.1 提交流程 ✅ MUST

```
开发者自测通过 → 提交代码 → 创建 PR/MR → 指定审查人 → 审查反馈 → 修改优化 → 审查通过 → 合并
```

### 1.2 审查者选择 💡 SHOULD

- 至少 1 名熟悉相关模块的开发者
- 涉及核心逻辑（如 K线计算、策略回测）需 2 人以上审查
- 涉及安全相关代码需安全意识强的开发者审查

---

## 二、CR 检查清单

### 2.1 功能正确性 ✅ MUST

- [ ] 需求理解是否正确
- [ ] 业务逻辑是否正确实现
- [ ] 边界条件是否处理（空值、极值、异常输入）
- [ ] 错误处理是否完善
- [ ] 数据计算是否准确（尤其量化相关代码）

### 2.2 代码质量 💡 SHOULD

- [ ] 命名是否清晰、有意义
- [ ] 函数/类职责是否单一
- [ ] 是否存在重复代码
- [ ] 注释是否充分（复杂逻辑必须有注释）
- [ ] 是否符合项目代码风格规范

### 2.3 性能 ✅ MUST（涉及大数据量/高频调用）

- [ ] 是否有 O(N²) 及以上复杂度的操作
- [ ] 数据库查询是否合理（避免 N+1、全表扫描）
- [ ] 是否存在内存泄漏风险
- [ ] 缓存使用是否合理
- [ ] 批量操作是否采用批量处理

### 2.4 安全性 ✅ MUST

- [ ] 输入是否校验（SQL 注入、XSS、命令注入）
- [ ] 敏感信息是否泄露（日志、错误信息、前端）
- [ ] 认证授权是否正确
- [ ] 是否存在越权访问风险
- [ ] 第三方依赖是否有已知漏洞

### 2.5 可维护性 💡 SHOULD

- [ ] 代码是否容易理解
- [ ] 是否过度设计
- [ ] 依赖关系是否清晰
- [ ] 配置是否合理
- [ ] 日志是否充分（问题可追踪）

### 2.6 测试 💡 SHOULD

- [ ] 核心逻辑是否有单元测试
- [ ] 测试用例是否覆盖主要场景
- [ ] 测试是否可重复执行

## 三、CR 反馈标准

### 3.1 反馈分级 ✅ MUST

| 级别 | 标记 | 说明 | 是否必须修改 |
|-----|------|------|-------------|
| **阻断** | 🔴 BLOCKER | 严重问题，必须修复才能合并 | ✅ 是 |
| **重要** | 🟡 IMPORTANT | 建议修复，有充分理由可讨论 | 💡 建议修改 |
| **建议** | 💡 SUGGESTION | 改进建议，可选采纳 | 📌 可选 |
| **提问** | ❓ QUESTION | 理解疑问，需作者解答 | — |

### 3.2 反馈原则 💡 SHOULD

1. **对事不对人**：评论代码，不评论人
2. **具体明确**：指明具体位置和问题，不泛泛而谈
3. **给出建议**：不仅指出问题，尽量给出改进方案
4. **解释原因**：说明为什么是问题，帮助作者理解
5. **尊重差异**：风格类问题如果不影响质量，可以妥协

### 3.3 好的反馈示例

```
🔴 BLOCKER: 这里存在 SQL 注入风险，userInput 直接拼接到 SQL 中了
建议使用 MyBatis-Plus 的参数绑定或 QueryWrapper。

位置：UserService.java 第 45 行
```

```
💡 SUGGESTION: 这个 for 循环可以用 stream 改写，更简洁
可以考虑：list.stream().map(...).collect(...)
```

### 3.4 不好的反馈示例 ❌

```
这段代码写得不好。
（太模糊，没说哪里不好、为什么不好、怎么改）
```

```
你怎么又写出这种代码？
（人身攻击，对人不对事）
```

## 四、作者回应规范

### 4.1 回应原则 ✅ MUST

- 认真阅读每条评论，逐一回应
- 同意修改的：修改后回复「已修改」或直接 resolve
- 不同意的：说明理由，讨论达成共识
- 疑问类：解答清楚后 resolve

### 4.2 修改提交 💡 SHOULD

- 针对审查意见的修改，单独提交或追加到原提交
- 提交信息可注明 `address review comments`
- 避免 force push（除非审查者明确要求）

## 五、项目特定审查重点

### 5.1 量化计算相关 🔴 重点

- 计算公式是否正确
- 复权处理是否正确
- 浮点精度是否考虑
- 边界值（停牌、涨跌停、退市）是否处理
- 数据对齐是否正确

### 5.2 数据获取相关 🔴 重点

- Tushare 限流是否处理
- 接口失败是否重试
- 数据异常是否校验
- 增量更新是否正确

### 5.3 安全相关 🔴 重点

- 用户输入是否校验
- 认证授权是否正确
- 敏感信息（token、密码）是否泄露
- SQL 注入风险

### 5.4 性能相关 🟡 关注

- 大量数据查询是否分页/批量
- K线计算是否缓存
- 定时任务是否避免重复执行
- 数据库查询是否走索引
