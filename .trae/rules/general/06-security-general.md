# 安全规范通用原则

> 适用于 Stock 项目所有子系统的安全开发通用指导原则。
> 具体语言/框架的安全规则见对应语言的安全规范。

---

## 一、安全开发核心原则

### 1.1 最小权限原则 ✅ MUST

- 每个组件/用户只拥有完成其功能所需的最小权限
- 数据库账号只授予必要的权限
- API 接口只暴露必要的数据
- 文件系统访问限制在必要范围内

### 1.2 纵深防御原则 💡 SHOULD

- 不要依赖单一防护措施
- 多层防御，一层被突破还有下一层
- 示例：输入校验 + 参数化查询 + 权限检查

### 1.3 失败安全（Fail-Safe）✅ MUST

- 系统失败时默认处于安全状态
- 认证失败 → 拒绝访问（而不是允许）
- 配置加载失败 → 使用安全默认值
- 异常情况 → 不泄露敏感信息

### 1.4 不要信任外部输入 ✅ MUST

- 所有外部输入都是不可信的
- 外部输入包括：用户输入、API 请求、第三方数据、文件、环境变量
- 必须进行校验、过滤、转义

### 1.5 安全默认值 💡 SHOULD

- 默认配置应该是安全的
- 调试功能默认关闭
- 不安全的功能默认禁用
- 需要用户明确开启才能使用

## 二、OWASP Top 10 关注

### 2.1 注入攻击（Injection）🔴 高危

**SQL 注入、命令注入、代码注入等**

- ✅ MUST 使用参数化查询/预编译语句
- ✅ MUST 对特殊字符进行转义
- ✅ MUST 避免拼接 SQL / 命令
- 💡 SHOULD 使用 ORM 框架（MyBatis-Plus / SQLAlchemy）

### 2.2 认证与授权失效 🔴 高危

**弱密码、会话管理不当、越权访问等**

- ✅ MUST 使用强密码策略
- ✅ MUST 密码使用哈希存储（BCrypt / Argon2）
- ✅ MUST 会话安全管理（过期、失效、HTTPS）
- ✅ MUST 接口做权限校验，防止越权
- 💡 SHOULD 启用 MFA（多因素认证，如 TOTP）

### 2.3 敏感数据泄露 🔴 高危

**密码、密钥、个人信息泄露**

- ✅ MUST 敏感数据加密存储
- ✅ MUST 敏感数据不在日志中输出
- ✅ MUST 错误信息不泄露系统细节
- ✅ MUST 传输层使用 HTTPS
- 💡 SHOULD 敏感数据脱敏显示

### 2.4 不安全的设计 🟡 中危

**架构设计缺陷、业务逻辑漏洞**

- 💡 SHOULD 安全左移，设计阶段考虑安全
- 💡 SHOULD 威胁建模
- 💡 SHOULD 最小化攻击面

### 2.5 安全配置错误 🟡 中危

**默认配置、调试信息泄露、不必要的功能**

- ✅ MUST 生产环境关闭调试模式
- ✅ MUST 修改默认账号密码
- ✅ MUST 移除不必要的功能和服务
- 💡 SHOULD 定期审计配置

### 2.6 易受攻击的组件 🟡 中危

**有漏洞的第三方依赖**

- 💡 SHOULD 定期扫描依赖漏洞
- 💡 SHOULD 及时更新有漏洞的依赖
- 💡 SHOULD 最小化依赖数量
- 💡 SHOULD 使用可信的依赖源

### 2.7 跨站脚本（XSS）🔴 高危

**前端注入恶意脚本**

- ✅ MUST 对用户输入进行转义
- ✅ MUST 输出时进行 HTML 转义
- 💡 SHOULD 使用 CSP（内容安全策略）
- 💡 SHOULD 使用框架自动转义（Thymeleaf 默认转义）

### 2.8 跨站请求伪造（CSRF）🟡 中危

**诱导用户执行非预期操作**

- ✅ MUST 关键操作使用 CSRF Token
- 💡 SHOULD 验证 Referer
- 💡 SHOULD 使用 SameSite Cookie

### 2.9 安全映射与监控不足 📌 低危

**缺少安全监控、审计日志不足**

- 💡 SHOULD 记录安全相关日志
- 💡 SHOULD 监控异常行为
- 💡 SHOULD 保留审计痕迹

### 2.10 服务端请求伪造（SSRF）🟡 中危

**服务端发起恶意请求**

- ✅ MUST 限制服务端请求的目标地址
- 💡 SHOULD 使用白名单
- 💡 SHOULD 禁止访问内网敏感地址

## 三、敏感数据处理

### 3.1 敏感数据分类 ✅ MUST

| 类别 | 示例 | 保护要求 |
|-----|------|---------|
| **机密** | Tushare token、API 密钥、数据库密码 | 加密存储，禁止日志输出 |
| **敏感** | 用户密码、TOTP secret | 哈希存储，禁止明文 |
| **个人信息** | 用户名、邮箱 | 限制访问，日志脱敏 |
| **内部** | 内部接口地址、系统架构 | 不对外暴露 |

### 3.2 密码存储 ✅ MUST

- 禁止明文存储密码
- 使用强哈希算法：BCrypt、Argon2、PBKDF2
- 加盐（salt），每个密码单独加盐
- 足够的迭代次数/计算成本

### 3.3 日志安全 ✅ MUST

- 日志中禁止输出密码、token、密钥
- 敏感信息（如手机号、邮箱）脱敏输出
- 错误信息不泄露堆栈、数据库结构等内部信息
- 生产环境不输出 debug 级别的日志

### 3.4 配置文件安全 ✅ MUST

- 敏感配置不提交到版本库
- 使用环境变量或配置文件管理
- 配置文件设置合适的权限
- 模板文件不包含真实敏感值

## 四、输入验证

### 4.1 验证原则 ✅ MUST

- **所有输入都需要验证**：用户输入、API 参数、第三方数据
- **白名单验证**：只允许已知合法的输入，拒绝其他一切
- **标准化**：验证前先标准化输入（编码、大小写等）
- **验证位置**：客户端验证 + 服务端验证（服务端是最后防线）

### 4.2 验证内容

| 类型 | 验证内容 |
|-----|---------|
| **类型** | 数据类型是否正确（数字、字符串、日期等） |
| **长度** | 长度是否在允许范围内 |
| **格式** | 格式是否符合要求（邮箱、手机号、股票代码） |
| **范围** | 数值是否在合法范围内 |
| **业务规则** | 是否符合业务逻辑约束 |

### 4.3 股票代码校验 💡 SHOULD

本项目中股票代码有两种格式，需校验：
- `ts_code`：如 `000001.SZ`、`600000.SH`（6位数字 + . + 交易所后缀）
- `symbol`：如 `000001`、`600000`（纯 6 位数字）

## 五、项目特定安全关注点

### 5.1 Tushare Token 安全 🔴 重点

- Token 不得硬编码在代码中
- Token 不得提交到 Git
- Token 不得输出到日志
- Token 配置文件加入 .gitignore
- 提供模板文件，不包含真实值

### 5.2 用户认证与授权 🔴 重点

- 密码使用 BCrypt 哈希存储
- Session 安全管理
- TOTP 两步验证
- 管理员接口做权限校验（@RequireAdmin）
- 防止越权访问（只能操作自己的自选股）

### 5.3 数据库安全 🔴 重点

- 使用 MyBatis-Plus 参数化查询，防止 SQL 注入
- SQLite 数据库文件设置合适的权限
- 数据库备份文件妥善保存
- 不对外暴露数据库端口

### 5.4 前端安全 🟡 关注

- Thymeleaf 默认转义，防止 XSS
- 用户输入输出都做转义
- 表单做 CSRF 防护
- 敏感操作需二次确认

### 5.5 Python 服务安全 🟡 关注

- 输入使用 Pydantic 验证
- 不暴露不必要的端口
- 限制请求大小和频率
- 不执行用户提供的代码

## 六、安全检查清单

开发完成后，对照以下清单检查：

- [ ] 所有外部输入都做了校验
- [ ] 没有 SQL 注入风险
- [ ] 没有 XSS 风险
- [ ] 密码/Token 等敏感信息安全存储
- [ ] 日志中没有敏感信息
- [ ] 认证授权正确实现
- [ ] 没有越权访问漏洞
- [ ] 错误信息不泄露内部细节
- [ ] 依赖没有已知高危漏洞
- [ ] 配置文件中没有硬编码的敏感信息
