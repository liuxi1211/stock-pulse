# 性能优化通用原则

> 适用于 Stock 项目所有子系统的性能优化通用指导原则。
> 具体语言/框架的性能优化规则见对应语言的性能规范。

---

## 一、性能优化核心原则

### 1.1 先测量后优化 ✅ MUST

**永远不要凭感觉优化！**

- 优化前必须先 profiling，找到真正的性能瓶颈
- 用数据说话，不做假设
- 优化后再次测量，确认效果

### 1.2 避免过早优化 💡 SHOULD

> "Premature optimization is the root of all evil." — Donald Knuth

- 先让代码正确运行，再考虑优化
- 优先保证代码的可读性和可维护性
- 只有在确有性能问题时才优化

### 1.3 二八定律（帕累托法则）✅ MUST

- 80% 的性能问题集中在 20% 的代码上
- 优化热点代码，收益最大
- 不要在冷路径上浪费时间

### 1.4 权衡取舍 💡 SHOULD

性能优化通常有代价，需要权衡：

| 优化方向 | 可能牺牲 |
|---------|---------|
| 空间换时间 | 内存占用增加 |
| 时间换空间 | 执行时间增加 |
| 缓存 | 内存占用 + 数据一致性 |
| 并发 | 复杂度 + 线程安全 |
| 批量处理 | 单次延迟增加 |

## 二、性能指标定义

### 2.1 常见性能指标

| 指标 | 说明 | 单位 |
|-----|------|------|
| **响应时间** | 请求从发收到响应的时间 | ms / s |
| **吞吐量** | 单位时间内处理的请求数 | QPS / TPS |
| **并发数** | 同时处理的请求数 | 个 |
| **CPU 使用率** | CPU 占用百分比 | % |
| **内存使用量** | 占用的内存大小 | MB / GB |
| **IO 等待** | 磁盘/网络 IO 等待时间 | ms |

### 2.2 性能目标 💡 SHOULD

优化前先明确目标：

- 接口响应时间：P95 < 500ms，P99 < 1s
- 页面加载时间：首屏 < 2s
- 数据处理：日更任务 < 30 分钟
- 内存占用：不超过可用内存的 70%

## 四、常见性能优化手段

### 4.1 算法与数据结构 ✅ MUST

- 选择合适的数据结构（数组 vs 链表 vs 哈希表 vs 树）
- 避免 O(N²) 及以上复杂度处理大量数据
- 善用排序、二分查找等优化手段
- 预处理/预计算，避免重复计算

### 4.2 缓存 💡 SHOULD

**适用场景**：
- 读取频繁、写入较少的数据
- 计算成本高的结果
- 允许一定程度的数据不一致

**缓存策略**：
- LRU（最近最少使用）
- TTL（过期时间）
- 写穿透 / 写回

### 4.3 批量处理 💡 SHOULD

- 数据库操作尽量批量执行
- 网络请求尽量合并
- 循环中避免重复 IO

### 4.4 并发与并行 📌 MAY

- CPU 密集型：多线程/多进程并行计算
- IO 密集型：异步/非阻塞 IO
- 注意线程安全和数据一致性

### 4.5 数据库优化 ✅ MUST

- 合理设计索引
- 避免全表扫描
- 避免 N+1 查询
- 分页查询大数据集
- 优化 SQL 写法

### 4.6 内存优化 💡 SHOULD

- 及时释放不再使用的对象
- 避免内存泄漏
- 大数据量处理考虑流式处理
- 对象池/连接池复用

## 五、项目特定性能关注点

### 5.1 数据获取与存储 🔴 重点

- Tushare 接口调用注意限流，避免触发频率限制
- 批量数据入库使用批量插入/替换，不用逐条插入
- 日线数据量大，查询必须走索引
- 合理设置缓存，减少重复查询

### 5.2 K 线计算 🔴 重点

- K 线计算结果缓存，避免重复计算
- 前复权计算注意数值精度，使用 BigDecimal（Java）
- 周/月 K 线聚合使用高效算法
- 单只股票 K 线计算应 < 10ms

### 5.3 策略回测 🟡 关注

- 回测引擎使用高性能实现（AKQuant Rust 引擎）
- 大数据量回测考虑内存优化
- 向量化运算替代循环
- 回测进度可监控、可中断

### 5.4 前端展示 🟡 关注

- K 线图大数据量使用 ECharts 大数据模式
- 分页加载，不一次性加载所有数据
- 图表数据按需加载
- 避免频繁重渲染

## 六、性能优化流程

```
1. 发现问题
   └─ 用户反馈 / 监控告警 / 主动发现

2. 定位瓶颈
   └─ 使用 profiling 工具收集数据
   └─ 分析热点函数/慢查询/内存占用

3. 分析原因
   └─ 是算法问题？数据结构问题？IO 问题？

4. 制定优化方案
   └─ 评估优化收益和成本
   └─ 选择最优方案

5. 实施优化
   └─ 小步迭代，每次优化一个点
   └─ 保持代码可读性

6. 验证效果
   └─ 性能测试对比
   └─ 确认无功能回归

7. 监控持续观察
   └─ 线上性能监控
   └─ 防止性能退化
```
