## 二、Bug 类问题详细分析与修复方案

### 问题 1：交易日历 SZSE 0 条记录（根因：拉取时 exchange 传 null）

**现象**：`exchange_coverage` 检测显示 SSE 10961 条，SZSE 0 条。

**根因定位**：
- [DataInitServiceImpl.java](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/DataInitServiceImpl.java#L180-L184) 第 181 行：
  ```java
  case TRADE_CAL -> {
      tradeCalService.fetchAndSaveTradeCal(null, fullStart, today);
  ```
  调用 `fetchAndSaveTradeCal` 时第一个参数 `exchange` 传了 `null`。
- Tushare 的 `trade_cal` 接口要求 `exchange` 参数（SSE/SZSE），传 null 时默认只返回 SSE。
- 项目规范要求「交易日历数据按交易所（SSE/SZSE）分组，防止数据混淆」，但当前全量/增量拉取都只拉了一个交易所。

**修复方案**：
1. 在 `DataInitServiceImpl.executeSingleStep` 的 `TRADE_CAL` 分支中，分别对 `SSE` 和 `SZSE` 调用一次 `fetchAndSaveTradeCal`。
2. 同步检查 `DailyUpdateTask` 等其他调用点，确保也是双交易所拉取。

**涉及文件**：
- `stock-watcher/src/main/java/com/arthur/stock/service/impl/DataInitServiceImpl.java`
- `stock-watcher/src/main/java/com/arthur/stock/task/DailyUpdateTask.java`（需检查）

### 问题 2：index_daily 空表时 price_logic 误报"通过"

**现象**：`index_daily` 表为空（0 条记录），但 `price_logic` 检测显示「通过，最近 30 天无异常」。

**根因定位**：
- [IndexDailyServiceImpl.java](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/IndexDailyServiceImpl.java#L102-L110) 第 103 行：
  ```java
  int anomalies = indexDailyMapper.countPriceAnomalies(thirtyDaysAgo);
  items.add(DataCheckItem.builder()
          .name("price_logic")
          .passed(anomalies == 0)
  ```
- 空表时 `countPriceAnomalies` 返回 0，被误判为"通过"。

**PRD 规定**（§4.5 空表兜底）：空表检测由框架层统一处理，业务层不做特殊判断。但当前框架层实现了 `empty` 检测追加，业务层检测项仍会执行并误报通过。

**修复方案**：
- 检查 `totalRows == 0` 或最新日期为空时，该检测项 message 改为"表为空，跳过检测"，`passed=true`（不影响 ERROR 状态，因为外层已有 empty ERROR）。

**涉及文件**：
- `stock-watcher/src/main/java/com/arthur/stock/service/impl/IndexDailyServiceImpl.java`

### 问题 3：income 空表时 net_income_anomaly 误报"通过"

**现象**：`income` 表为空（0 条记录），但 `net_income_anomaly` 检测显示「通过，近 3 季度无净利润超营收 10 倍异常」。

**根因定位**：
- [IncomeServiceImpl.java](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/IncomeServiceImpl.java#L217-L227) 第 219 行：
  ```java
  int exceedsCount = incomeMapper.countNetIncomeExceedsRevenue(nineMonthsAgo);
  items.add(DataCheckItem.builder()
          .name("net_income_anomaly")
          .passed(exceedsCount == 0)
  ```
- 空表 SQL COUNT 返回 0 → 误判通过。

**修复方案**：
- 检查 `totalRows == 0` 或 `maxEndDate == null` 时，message 改为"无数据，跳过检测"。

**涉及文件**：
- `stock-watcher/src/main/java/com/arthur/stock/service/impl/IncomeServiceImpl.java`

## 三、缺失校验器补全方案（20 张表，严格对照 PRD §4.3）

### 总体设计

参考 PRD §4.3 各表校验项明细，为 20 张缺失的表实现 `DataCheckable` 接口。

**通用检测项（由 DataGovernanceServiceImpl 外层统一追加，无需业务层实现）**：
- `empty`：空表检测（ERROR）— 框架层已实现
- `row_delta`：行数变动检测（WARN）— 框架层已实现

**以下是各表需要在 `checkData()` 中实现的检测项（严格对照 PRD §4.3）**：

### 3.6 每个校验器的实现内容

每个 Service 需要：
1. `implements DataCheckable`
2. 实现 `getTableCode()` 方法，返回对应的 `InitStep.xxx.getCode()`
3. 实现 `checkData()` 方法，包含：
   - `totalRows`：总行数
   - `latestDate`：最新日期（trade_date / ann_date / 其他主日期字段）
   - 按上表设计的 2~4 个检测项
4. 在对应的 Mapper 接口中添加所需查询方法
5. 在 Mapper XML 中补充对应 SQL（MySQL/SQLite 双方言兼容）

## 五、风险与注意事项

- **问题 1 修复后**：需要重新全量拉取 trade_cal 才能补全 SZSE 数据，但代码修复本身不影响已有 SSE 数据。
- **问题 2、3 的修复**：仅影响检测消息展示，不改变表数据，风险极低。
- **新增 20 个校验器**：
  - 工作量较大，但每个校验器模式统一（参考 IncomeServiceImpl / IndexDailyServiceImpl 模板）
  - 仅新增检测能力，不修改现有业务逻辑
  - Mapper 新增的都是只读查询，无写入风险
  - SQL 需注意 MySQL/SQLite 双方言兼容（参考现有 Mapper 写法）
- 所有修改均在 stock-watcher 模块，不影响 stock-engine。
- 严格对照 PRD §4.3 的检测项设计实现，不随意增减检测项。
