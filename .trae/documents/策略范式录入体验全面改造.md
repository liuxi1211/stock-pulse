# 策略范式录入体验全面改造

> 目标：消除用户在策略编辑器里"随便录入、填了不生效"的困惑，把择时/轮动两种范式的前端呈现彻底分家，并在后端补齐对"范式下不该出现的字段"的硬校验。

---

## 一、背景与现状

### 1.1 现状（已核实）

**前端**（`stock-watcher/.../templates/quant/strategies/editor.html` + `static/js/strategy-editor.js`）：
- 已有范式切换按钮（择时/轮动），通过 `data-paradigm` 属性控制 Tab 显隐。
- **但「选股范围」Tab 标记为 `data-paradigm="both"`，两种范式共用**，里面塞了 5 个混合 fieldset：universe 下拉、top_n、手动股票池、**选股条件树**、ranking、filters。
- 择时范式下：universe 强制 manual（已做），但**选股条件树、ranking、top_n 仍可填写**——填了不生效，产生困惑。
- 轮动范式下：手动股票池硬上限 50，与"全池过滤"语义并存，用户搞不清该用 manual 还是 csi300。

**后端**（`stock-engine/services/strategy/`）：
- `validator.py:_validate_signals_universe` 校验 signals 范式 universe=manual 且 ≤10 只。
- **但完全没有校验"signals 范式下 screen_config.conditions / ranking / top_n 是否为空"**——这些字段填了会被静默忽略（compiler 不消费）。
- `compiler.py` signals 路径只读 `signals.buy/sell`，不读 screen_config.conditions。

### 1.2 核心痛点

1. 择时范式下选股条件树可见可填，但回测不执行 → **用户疑惑**
2. 后端无任何拦截 → 数据层无保障
3. 前后端规则不对齐 → 前端放行 / 后端静默忽略

---

## 二、改造决策（已确认）

| 决策点 | 选择 |
|---|---|
| 择时范式下选股条件可见性 | **直接隐藏**（不让填） |
| 选股范围 Tab 拆分方式 | **拆成两个独立 Tab**，各带 `data-paradigm` |
| 后端处理 | **新增禁用错误码**（升级为错误，阻断保存） |
| 交付重点 | 消除困惑(主) + 强化校验 + 提升效率 + 前后端对齐 |

---

## 三、详细改造方案

### 3.1 后端改造（stock-engine）

#### 3.1.1 新增错误码 `SIGNALS_SCREEN_CONFIG_FORBIDDEN`

**文件**：`stock-engine/services/strategy/errors.py`

在 `SIGNALS_UNIVERSE_TOO_LARGE`（L53-L57）之后新增：

```python
SIGNALS_SCREEN_CONFIG_FORBIDDEN = (
    "SIGNALS_SCREEN_CONFIG_FORBIDDEN",
    "signals（择时）范式下，screen_config 仅允许 universe=manual + stocks 标的列表；"
    "不允许填写选股条件(conditions)、排序(ranking)、Top N(top_n)、静态过滤(filters)。"
    "如需截面选股/排序，请改用 rebalance（轮动）范式。",
)
```

#### 3.1.2 在 validator 新增校验方法

**文件**：`stock-engine/services/strategy/validator.py`

1. 在 `validate()` 方法（L86-L106）第 4 步 `_validate_signals_universe` 调用之后，新增第 5 步：
   ```python
   # 5. signals 范式下 screen_config 仅允许 universe+stocks，禁止 conditions/ranking/top_n/filters
   self._validate_signals_screen_fields(config, errors)
   ```

2. 新增 `_validate_signals_screen_fields` 方法（紧跟 `_validate_signals_universe` 之后）：
   ```python
   def _validate_signals_screen_fields(
       self, config: StrategyConfigModel, errors: List[StrategyValidationError]
   ) -> None:
       """signals（择时）范式下 screen_config 的字段禁用约束。

       signals 范式只消费 screen_config.universe + stocks（圈定回测 symbols）；
       conditions / ranking / top_n / filters 即使填写也不会被 compiler 执行，
       属于"静默忽略"。为防止用户误填产生困惑，升级为错误。
       """
       tc = config.trading_config
       if tc is None or tc.signals is None:
           return
       sc = config.screen_config
       if sc is None:
           return  # screen_config 缺失已由 _validate_signals_universe 处理

       forbidden_paths = []
       if sc.conditions is not None:
           forbidden_paths.append(("screen_config.conditions", "选股条件"))
       if sc.ranking is not None:
           forbidden_paths.append(("screen_config.ranking", "排序规则"))
       if sc.top_n is not None:
           forbidden_paths.append(("screen_config.top_n", "Top N"))
       if sc.filters is not None:
           forbidden_paths.append(("screen_config.filters", "静态过滤"))

       if forbidden_paths:
           code, msg = ErrorCode.SIGNALS_SCREEN_CONFIG_FORBIDDEN
           for path, label in forbidden_paths:
               errors.append(
                   StrategyValidationError(path=path, code=code, message=msg)
               )
   ```

#### 3.1.3 更新测试

**文件**：`stock-engine/tests/services/strategy/test_validator.py`

新增测试用例：
- `test_signals_with_screen_conditions_rejected`：signals 范式 + screen_config.conditions 非空 → 期望 `SIGNALS_SCREEN_CONFIG_FORBIDDEN`
- `test_signals_with_ranking_rejected`：signals 范式 + screen_config.ranking 非空 → 期望错误
- `test_signals_with_top_n_rejected`：signals 范式 + screen_config.top_n 非空 → 期望错误
- `test_signals_with_only_universe_stocks_passes`：signals 范式 + 仅 universe+stocks → 通过

**文件**：`stock-engine/tests/services/strategy/test_api.py`（如存在）

补充端到端校验测试。

#### 3.2.2 调整 JS 联动逻辑（strategy-editor.js）

**文件**：`stock-watcher/src/main/resources/static/js/strategy-editor.js`

**改造点 1：`ERROR_TAB_MAP`**（L85-L95）

新增/调整映射，让后端错误能定位到正确的 Tab：
```javascript
ERROR_TAB_MAP: [
    { prefix: 'name',                              tab: 'tab-basic' },
    { prefix: 'screen_config.stocks',              tab: 'tab-pool' },      // 择时股票池
    { prefix: 'screen_config.universe',            tab: 'tab-screen' },    // 轮动选股(默认)
    { prefix: 'screen_config.conditions',          tab: 'tab-screen' },
    { prefix: 'screen_config.ranking',             tab: 'tab-screen' },
    { prefix: 'screen_config.top_n',               tab: 'tab-screen' },
    { prefix: 'screen_config.filters',             tab: 'tab-screen' },
    { prefix: 'screen_config',                     tab: 'tab-screen' },    // 兜底
    // ... 其余不变
],
```

> ⚠️ 由于 `screen_config.stocks` 在两种范式都可能出现，但择时范式下 stocks 错误应定位到 `tab-pool`。需要在 `markErrorTabs` 里结合当前 paradigm 判断（见改造点 3）。

**改造点 2：`applyParadigm`**（L1348-L1386）

原有的 Tab 显隐逻辑已通过 `data-paradigm` 自动处理（拆 Tab 后两个新 Tab 各带范式标记），无需大改。但要确保切换范式时被隐藏 Tab 的 active 态正确切回 basic。

**改造点 3：`collectStateFromForm`**（约 L918-L980）

择时范式收集时，**主动清空** screen_config.conditions/ranking/top_n/filters（与后端校验对齐，防止历史脏数据）：
```javascript
if (this.paradigm === 'signals') {
    // 择时范式：screen_config 只保留 universe + stocks
    s.screen_config = s.screen_config || {};
    s.screen_config.universe = 'manual';
    s.screen_config.stocks = this._manualStocks.slice();
    delete s.screen_config.conditions;
    delete s.screen_config.ranking;
    delete s.screen_config.top_n;
    delete s.screen_config.filters;
} else {
    // 轮动范式：正常收集全部字段
    // ... 现有逻辑
}
```

**改造点 4：范式切换时的字段隔离**

`setParadigm` 触发时，除了现有联动，额外做：
- 切到 signals：清空轮动相关 DOM 状态（避免残留），universe 强制 manual
- 切到 rebalance：保留 universe 下拉可选

**改造点 5：`markErrorTabs` 增强**

在按 path 前缀匹配 Tab 时，若 path 是 `screen_config.stocks`，根据 `this.paradigm` 决定定位到 `tab-pool`（signals）还是 `tab-screen`（rebalance）。

### 3.3 模板与数据兼容

#### 3.3.1 现有策略数据迁移

**问题**：数据库里已有的 signals 范式策略，如果 config_json 里带了 conditions/ranking（历史录入），改造后会被后端校验拦截，无法保存。

**处理方案**：
- 不做数据迁移脚本（watcher 侧）。
- 编辑老策略时，前端 `collectStateFromForm` 会主动清空这些字段，保存后自然净化。
- 如果老策略纯查看不编辑，后端 validate 接口会返回错误，但**只在校验时触发，不影响已有策略的回测执行**（回测走 compiler，compiler 仍静默忽略）。

#### 3.3.2 内置模板核对

**文件**：`stock-watcher/src/main/resources/strategies/templates/*.json`

核对 5 个内置模板（dual_ma/macd_short/volume_price 是 signals；low_pe_value/small_cap 是 rebalance）：
- signals 模板：确认 `screen_config` 里只有 universe+stocks，无 conditions/ranking/top_n/filters。
- 若有不合规字段，清理掉。

## 七、风险与假设

1. **假设**：watcher 侧无独立的策略校验逻辑（校验全在 engine），因此后端改造只需动 engine。
2. **风险**：已有 signals 策略若 config_json 带 conditions，编辑时会被校验拦截。**缓解**：前端 collectStateFromForm 主动清空，保存即净化。
3. **风险**：Tab 拆分可能影响版本对比页面（strategy-versions.js）的 config 展示。**缓解**：版本对比是只读 JSON diff，不受 Tab 拆分影响，但需回归测试。
4. **不做**：不引入前端框架（仍是原生 JS + Bootstrap），不重构条件树组件。
