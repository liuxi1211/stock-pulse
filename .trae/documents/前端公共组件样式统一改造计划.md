# 前端公共组件样式统一改造计划

## 一、问题核查结论

经过对 `stock-watcher` 前端项目的全面核查，确认存在**公共组件样式不统一**的问题，具体表现如下：

### 1.1 CSS 分层混乱，职责不清

| 文件 | 应有职责 | 实际情况 | 问题 |
|---|---|---|---|
| `theme.css` | 仅主题 Design Tokens | ✅ 符合预期 | 三套主题（azure/mist/cyber）变量完整 |
| `components.css` | 公共组件样式 | 有基础，但不完整 | 部分组件散落在其他文件 |
| `custom.css` | 工具类/辅助类 | 混入了大量组件样式 | 与 components.css 职责重叠 |
| 页面级 CSS | 页面专属样式 | 重复定义全局组件 | 破坏了样式统一性 |

### 1.2 重复定义，多套实现不一致

## 二、改造目标

1. **统一组件体系**：所有常用组件在 `components.css` 中有且仅有一套标准实现，三套主题自动适配
2. **消除重复定义**：删除页面级 CSS 中重复的组件样式，复用全局类
3. **补全组件库**：补齐常用组件（印章按钮、面包屑、按钮组、校验状态等）
4. **消除硬编码**：所有颜色走 `var()`，透明度用 `color-mix()`
5. **明确分层职责**：
   - `theme.css` → 纯 Design Tokens（变量）
   - `components.css` → 公共组件（按钮、表单、卡片、表格、模态框…）
   - `custom.css` → 纯工具类/辅助类（间距、文字、flex 快捷类等）
   - 页面 CSS → 仅页面专属布局和业务组件

## 七、质量现状评估（改造前基线）

### 7.1 通用组件质量现状

经过对 `components.css` / `custom.css` / `theme.css` 的全面核查，当前状态：

| 维度 | 评估 | 说明 |
|---|---|---|
| **主题 Token 体系** | ✅ 完善 | 三套主题（azure/mist/cyber）的颜色/阴影/渐变/模糊 token 完整，所有组件都基于 var()，无主题遗漏 |
| **按钮体系** | ⚠️ 基本够用但有缺口 | 已有 primary/secondary/outline-primary/outline-secondary/danger/outline-danger/success/icon，共 8 种；但缺 warning/info/ghost/link 等变体，缺 btn-group/btn-block/btn-loading 等功能类 |
| **表单体系** | ⚠️ 单尺寸 | 仅一种尺寸（padding 10/14，font 13px），缺 sm/lg 尺寸，缺校验态 is-valid/is-invalid，缺 form-check/radio 统一样式 |
| **卡片体系** | ✅ 较好 | `.card` + `.card-header` + `.card-title-icon` 已完善，有 hover 动效 |
| **Stat Card** | ❌ 三套实现不一致 | components.css / custom.css / factor-library.css 各一套，padding、字号、顶部光带粗细都不同 |
| **Tabs / Switch / Toast / Tooltip / Popover / Progress / Empty State** | ⚠️ 有实现但放错位置 | 都在 custom.css 里，质量尚可，但不属于工具类，应迁入组件层 |
| **硬编码色值** | ❌ 有少量 | components.css 第 374~389 行的 `.status-dot.status-online/offline/unknown` 用了 `#22c55e` / `#ef4444` / `#9ca3af`，破坏主题一致性 |

### 7.2 页面级自定义样式 vs 通用样式对比

| 页面自定义样式 | 通用样式是否有 | 谁的质量更高 | 结论 |
|---|---|---|---|
| backtest.css `.btn-bt` 系列（6 色 + 2 尺寸） | 部分有（缺 warning/success/ghost） | 页面级变体更丰富 | **整合**：把页面级的优秀变体吸收进通用组件 |
| backtest.css `.btn-seal`（印章按钮） | ❌ 无 | 页面级独有 | **新增**：印章按钮纳入通用组件库 |
| strategy.css `.btn-*` 作用域覆盖 | ✅ 有 | 差异不大，全局已够用 | **删除**：页面级覆盖，直接用全局 |
| screener.css `.seal-btn`（印章按钮） | ❌ 无 | 与 backtest 各有千秋 | **整合**：统一为一套全局印章按钮（取两者精华） |
| factor-library.css `.stat-card` | ⚠️ 有但不一致 | 各有特色（factor 版结构更丰富，有 stat-foot/.stat-tag） | **整合**：以 factor 版为蓝本完善全局 stat-card |
| factor-library.css `.sort-btn` / `.mini-btn` | ⚠️ 部分有 | 全局 btn-icon 可替代 | **替换**：用全局 `.btn-icon` + `.btn-sm` |

### 7.3 核心结论

**当前通用组件不是"差"，而是"不全、不统一、放错位置"。**
- ✅ theme.css 三套主题 token 非常完善，是改造的坚实基础
- ✅ components.css 的按钮/卡片/表单基础质量不错，直接可用
- ⚠️ 缺口在于：按钮变体不够、表单缺尺寸和校验态、组件散落各处
- ❌ 最大问题是：**同一个组件有多套实现**（stat-card 三套、印章按钮两套），导致视觉不一致

**改造策略不是"推倒重来"，而是"升维整合"**：把各页面的优秀实现吸收进通用组件库，统一后质量 > 任一单页面实现。

## 八、稳妥实施策略（确保效果不降级）

### 8.1 实施原则

1. **先建后拆**：先完善通用组件（Phase 1+2），确保通用组件的质量 ≥ 各页面自定义实现，再动页面
2. **增量替换**：页面级 CSS 不直接删，先加全局类，验证通过后再删页面级重复样式
3. **逐个验证**：每改一个页面，都在三套主题下验证视觉效果
4. **可回退**：所有改动在独立分支，页面级样式先注释而非删除，验证后再清理

### 8.2 分阶段验证节点

| 阶段 | 验证内容 | 通过标准 |
|---|---|---|
| **Phase 1 完成后** | components.css 新补全的组件（按钮变体/表单尺寸/印章按钮/stat-card 统合） | 在 demo 页或任一页面验证：三套主题下视觉正常、状态完整、动效流畅 |
| **Phase 2 完成后** | 从 custom.css 迁入的组件（Tabs/Switch/Toast 等） | 与迁入前视觉一致或更好，无功能回归 |
| **Phase 3 每个页面** | 该页面的按钮/表单/卡片等 | 与改造前对比：① 三套主题都正常 ② 视觉不降级 ③ 功能不缺失 |

### 8.3 兜底方案

| 风险场景 | 兜底措施 |
|---|---|
| 某页面替换全局样式后视觉变差 | 立即保留该页面的自定义样式，回退到改造前状态，分析差异后再优化通用组件 |
| stat-card 统合后某页面不适应 | 页面级用 `--stat-accent` / 覆盖少量 CSS 变量微调，不重写整个组件 |
| 印章按钮统一后某页面风格不搭 | 提供 `.btn-seal-sm` / `.btn-seal-lg` 尺寸变体，通过尺寸适配 |
| 主题切换后某组件颜色异常 | 检查是否有硬编码色值，替换为 var() + color-mix |

### 8.4 改造后质量承诺

统一后的通用组件库将在以下维度**全面优于**当前碎片化状态：

| 维度 | 改造前 | 改造后 |
|---|---|---|
| **一致性** | 同一种按钮 3 种写法、stat-card 3 套实现 | 所有页面用同一套组件，视觉 100% 统一 |
| **主题适配** | 硬编码色值在 cyber 主题下可能刺眼 | 100% 基于 theme token，三套主题都经过设计 |
| **组件丰富度** | 按钮 8 种变体 | 按钮 12+ 种变体 + 印章按钮 + 按钮组 + 加载态 |
| **表单完整性** | 单尺寸、无校验态 | sm/md/lg 三尺寸 + 校验态 + checkbox/radio 统一样式 |
| **可维护性** | 改个按钮颜色要找 5 个文件 | 改 components.css 一处，全站生效 |
| **新页面开发** | 复制粘贴其他页面的样式 | 直接用全局类，开箱即用 |
