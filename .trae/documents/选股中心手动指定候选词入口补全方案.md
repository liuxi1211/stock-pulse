# 选股中心「手动指定」候选词入口补全方案

## 一、背景与目标

### 1.1 现状问题
选股中心的候选池（universe）支持 4 种取值（`all_a_shares`/`csi300`/`csi500`/`manual`），但前端"手动指定"是个**半成品入口**：
- 候选池下拉框有"手动指定"选项
- 但**没有录入股票代码的 UI**，[screener.js collectConfig() L557-559](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/resources/static/js/screener.js#L557-559) 选 manual 时直接 `cfg.stocks = []`
- 后端 [ScreenConfigValidator.validateUniverse L87-93](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/util/ScreenConfigValidator.java#L87-93) 强制校验 `universe=manual 必须非空 stocks`，导致前端走 manual 路径必然报错

### 1.2 候选词归属方案的设计意图（回答用户疑问）
候选词（universe）算在方案里而非每次运算手动指定，有以下好处：
- **可复现**：同方案多次运行候选池固定，结果可对比
- **幂等缓存**：universe 是缓存 key 的一部分（[toParamsJson L903-912](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/ScreenerServiceImpl.java#L903-912)），24h 内同参数复用结果
- **批量/定时调度**：方案可被定时任务直接跑，无需人工每次传参
- **配置版本管理**：universe 与条件/排序/过滤作为整体快照存档

现状已支持运行时覆盖（`POST /run` 顶层 `universe` 覆盖方案值），是"方案内置默认 + 运行时可覆盖"的混合模式。

### 1.3 本次目标
1. **补全前端手动指定入口**：universe 切换为 manual 时，展示"搜索 + 多选标签云"区域，支持添加/删除候选股
2. **打通 code 口径**：后端 `/search/suggest` 返回 tsCode（而非 symbol），与方案 `stocks` 数组口径对齐
3. **支持运行时临时编辑**：顶部运行条选 manual 时可临时改候选股（不改方案）
4. **manual 单独设上限 50 只**（用户明确要求，区别于全市场的 500）

## 二、范围与决策汇总

| 决策项 | 选定方案 |
|---|---|
| 股票搜索数据源 | 复用现有 `GET /search/suggest`，后端 VO 增返 tsCode |
| 100 只股票展示形态 | 内联标签云 + 滚动 |
| manual UI 触发时机 | universe 切换为 manual 才显示 |
| code 口径对齐 | 后端 SuggestItemVO 增加 tsCode 字段（推荐，最干净） |
| 已保存 stocks 加载 | 批量拉详情回填名称（新增批量接口） |
| 运行时临时编辑 | 支持（后端 resolve() 增加运行时 stocks 处理） |
| manual 数量上限 | 50 只（新增常量 `SCREEN_MANUAL_MAX_CANDIDATES = 50`） |

- 现有 resolveUniverse 用 `SCREEN_MAX_CANDIDATES=500` 截断，manual 需单独更紧上限

## 五、数据流（改动后）

### 5.1 方案配置保存（manual）
```
用户 universe=manual → 搜索添加股票 → state.manualStocks
  → collectConfig: cfg.stocks = [tsCode, ...]
  → POST /screener/plans (screenConfig JSON 含 stocks)
  → ScreenConfigValidator 校验非空 + ≤50
  → 落库 screen_plan.screen_config
```

### 5.2 方案加载（manual）
```
GET /screener/plans/{id} → screenConfig.stocks = [tsCode, ...]
  → applyConfig → loadManualStocksDetail(tsCodes)
  → GET /search/batch?tsCodes=... → [{tsCode, code, name}]
  → state.manualStocks + renderManualCloud
```

### 5.3 运行（无临时覆盖）
```
POST /screener/plans/{id}/run {date, universe:"manual"}
  → resolve(): stocks = config.stocks（方案值）
  → resolveUniverse("manual", [方案tsCodes], config)
  → buildCandidates → engine
```

### 5.4 运行（临时编辑候选股）
```
用户点"编辑候选股" → 弹层改 → state.runTimeStocks = [新tsCodes]
  → runScreen: POST /run {date, universe:"manual", stocks:[新tsCodes]}
  → resolve(): stocks = req.stocks（运行时值优先）
  → resolveUniverse("manual", [运行时tsCodes], config)
  → 运行完成清空 state.runTimeStocks
```

## 七、假设与决策

1. **假设**：StockBasicDO 已有 tsCode 字段（探索确认 `@TableName("stock_basic")` 含 tsCode，resolveUniverse L578 已用 `StockBasicDO::getTsCode`）
2. **决策**：SuggestItemVO 加 tsCode 而非替换 code，避免破坏 dashboard 现有依赖（dashboard 用 item.code）
3. **决策**：批量接口用 GET + 逗号分隔 tsCodes 参数（简单），若超长可改 POST body（暂不需要，50 只 tsCode 长度可控）
4. **决策**：运行时编辑用 inline 弹层（与方案编辑同组件），不引入新 modal 框架
5. **决策**：manual 上限 50 既能覆盖典型自选场景，又控制 HTTP 体积和 engine 计算量
6. **范围外**：csi300/csi500 成分股表建立（保持现状降级全市场，本次不动）
7. **范围外**：方案内缓存 stocks 名称（决策用批量拉详情，避免数据冗余和一致性负担）
