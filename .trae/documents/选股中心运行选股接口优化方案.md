# 选股中心「运行选股」接口性能与架构优化方案

> 视角：高级 Java 后端开发。聚焦 `ScreenerController.runPlan → ScreenerServiceImpl` 链路。
> 范围：runPlan 核心链路 + 追踪链路（applyTracking / getLockDetail / buildContributions）。
> 决策已锁定：**同步执行 + 批量 SQL + CompletableFuture 并行 + daily_quote distinct 缓存 + 幂等缓存**。

## 二、改造目标与原则

1. **批量替代循环**：N+1 单查 → 1~2 次批量 IN 查询。
2. **缓存复用**：交易日历、最新交易日、幂等结果用 Caffeine 缓存。
3. **并行拼装**：candidates 拼装（CPU 密集 + 内存映射）用 CompletableFuture 并行。
4. **职责分层**：拆分 ScreenerServiceImpl，保持对外接口不变。
5. **KISS / 最小改动**：不引入新表、不改 engine 契约、不改前端 API。

```java
List<DailyQuoteDO> selectOhlcvByCodesAndDateRange(
```

### Step 6：重构 ScreenerServiceImpl.runPlan（核心）

#### 6.1 overrides 合并重构（提取 ResolutionContext）

新增内部 record / DTO：

```java
private record ScreenResolution(
        String universe, String isoDate, Object conditions,
        Object ranking, FiltersDTO filters, Integer topN) {}
```

把 163-196 行裸 JSON 合并逻辑提取为：

```java
private ScreenResolution resolve(ScreenPlanDO plan, ScreenPlanRunRequestDTO req) {
    JSONObject config = parseConfig(plan.getScreenConfig());
    ScreenResolution base = ScreenResolution.from(config);          // 强类型读取
    if (req == null) return base;
    return base.override(req);                                       // req 字段优先，overrides 二次合并
```

`override` 方法内统一类型转换与校验，消除 `instanceof Map` 与多重 firstNonNull。

#### 6.2 candidates 并行批量拼装

替换 [`buildCandidates:620`](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/ScreenerServiceImpl.java#L620)：

```java
private Map<String, CandidateStockDTO> buildCandidates(List<StockBasicDO> stocks) {
    List<String> codes = stocks.stream().map(StockBasicDO::getTsCode).toList();
    // ① 一次批量 SQL 取回所有 OHLCV
    Map<String, List<DailyQuoteDO>> ohlcvMap = dailyQuoteService.queryRecentOhlcvByCodes(codes, SCREEN_HISTORY_BARS);
    // ② 并行拼装（map 结构 -> DTO）
    List<CompletableFuture<CandidateStockDTO>> futures = stocks.stream()
            .map(b -> CompletableFuture.supplyAsync(
                    () -> buildOneCandidate(b, ohlcvMap.getOrDefault(b.getTsCode(), List.of())),
                    screenerExecutor))
            .toList();
    // ③ join 合并
    Map<String, CandidateStockDTO> result = new LinkedHashMap<>(stocks.size());
    for (int i = 0; i < stocks.size(); i++) {
        result.put(stocks.get(i).getTsCode(), futures.get(i).join());
    }
    return result;
}
```

`buildOneCandidate` 改为接收已查好的 `List<DailyQuoteDO>`（直接截尾 60 根映射），不再触发 DB。

#### 6.3 runPlan 主流程

```java
@Override
public ScreenResultVO runPlan(Long id, ScreenPlanRunRequestDTO req) {
    ScreenPlanDO plan = requirePlan(id);
    ScreenResolution r = resolve(plan, req);

    // 幂等缓存
    ScreenResultVO cached = resultCache.get(id, r.toParamsJson());
    if (cached != null) return cached;

    List<StockBasicDO> universeStocks = resolveUniverse(r.universe(), parseConfig(plan.getScreenConfig()));
    Map<String, CandidateStockDTO> candidates = buildCandidates(universeStocks);

    SnapshotRequestDTO snapshotReq = buildSnapshotReq(r, candidates);
    log.info("选股执行 plan={} universe={} date={} candidates={}", id, r.universe(), r.isoDate(), candidates.size());

    SnapshotResultVO snapshot = screenerClient.runSnapshot(snapshotReq);
    Long resultId = persistResult(id, toCompactDate(r.isoDate()), snapshot, r.toParamsJson());
    ScreenResultVO vo = toResultVO(screenResultMapper.selectById(resultId));
    resultCache.put(id, r.toParamsJson(), vo);
    return vo;
}
```

> persistResult 已是 `@Transactional` 且只包 DB 写，HTTP 在外，符合现有约束（事务边界正确，无需改动）。

### Step 7：追踪链路重构（applyTracking / getLockDetail / buildContributions）

#### 7.1 applyTracking 替换全表扫描

- `listSortedTradeDates()` → `tradeCalendarService.getSortedTradeDates()`。
- `getClose(symbol, date)` 循环 → 收集所有 (symbol,date) 对，一次性调 `selectClosesByCodesAndDate` 批量查，构造 `Map<(symbol,date), BigDecimal>` 内存查询。
- 基准 `getClose(BENCHMARK_CODE, ...)` 同批查（把 BENCHMARK_CODE 一起塞进 codes 列表）。

#### 7.2 buildContributions 批量化

- 锁定日收盘价：批量 `selectClosesByCodesAndDate(symbols, lockDate)`。
- 最新日收盘价：批量 `selectClosesByCodesAndDate(symbols, latestDate)`。
- 单只股 getClose 全部改为 Map.get。

### Step 8：事务边界复核

- `persistResult`：保持 `@Transactional`，仅 DB 写。
- `applyTracking`：保持 `@Transactional`，update 单条 lock。
- `runPlan`：**不加事务**（HTTP 调用在事务内是反模式，DB 写已下沉 persistResult）。

## 五、假设与决策

1. **不引入新表**：交易日历走 daily_quote distinct + Caffeine，符合"最小改动"。
2. **同步执行**：保留 runPlan 同步返回 VO 契约（前端不改），但内部用 CompletableFuture 并行拼装 + 批量 SQL，把 500 股的拼装耗时从串行秒级降到并行百毫秒级。
3. **幂等粒度**：planId + 参数 SHA-256，TTL 24h。命中时附响应头标识（避免重复打 engine）。
4. **startDate 估算**：`recentBars × 2` 自然日回溯，保守覆盖节假日；不足时 engine 会按已有数据计算（不影响正确性）。
5. **engine 不可达**：保持现有 AbstractEngineClient 抛 BusinessException 的兜底，不额外处理。
6. **追踪并发**：applyTracking 由 `ScreenLockTrackingTask` 串行调用，批量查询已消除 N+1，无需额外并发。
