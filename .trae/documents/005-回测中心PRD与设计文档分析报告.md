## 一、总体结论（TL;DR）

**整体评价：设计质量在「域 B 量化研究」三个锚点中是最高的，主链路（单次回测端到端）逻辑闭环、akquant API 映射准确、硬约束对齐到位。** 但存在 **3 个致命问题 + 5 个高严重性问题**，若不处理会直接导致开发返工或用户用不起来。

| 严重性 | 数量 | 典型代表 |
|---|---|---|
| 🔴 致命 | 3 | eval 注入设计、四范式复杂度失控、PRD/spec 双源不一致 |
| 🟠 高 | 5 | 部分功能无真实价值、compiler 隐含未文档化的工作��、benchmark 缺失影响决策 |
| 🟡 中 | 6 | 前端步骤冗长、Open Questions 状态混乱、文档缺口等 |
| 🟢 低 | 4 | 命名/编号约定、字段小遗漏等 |

**建议的最小改造范围**（若后续要改）：
1. 🔴 **砍 GRID/WF 的 eval 表达式**，改结构化前端编辑（或直接删 constraint/resultFilter 字段，第一版不暴露）
2. 🔴 **明确分波**：第一波只做 SINGLE + P0 信号驱动单标的；GRID/WF + rebalance/exit 进第二波
3. 🔴 **消除 PRD/spec 双源**：PRD 为需求真相、spec 为开发契约，删除 spec 中的重复段落
4. 🟠 **补 benchmark**：A 股回测不对比沪深300，夏普/超额收益无意义
5. 🟠 **降级「应用最优参数」按钮**：第一版只展示最优参数，不自动创建策略版本（风险高）

## 二、🔴 致命问题（3 项，必须处理）

### 🔴-1：GRID/WF 的 `constraint`/`resultFilter` 用 `eval` 求值——设计层面就是注入漏洞

**位置**：PRD [FR-2](file:///d:/lcProject/stock-pulse/sdlc/prd/005-回测中心/回测中心PRD.md) 第 207-208 行、[NFR-5](file:///d:/lcProject/stock-pulse/sdlc/prd/005-回测中心/回测中心PRD.md) 第 553 行、spec.md [FR-2](file:///d:/lcProject/stock-pulse/.trae/specs/007-backtest-center/spec.md) 第 79 行、checklist.md 第 66 行、tasks.md [SubTask 7.1](file:///d:/lcProject/stock-pulse/.trae/specs/007-backtest-center/tasks.md)

**问题描述**：
PRD FR-2 把 `constraint` 和 `resultFilter` 定义为「Python lambda 表达式字符串」，engine 侧用 `eval` 受控求值。AC-12 虽要求「白名单 globals 防注入」，但这是一个 **设计层面的反模式**：

```json
"constraint": "fast < slow",
"resultFilter": "max_drawdown_pct < 20"
```

```python
# tasks SubTask 7.1 的写法
constraint=lambda p: p["fast"] < p["slow"]   # ← 这不是字符串 eval，但 PRD FR-2 写的是字符串
```

**核心矛盾**：PRD FR-2 写的是字符串表达式（`"fast < slow"`），但 [06-optimization.md §1](file:///d:/lcProject/stock-pulse/.trae/rules/akquant/06-optimization.md) 的 `run_grid_search(constraint=...)` 要求的是 **可调用对象**（`f(params)->bool`）。两者类型对不上。而且：
- 若用 `eval("fast < slow", {"__builtins__": {}}, params)`，白名单极易绕过（Python 沙箱逃逸是经典难题，`().__class__.__bases__[0].__subclasses__()` 即可拿到 os）
- 若改成 `eval(constraint, {}, {"m": metrics})`，用户得自己写 `m["max_drawdown_pct"] < 20`——这要求用户懂 Python dict 语法，对「个人 A 股投资者」完全不友好
- 多进程 pickle（`max_workers>1`）时，lambda **不可 pickle**（[09-pitfalls-conventions.md](file:///d:/lcProject/stock-pulse/.trae/rules/akquant/09-pitfalls-conventions.md) 明确警告），constraint 作为 lambda 传入会直接报错

**用户价值角度**：散户研究者根本不会写 `"fast < slow"` 这种表达式。他们要的是前端勾选「fast 必须 < slow」这种结构化约束。

**改造建议**：
- **方案 A（推荐，低风险）**：第一版直接 **删掉** `constraint`/`resultFilter` 字段。约束逻辑由前端「参数组合预览」做（如用户填 fast=[5,10], slow=[20,30]，前端自动跳过 fast≥slow 的组合）。过滤逻辑由结果表的「筛选」UI 做（前端按 max_drawdown_pct 排序/过滤）。
- **方案 B（中等成本）**：保留字段但改成 **结构化 DSL**：`{"left":"fast", "op":"<", "right":"slow"}`，engine 侧用安全求值器（不用 eval）。但这会引入一套新的条件模型，与统一 Schema §4 的 ConditionTree 重复造轮子。
- **方案 C（不推荐）**：保留 eval 但只在单进程模式（`max_workers=1`）下允许，且严格白名单。治标不治本。

**影响范围**：PRD FR-2、NFR-5、AC-12；spec FR-2；checklist §三、§六；tasks Task 7.1。

### 🔴-2：compiler 四范式 + 混合优先级的复杂度远超第一波范围

**位置**：PRD [FR-6a](file:///d:/lcProject/stock-pulse/sdlc/prd/005-回测中心/回测中心PRD.md) 第 348-363 行、spec FR-6 第 114-117 行、tasks Task 4（SubTask 4.1~4.5）

**问题描述**：
compiler 要同时支持：
1. `signals.buy/sell`（信号驱动，ConditionTree + TradingConditionEngine）
2. `rebalance`（调仓驱动，本地 ConditionEngine + ranking + `rebalance_to_topn`）
3. `exit.bracket`（OCO 括号单，`place_bracket_order`）
4. `exit.rules`（动态出场 ConditionTree，支持 ref）
5. **混合范式优先级**：「调仓日优先调仓，非调仓日执行 signals+exit」

PRD 自己在 FR-6a 末尾标注「⚠️ 分阶段实现：P0 先支持 signals 单标的 → P1 rebalance → P2 exit.bracket/rules」，但：
- spec.md **没有体现分阶段**，把 FR-6 的 compiler 描述平铺
- checklist §二 把 P0/P1/P2 并列成 4 个 checkbox，没有「第一波只做 P0」的明确 gating
- tasks Task 4 把 SubTask 4.1~4.5 列在一起，依赖图说「Task 5 依赖 Task 4」——暗示要全做完

**核心问题**：
- **动态生成 akquant Strategy 子类** 本身就是高难度工作（要处理 `__init__` 参数注入、`warmup_period` 推断、pickle 兼容、受控命名���间）。单单 P0（signals 单标的）就是一个完整的挑战。
- 加上 rebalance 需要在 engine 本地复现 003 的选股逻辑（ConditionEngine + factor_calculator + ranking + filters），等于把选股中心的内核在回测里重写一遍——这违反了「选股逻辑归 003」的边界（PRD [Non-Goals](file:///d:/lcProject/stock-pulse/sdlc/prd/005-回测中心/回测中心PRD.md) 第 72 行自己说了「不另写」）。
- 混合范式优先级（PRD Open Question 第 7 项）至今 **未决**，却是 compiler 的核心分支逻辑。

**用户价值角度**：
- 双均线单标的（signals）是散户最高频场景，必须做。
- 多因子调仓（rebalance）是进阶场景，用户少，且选股中心已经能跑——回测里重跑一遍价值有限。
- exit.bracket（静态止损止盈）有价值但可后置。
- 混合范式几乎没人用（散户不会同时写 signals 和 rebalance）。

**改造建议**：
- **第一波（P0）**：只做 signals 驱动单标的 + 固定池（`order_target_percent`/`buy`/`sell`/`close_position`）。这是 README 第一波「6~8 页」的合理范围。
- **第二波（P1）**：rebalance 调仓。此时要先解决「engine 本地选股如何复用 003 内核而不重写」——可能要把 ConditionEngine 抽成共享包。
- **第三波（P2）**：exit.bracket/rules + 混合范式。
- spec/checklist/tasks 必须 **显式标注分波 gating**，不能平铺。

**影响范围**：PRD FR-6a、§10 Open Question 第 7 项；spec FR-6；checklist §二；tasks Task 4 全部。

## 三、🟠 高严重性问题（5 项）

### 🟠-1：benchmark（基准）缺失——A 股回测不对比沪深300，夏普/超额收益无决策价值

**位置**：PRD [Assumptions](file:///d:/lcProject/stock-pulse/sdlc/prd/005-回测中心/回测中心PRD.md) 第 599 行「基准第一版可选不传」、Open Question 第 2 项

**问题描述**：
- A 股散户回测的核心问题是「我的策略跑赢沪深300了吗」。不叠加 benchmark，`sharpe_ratio` 只是个孤立数字，用户无法判断好坏。

**用户价值**：🔴 **高**。没有 benchmark 的回测报告对散户决策价值减半。

**改造建议**：
- 第一版 **强制** 默认 benchmark=沪深300（`000300.SH`）。watcher 拼装 kline_data 时一并拼装 benchmark 的 OHLCV，传入 engine 的某个独立字段。
- 若用户 config 指定别的 benchmark（如中证500），覆盖默认。
- 报告页净值曲线必须叠加 benchmark（否则 FR-11 的「vs 基准」是空话）。

### 🟠-2：「应用最优参数」按钮自动创建策略版本——风险高、价值低

**位置**：PRD [FR-12](file:///d:/lcProject/stock-pulse/sdlc/prd/005-回测中心/回测中心PRD.md) 第 487 行、spec FR-12 第 166 行

**问题描述**：
- GRID 寻优找到 Top1 参数后，点「应用最优参数」会调 `PUT /api/strategies/{id}/config` **自动创建新策略版本**。
- 问题：网格寻优天然容易过拟合（在历史数据上找最优 = 曲线拟合）。自动把过拟合参数固化为新版本，等于鼓励用户用过拟合参数。
- Walk-forward（FR-3）本来是用来检测过拟合的，但 PRD 把「应用最优参数」放在 GRID 结果页（FR-12），不在 WF 通过后才启用——逻辑上本末倒置。
- 用户价值：低。散户手动改参数也就点两下，自动化收益小；但自动化的「过拟合背书」危害大。

**改造建议**：
- 第一版「应用最优参数」**降级为「复制参数到剪贴板/跳转编辑器」**，让用户手动确认是否创建新版本。
- 或者：只有当该参数组合 **同时通过 WF 验证**（样本内外收益差 < 阈值）时，才允许「应用」。

### 🟠-4：watcher 重启后 RUNNING→FAILED，但不支持断点续跑——长任务（WF 5min）用户体验差

**位置**：PRD [NFR-3](file:///d:/lcProject/stock-pulse/sdlc/prd/005-回测中心/回测中心PRD.md) 第 535 行、Assumptions 第 600 行

**问题描述**：
- WF 回测单任务可达 5min（NFR-1）。若 watcher 在第 4min 重启（部署/崩溃），任务直接 FAILED，用户 4min 白等。

**用户价值**：中。WF 是进阶功能，但 5min 任务失败重跑很烦。

**改造建议**：
- 第一版至少提供「一键重跑」按钮（复用原 config 重新提交），不做断点续跑。
- Open Question 里明确标注「断点续跑受 engine 硬约束限制，不在范围」。

## 四、🟡 中严重性问题（6 项）

### 🟡-1：`rebalance_to_topn` 在 akquant 权威手册 03-strategy-api.md 未收录（文档缺口）

**事实**：`rebalance_to_topn` 在 [strategy.py:982](file:///d:/lcProject/stock-pulse/akquant-0.2.47/python/akquant/strategy.py) 真实存在，签名完整。但 [.trae/rules/akquant/03-strategy-api.md](file:///d:/lcProject/stock-pulse/.trae/rules/akquant/03-strategy-api.md) 的「下单方法」章节 **只字未提**，只在 [08-recipes.md](file:///d:/lcProject/stock-pulse/.trae/rules/akquant/08-recipes-stock-engine.md) 配方里出现。
**影响**：PRD/spec 大量引用 `rebalance_to_topn`，但开发者查手册查不到，会误以为不存在。
**改造建议**：补 03-strategy-api.md §6.2，加 `rebalance_to_topn` 完整签名（含 `min_score`/`rebalance_tolerance` 等 PRD 未提的参数）。

### 🟡-2：编号体系混乱（业务 005 vs spec 007）

PRD 自己解释了（「`.trae/specs/005-*` 已被策略可视化编辑器占用」），但实际结果是：搜索 `005` 找到 PRD，搜索 `007` 找到 spec，跨文档引用容易迷路。
**改造建议**：spec 目录改名为 `007-backtest-center（业务005）`，或在 PRD/spec 顶部加双向链接（PRD 已有，spec 缺反向链接）。

### 🟡-3：Schema 字段小遗漏（`min_score`/`rebalance_tolerance`/`replace_method`）

`rebalance_to_topn` 真实签名有 `min_score`、`rebalance_tolerance`、`allow_leverage`、`rebalance_tolerance` 等参数，Schema §3.3.4 的 `rebalance` 表只映射了 `weight_mode`/`max_single_position`/`long_only`，漏了 `min_score`（最低分数阈值，实用）和 `rebalance_tolerance`（调仓容忍度，减少换手）。
**改造建议**：Schema §3.3.4 补字段；PRD FR-6a 的 rebalance 映射同步补。

### 🟡-4：报告导出 HTML 的落盘策略未定（Open Question 第 4 项）

PRD FR-11 说「导出 HTML 报告（调 engine `result.report()` 生成，watcher 临时目录落盘后下载）」，但 engine 硬约束是「只返回 JSON，不落业务库」。`result.report()` 生成 HTML 文件，engine 要不要落盘？落哪里？watcher 怎么拿？
**改造建议**：第一版 **砍掉 HTML 导出**，只保留 CSV 导出（trades 表前端生成）。HTML 报告让用户直接访问 engine `/docs` 跑（开发模式）。

### 🟡-5：任务取消「仅 PENDING 可靠取消」——RUNNING 占绝大多数时间窗口

PRD FR-5 说「RUNNING 第一版需等完成或超时」。但 SINGLE 回测 10s~60s，GRID 90s，WF 5min——大部分时间任务都在 RUNNING。用户点取消没反应，体验差。
**改造建议**：第一版 **隐藏取消按钮**（或对 RUNNING 显示「取消请求已记录，任务将在当前 bar 完成后终止」），不要给用户假期望。

### 🟡-6：AC-6「调仓日选股与 003 一致」是强一致性约束，但无测试机制

AC-6 要求「同一 screen_config，同一日期，同一候选池，003 和回测选出的股票集合一致」。但 PRD 没定义「如何对比」（选股结果存哪？回测调仓日选股结果存哪？）。
**改造建议**：与 🔴-2 合并处理——第一波不做 rebalance，AC-6 自动归位到第二波。

### 6.2 散户典型使用路径（验证功能闭环）

```
用户在 004 创建双均线策略 → 保存为 v1（VERIFIED）
  ↓
进回测中心 → 选策略 v1 → 选 SINGLE 模式 → 改资金/日期 → 提交
  ↓
等 10s → 看报告：净值 vs 沪深300、回撤、夏普、交易明细
  ↓
（决策点：跑赢沪深300了吗？最大回撤能接受吗？）
  ↓
若满意 → 进 GRID 寻优（第二波）找更好参数
若不满意 → 回 004 改策略
```

**闭环验证**：第一波做 SINGLE + benchmark + 报告 + 对比，闭环成立。GRID/WF 是增强，非必需。

### 6.3 易用性风险点

1. **「策略版本必须 VERIFIED/ACTIVE 才能 回测」**——用户若在 DRAFT 状态想快速试跑，要先去 004 改状态。建议：允许 DRAFT 回测，但报告页标注「草稿版本回测，未经验证」。
2. **3 模式 Tab 平级**——90% 用户只用 SINGLE，应默认选中。
3. **前端常量集中声明**（BROKER_PROFILES/SORT_METRICS）——注明「权威来源 engine」是对的，但 **这些常量应该由 engine 提供一个 `/python/v1/backtest/constants` 接口动态返回**，而不是前端硬编码（否则 engine 改了 broker_profile，前端不同步）。
