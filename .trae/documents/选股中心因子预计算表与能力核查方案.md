> 双目标：① 新建因子预计算表，定时器预计算常用技术面因子 + 补齐基本面缺口；② 核查并记录当前选股引擎对复杂多条件/多因子组合的真实支持情况与短板。

## 一、当前状态分析（核查结论 + 数据缺口）

### 1.1 选股引擎能力核查（第二个问题）

源文件：[engine.py](file:///d:/lcProject/stock-pulse/stock-engine/services/screener/engine.py)、[ranking.py](file:///d:/lcProject/stock-pulse/stock-engine/services/screener/ranking.py)、[condition.py](file:///d:/lcProject/stock-pulse/stock-engine/models/schemas/condition.py)、[factors.default.json](file:///d:/lcProject/stock-pulse/stock-engine/data/factors.default.json)

**结论：engine 算法层完全支持多因子 + 多条件 AND/OR 任意嵌套组合，设计完整。**

| 能力 | 支持情况 | 说明 |
|---|---|---|
| 条件嵌套 | ✅ 完全支持 | `ConditionTree(operator=AND/OR, conditions=[...])` 可任意深度递归 |
| 比较算子 | ✅ 6 种 | `> < >= <= == !=`；`between` 用 `AND(>=, <=)` 组合表达 |
| 算术表达式 | ✅ 4 种 | `+ - * /`，除零降级 0.0，NaN 安全 |
| 多因子排序 | ✅ `single` / `composite` | composite 用 z-score 标准化 + 加权综合分 |
| 因子混用 | ✅ 技术+基本面可同树 | 47 个因子定义（24 技术 + 18 基本面 + 5 价格直通）|
| 截面/时序隔离 | ✅ 静态拦截 | `cross_up/cross_down`/`ref` 在选股路径提前 422 拒绝 |

**🟡 已知短板（不阻塞功能，但需留意）**：
1. **线程安全隐患**：[ranking.py L188-190](file:///d:/lcProject/stock-pulse/stock-engine/services/screener/ranking.py#L188-L190) 的 `_COMPOSITE_SCORE_CACHE` 是模块级 dict，并发 composite ranking 会互相覆盖。当前 `main.py reload=True` 单 worker 暂不暴露，多 worker 部署需修复（建议改 thread-local 或入参传递）。
2. **无原生 `between/in` 算子**：需用 AND/OR 组合表达，功能可达但条件树稍繁琐。

## 二、改造目标与原则

1. **预计算消除重算**：常用参数的技术面因子每日收盘后预计算入库，选股直接查表。
2. **补齐基本面缺口**：新建 daily_basic + fina_indicator 表 + Tushare 对接，fundamentals 有真实数据。
3. **回退兜底**：预计算表未命中（自定义参数/缺数据）时，透明回退到现有 engine 实时计算路径，零功能损失。
4. **复用现有因子元数据**：参数白名单、因子定义全部从 engine 的 `factors.default.json` 派生，单一数据源。
5. **KISS / 最小侵入**：不改 engine Python 侧算法，不改选股 HTTP 契约；只在 watcher 侧加表、加定时器、改 `buildOneCandidate`。

## 三、详细方案

### Part A：技术面因子预计算表

#### A.1 新建表 `factor_snapshot`（按交易日 × 股票 × 因子签名存当日值）

文件：[schema-sqlite.sql](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/resources/schema-sqlite.sql) + [schema-mysql.sql](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/resources/schema-mysql.sql)

```sql
CREATE TABLE IF NOT EXISTS factor_snapshot (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    trade_date   TEXT    NOT NULL,          -- YYYYMMDD
    ts_code      TEXT    NOT NULL,          -- 000001.SZ
    factor_key   TEXT    NOT NULL,          -- MA / MACD / KDJ ...
    params_json  TEXT    NOT NULL DEFAULT '{}',  -- {"timeperiod":5}
    output_index INTEGER NOT NULL DEFAULT 0,-- 多输出因子切片（MACD 0/1/2）
    factor_value REAL,                       -- NaN 存 NULL
    updated_at   TEXT,                       -- ISO 时间戳
    UNIQUE (trade_date, ts_code, factor_key, params_json, output_index)
);
CREATE INDEX IF NOT EXISTS idx_factor_snapshot_lookup
    ON factor_snapshot (trade_date, ts_code, factor_key);
CREATE INDEX IF NOT EXISTS idx_factor_snapshot_date ON factor_snapshot (trade_date);
```

设计要点：
- **粒度**：一行 = 一个交易日 × 一只股票 × 一个因子签名（factorKey+params+outputIndex）的当日值。
- **为什么存「当日值」而非序列**：选股 99% 场景只用选股日的因子值（`compute_single → [-1]`）；序列场景（如条件引用历史值）极少，保留 engine 实时路径兜底。
- **params_json 序列化**：排序后的 JSON 字符串（如 `{"timeperiod":20}`），保证 `(trade_date, ts_code, factor_key, params_json, output_index)` 唯一。
- **NaN 存 NULL**：查询时 `IFNULL` 或在 Service 层处理。

#### A.3 预计算定时任务

文件：**新建** `task/FactorSnapshotTask.java`、**新建** `service/FactorSnapshotService.java` + Impl

**核心流程**（每日 16:30 收盘数据落库后触发）：

```
1. 取最新交易日 tradeDate（tradeCalendarService.getLatestTradeDate）
2. 取全市场在市股票 stock_basic（list_status='L'）
3. 批量取每只股票近 maxLookback 根 OHLCV（maxLookback = 白名单因子所需最大窗口，如 MA250 需 250 根）
   → 用 dailyQuoteService.queryRecentOhlcvByCodes(codes, maxLookback)
4. 对每只股票：
   遍历白名单每个 (factorKey, params)：
     调 engine 批量因子计算接口 /python/v1/factors/compute（如已存在）
     或本地按签名调 akquant（需 engine 暴露批量计算 API，见 A.4）
     对多输出因子展开每个 outputIndex
     → 收集 (tradeDate, tsCode, factorKey, paramsJson, outputIndex, value) 行
5. 批量 UPSERT 入 factor_snapshot（按 UNIQUE 键去重）
```

定时器（参考现有 `DailyUpdateTask` 风格）：

```java
@Component
@RequiredArgsConstructor
@Slf4j
public class FactorSnapshotTask {
    private final FactorSnapshotService factorSnapshotService;

    // 每日 16:30（晚于 DailyUpdateTask 拉日线的时间）
    @Scheduled(cron = "0 30 16 * * MON-FRI")
    public void computeDaily() {
        try {
            int n = factorSnapshotService.computeForLatestTradeDate();
            log.info("因子预计算完成，共 {} 行", n);
        } catch (Exception e) {
            log.error("因子预计算失败", e);
        }
    }
}
```

#### C.1 buildOneCandidate 改造：从预计算表读 + 基本面表读

文件：[ScreenerServiceImpl.java](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/ScreenerServiceImpl.java) `buildOneCandidate`

```java
private CandidateStockDTO buildOneCandidate(StockBasicDO b, List<DailyQuoteDO> quotes) {
    CandidateStockDTO c = new CandidateStockDTO();
    // OHLCV 历史保持（engine 对未命中预计算的因子仍需实时算）
    c.setOhlcvHistory(buildOhlcvBars(quotes));

    // 基本面：从 daily_basic + fina_indicator 查（替换原 emptyMap）
    c.setFundamentals(buildFundamentals(b.getTsCode(), isoDate));

    // meta：补真实 ST/停牌/涨跌停标记
    c.setMeta(buildMeta(b, quotes));
    return c;
}

/** 组装基本面因子 Map（factorKey -> value），缺数据 NaN/缺省 */
private Map<String,BigDecimal> buildFundamentals(String tsCode, String isoDate) {
    Map<String,BigDecimal> m = new LinkedHashMap<>();
    DailyBasicDO db = dailyBasicMapper.selectById(tsCode, toCompactDate(isoDate));
    if (db != null) {
        m.put("PE_TTM", db.getPeTtm());
        m.put("PB", db.getPb());
        m.put("PS_TTM", db.getPsTtm());
        m.put("DV_RATIO", db.getDvRatio());
        m.put("TOTAL_MV", db.getTotalMv());
        m.put("CIRC_MV", db.getCircMv());
        m.put("TURNOVER_RATE", db.getTurnoverRate());
    }
    FinaIndicatorDO fi = finaIndicatorMapper.selectLatestBefore(tsCode, toCompactDate(isoDate));
    if (fi != null) {
        m.put("ROE_TTM", fi.getRoe());
        m.put("ROA_TTM", fi.getRoa());
        m.put("GROSS_MARGIN", fi.getGrossprofitMargin());
        m.put("NETPROFIT_MARGIN", fi.getNetprofitMargin());
        m.put("REVENUE_YOY", fi.getRevenueYoy());
        m.put("PROFIT_YOY", fi.getDtNetprofitIsYoy());
        m.put("DEBT_TO_ASSETS", fi.getDebtToAssets());
        // CURRENT_RATIO 按需
    }
    return m;
}
```

### Part D：补齐 meta 真实标记（ST/停牌/涨跌停）

文件：[ScreenerServiceImpl.java](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/ScreenerServiceImpl.java) `buildMeta`

```java
private CandidateMetaDTO buildMeta(StockBasicDO b, List<DailyQuoteDO> quotes) {
    CandidateMetaDTO meta = new CandidateMetaDTO();
    meta.setIndustry(b.getIndustry());
    meta.setListDate(toIsoDate(b.getListDate()));
    // ST：stock_basic.name 含 "ST"/"*ST"
    meta.setIsSt(b.getName() != null && b.getName().toUpperCase().contains("ST"));
    // 停牌/涨跌停：从末根 quote 的 pct_chg 判定
    if (quotes != null && !quotes.isEmpty()) {
        DailyQuoteDO last = quotes.get(quotes.size() - 1);
        BigDecimal pct = last.getPctChg();
        meta.setIsSuspended(last.getVol() != null && last.getVol().signum() == 0);
        if (pct != null) {
            meta.setIsLimitUp(pct.compareTo(new BigDecimal("9.9")) >= 0);   // 简化，主板10%/创业板20%需按 ts_code 前缀区分
            meta.setIsLimitDown(pct.compareTo(new BigDecimal("-9.9")) <= 0);
        }
    }
    return meta;
}
```

> 涨跌停阈值精确判定需按板块（主板 10%、创业板/科创板 20%、ST 5%），Phase 1 简化为主板 9.9%，Phase 2 精确化。

## 五、假设与决策

1. **预计算粒度**：存「交易日当日值」，选股 99% 场景只用当日值；历史值回退 engine 实时算。
2. **参数白名单**：覆盖 MA[5/10/20/60/120/250]、EMA[5/10/12/20/26]、RSI[6/12/14/24]、MACD/BOLL/KDJ/ATR 默认参数等常用组合（约 90% 场景）；未命中回退 engine 实时算，零功能损失。
3. **回退透明**：预计算表未命中（自定义参数 / 缺数据）→ buildOneCandidate 仍传 ohlcv_history，engine 走原实时路径，用户无感知。
4. **engine 边界**：engine 仍不触库（AC-11），预计算由 watcher 定时任务调 engine 的「批量因子计算」接口算好后入库，符合硬约束。
5. **涨跌停判定**：Phase 1 简化为主板 9.9%（覆盖大部分场景），Phase 2 按板块精确化。
6. **不动选股算法**：engine 算法层已足够强大（多因子/多条件/嵌套），本次只补数据缺口，不改算法。
7. **线程安全隐患记录**：ranking.py 的 `_COMPOSITE_SCORE_CACHE` 多 worker 风险记录在案，本次不修（单 worker 不暴露），待多 worker 部署时单独处理。
