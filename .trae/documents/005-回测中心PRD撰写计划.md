## PRD 章节大纲（executor 撰写时按此结构展开）

### 1. Overview
- **Summary**：在 StockPulse 双系统架构下落地「回测中心」（005），打通「策略 → 回测 → 报告 → 对比」决策链的最后一公里。消费 004 策略管理已校验的 config_json，由 watcher 编排数据 + 调 engine 执行 akquant 回测，全量持久化结果供前端报告与横向对比。
- **Purpose**：让用户对已保存的策略版本发起**单次回测 / 网格参数寻优 / Walk-forward 滚动验证**三种研究，查看完整绩效报告（净值/回撤/月度热力/交易明细/指标），并支持多次回测横向对比，量化评估策略优劣。
- **Target Users**：量化研究用户（前端页面操作）、策略版本对比（004 联查回测摘要）、信号中心（未来读取回测验证过的策略激活）。

### 2. Goals（≈10 条）
- 在 watcher（Java）侧实现回测任务管理：异步提交、状态轮询、结果查询、对比
- 新建数据库表 `quant_backtest`（主表/任务）+ `quant_backtest_report`（报告全量 JSON）+ `quant_backtest_optimize`（网格/WF 结果）
- 在 engine（Python）侧实现回测执行引擎：JSON config → akquant Strategy 子类动态编译、`aq.run_backtest` 调用、`BacktestResult` 序列化
- 实现三种回测模式：单次（FR-1）、网格寻优（FR-2，`aq.run_grid_search`）、Walk-forward（FR-3，`aq.run_walk_forward`）
- 在 watcher 侧实现行情/候选池数据编排：按 `screen_config.universe` + `backtest_config.date range` 从 SQLite 拼装 kline_data + 候选股票，HTTP 传 engine
- 实现 watcher → engine 的 `BacktestClient`（继承 `AbstractEngineClient`）
- 实现前端回测中心全套页面：列表、配置、报告、优化结果、WF 分析、对比（共 6 页）
- 实现 `StrategyVersionCompareDTO` 真实回填（替换当前 TODO 占位）
- 严格遵守 engine 不触库硬约束；严格对齐 akquant 0.2.47 API

### 5. Functional Requirements

#### FR-1: 单次回测（核心链路）
**watcher 侧**：
- `POST /api/backtest/run` - 提交单次回测任务（异步）
  - 请求体：`strategyId` + `versionNo`（必填，指向已 VERIFIED 策略版本）+ `mode=SINGLE` + 可选覆盖参数（initial_cash/date range 等覆盖 backtest_config）
  - 返回：`taskId`（立即返回，状态 PENDING）
- watcher 编排（`BacktestService.run`）：
  1. 按 strategyId+versionNo 读 `quant_strategy_version.config_json`
  2. 校验 config 对应策略版本 status ∈ {VERIFIED, ACTIVE}（DRAFT/ARCHIVED 拒绝）
  3. 解析 `screen_config.universe` → 从 SQLite 拼装候选股票列表
  4. 按 `backtest_config.start_date/end_date` 从 SQLite 拼装每只股票的 kline_data（list[dict]）
  5. 调 `BacktestClient.runSingle(config, klineMap, candidates)` → engine
  6. engine 返回序列化结果 → watcher 落 `quant_backtest` + `quant_backtest_report`
  7. 更新 task 状态 SUCCESS/FAILED
- `GET /api/backtest/tasks/{taskId}` - 轮询任务状态（PENDING/RUNNING/SUCCESS/FAILED + progress）
- `GET /api/backtest/{backtestId}/report` - 获取完整报告（metrics + equity_curve + trades + positions）

**engine 侧**：
- `POST /python/v1/backtest/run` - 单次回测执行
  - 请求：`BacktestRequest{ strategy_config: dict, kline_data: dict[str,list[dict]], candidates?: list }`
  - 编排（`services/backtest/runner.py`）：
    1. `StrategyConfigModel(**strategy_config)` 解析（复用 004 模型）
    2. `data_adapter.kline_to_df_map(kline_data)` → `Dict[str, pd.DataFrame]`
    3. `compiler.compile(config)` → akquant `Strategy` 子类
    4. `build_backtest_kwargs(config.backtest_config)` → run_backtest 参数 dict
    5. `aq.run_backtest(data=data_map, strategy=strat_cls, **kwargs)`
    6. `result_serializer.serialize(result)` → JSON（用 [05-result-metrics.md §5](file:///d:/lcProject/stock-pulse/.trae/rules/akquant/05-result-metrics.md) 的 `serialize_result` 模板）
  - 响应：`BacktestResponse{ metrics, equity_curve, daily_returns, trades, orders, positions }`

#### FR-2: 网格参数寻优
- `POST /api/backtest/optimize` - 提交网格寻优任务（异步）
  - 请求：`strategyId` + `versionNo` + `mode=GRID` + `paramGrid: dict`（如 `{"fast":[5,10], "slow":[20,30]}`）+ `sortBy?` + `maxWorkers?` + `constraint?` + `resultFilter?`
  - 注意：paramGrid 的 key 必须是策略 `__init__` 形参名（`strict_strategy_params=True` 时校验）
- engine `POST /python/v1/backtest/optimize`：
  - 调 `aq.run_grid_search(strategy=strat_cls, param_grid=paramGrid, data=data_map, sort_by=sortBy, max_workers=maxWorkers, return_df=True, **backtest_kwargs)`
  - **不传 `db_path`**（硬约束：engine 不写 sqlite；akquant 内部 sqlite 是其自身实现，但默认不启用断点续传）
  - 返回 DataFrame → 转 JSON（参数列 + 指标列）
- watcher 落 `quant_backtest_optimize`（主记录）+ 报告（参数组合结果表 JSON）
- 前端「参数优化结果」页：结果表（按 sort_by 排序）、参数敏感度热力图（ECharts）、「应用最优参数」按钮���创建新策略版本）

#### FR-3: Walk-forward 滚动验证
- `POST /api/backtest/walk-forward` - 提交 WF 任务（异步）
  - 请求：`strategyId` + `versionNo` + `mode=WALK_FORWARD` + `paramGrid` + `trainPeriod` + `testPeriod` + `metric?` + `compounding?`
- engine `POST /python/v1/backtest/walk-forward`：
  - 调 `aq.run_walk_forward(strategy, param_grid, data, train_period, test_period, metric, ...)`
  - 返回拼接的样本外资金曲线 DataFrame + 每段最优参数
- 前端「WF 分析」页：滚动训练/验证分段时序图、每段最优参数表、过拟合判定指标（样本内外收益稳定性）

#### FR-4: 回测任务数据模型（watcher/Java 侧）
新建数据库表（同步更新 schema-sqlite.sql / schema-mysql.sql）：
- `quant_backtest`（主表/任务表）：
  - `id` INTEGER PK AUTOINCREMENT
  - `task_id` TEXT NOT NULL UNIQUE（业务 ID）
  - `strategy_id` TEXT NOT NULL（FK 业务 ID）
  - `version_no` INTEGER NOT NULL
  - `mode` VARCHAR(16) NOT NULL（SINGLE/GRID/WALK_FORWARD）
  - `status` VARCHAR(16) DEFAULT 'PENDING'（PENDING/RUNNING/SUCCESS/FAILED/CANCELLED）
  - `progress` INTEGER DEFAULT 0（0-100）
  - `error_message` VARCHAR(1024)
  - `override_config` TEXT（覆盖 backtest_config 的 JSON，如临时改 initial_cash/date）
  - `param_grid` TEXT（GRID/WF 模式的参数网格 JSON）
  - `created_by` VARCHAR(64)（用户名，预留）
  - `started_at` / `finished_at` / `created_at` VARCHAR(32)（UTC ISO8601）
  - INDEX(strategy_id, version_no), INDEX(status), INDEX(mode)
- `quant_backtest_report`（报告全量表，SINGLE 模式）：
  - `id` INTEGER PK AUTOINCREMENT
  - `backtest_id` INTEGER NOT NULL（FK → quant_backtest.id）
  - `metrics_json` TEXT NOT NULL（核心指标 JSON：total_return_pct/sharpe/max_drawdown_pct/win_rate/...）
  - `equity_curve_json` TEXT（净值曲线 JSON：dates+values）
  - `daily_returns_json` TEXT
  - `trades_json` TEXT（交易明细 JSON 数组）
  - `orders_json` TEXT
  - `positions_json` TEXT
  - `created_at` VARCHAR(32)
  - UNIQUE(backtest_id)
- `quant_backtest_optimize`（网格/WF 结果表）：
  - `id` INTEGER PK AUTOINCREMENT
  - `backtest_id` INTEGER NOT NULL（FK → quant_backtest.id）
  - `result_type` VARCHAR(16)（GRID/WALK_FORWARD）
  - `results_json` TEXT NOT NULL（参数组合结果表 JSON）
  - `wf_equity_json` TEXT（WF 模式的样本外拼接曲线）
  - `best_params_json` TEXT（最优参数）
  - `created_at` VARCHAR(32)
- 创建 DO + Mapper：`QuantBacktestDO/Mapper`、`QuantBacktestReportDO/Mapper`、`QuantBacktestOptimizeDO/Mapper`
- 枚举：`BacktestModeEnum`、`BacktestStatusEnum`（实现 DisplayableEnum）

### 6. Non-Functional Requirements

#### NFR-1: 性能
- 单次回测（单标的，1 年日线）< 10s（含 HTTP 往返）
- 单次回测（组合 30 标的，3 年日线）< 60s
- 网格寻优（3×3=9 组合，单标的 1 年）< 90s（max_workers=4 并行）
- Walk-forward（train 120 + test 30，滚动 5 段）< 5min
- 任务状态轮询响应 < 200ms
- 报告页加载（含 trades 序列化）< 1s

#### NFR-2: 正确性
- 回测结果指标与 akquant `BacktestResult.metrics` 完全一致（序列化无损）
- A 股规则强制：`t_plus_one=True`、`lot_size=100`、`stamp_tax_rate` 仅卖出扣、`slippage` 用 dict
- 调仓日选股结果与 003 选股中心同条件同结果（共用 ConditionEngine + factor_calculator）
- ��务状态机：PENDING→RUNNING→SUCCESS/FAILED/CANCELLED，无非法跃迁
- 序列化 NaN/Inf → null，Timestamp → isoformat，Timedelta → seconds（可 JSON 序列化）

#### NFR-3: 可靠性
- engine 不可用时 watcher 返回 503，不写脏数据；任务状态置 FAILED + error_message
- 回测异常（编译错误/数据不足）engine 返回结构化错误，watcher 落 FAILED 状态 + 错误详情
- 任务持久化：watcher 重启后 PENDING/RUNNING 任务需重置为 FAILED（标「引擎中断」）或支持恢复（第一版重置为 FAILED）
- 长任务超时：read timeout 300s，超时后任务标 FAILED
- 并发：单 watcher 实例下任务串行或受限并发（第一版串行，避免 SQLite 写冲突；未来多实例需任务队列）

#### NFR-4: 可维护性
- engine 侧 `services/backtest/` 按职责拆分（compiler/runner/adapter/serializer/optimizer），不堆单文件
- 复用 004 的 `StrategyConfigModel`/`BacktestConfigModel`，不自造模型
- 序列化模板严格对齐 [05-result-metrics.md §5](file:///d:/lcProject/stock-pulse/.trae/rules/akquant/05-result-metrics.md)
- 前端 JS IIFE 模块（window.Backtest���，常量集中声明注明权威来源
- watcher 分层 Controller→Service→Mapper，与 ScreenerServiceImpl 风格一致

#### NFR-5: 安全性
- 所有 /api/backtest/** 和页面路由需认证（AuthInterceptor）
- 回测请求只接受已保存策略的 strategyId+versionNo，不接受任意 JSON config（防注入）
- engine 编译器在受控命名���间执行（白名单 globals，禁用 builtins 危险函数）
- 删除回测记录需确认（前端 confirm + 后端软删除/物理删除策略待定，第一版物理删除连带报告）

#### NFR-6: 兼容性
- akquant 锁定 0.2.47，参数名以该版本为准
- config_json 字段对齐统一 Schema v1.0
- 数据库 schema 用 CREATE TABLE IF NOT EXISTS / 增量迁移
- 前端页面骨架与 003/004 一致（fragments/common.html 复用）

### 7. Constraints
- **engine**: Python 3.12 + FastAPI + Pydantic 2.x + akquant 0.2.47
- **watcher**: Java 21 + Spring Boot 4.0.6 + MyBatis-Plus + Caffeine + SQLite(WAL)
- **前端**: Thymeleaf + Bootstrap 5 + ECharts 5 + lightweight-charts + 原生 JS
- **硬约束**: engine 禁 sqlite3/sqlalchemy/直连 .db；交互单向 watcher→engine；前端不直连 engine
- **业务约束**: 仅 A 股股票日线回测；仅消费已保存策略版本；t_plus_one/lot_size 强制
- **依赖约束**: 002 因子库、003 选股引擎、004 策略管理 + BacktestConfigModel 已就绪

### 8. Assumptions
- 行情数据前复权、清洗由 watcher 上游完成（Phase 0 已实现）
- 候选股票池（universe=all_a_shares/csi300/csi500）的成分股由 watcher 从 SQLite 读取
- 回测所需基本面因子（若 config 引用 PE_TTM 等）由 watcher 一并拼装入 kline_data 的 extra 字段传入 engine
- 单 watcher 实例部署（第一版任务串行；多实例需引入任务队列，超出本 PRD 范围）
- 网格寻优/WF 的 paramGrid 由用户在前端编辑（key 选策略 `__init__` 形参名，value 填数组）；策略形参 schema 可通过 `akquant.get_strategy_param_schema` 反推（未来增强，第一版前端手动填写）
- 基准（benchmark）第一版可选不传（净值曲线只显示策略）；未来支持 config 指定 index 代码
- 任务恢复（watcher 重启后 RUNNING 任务）第一版重置为 FAILED，不支持断点续跑

| 风险 | 缓解 |
|---|---|
| JSON config → Strategy 动态编译复杂（on_bar 逻辑多变） | 分范式实现：先支持 signals 驱动单标的（最简），再支持 rebalance 调仓，再支持 exit.bracket/rules |

| trades 数据量大（一笔回测数百行） | 全量持久化 JSON（用户已决策）；报告页分页加载 |
