## P1 · `scope`（适用范围）字段冗余

### 现状

基本信息页的「适用范围」单选卡片（`single` / `portfolio` / `mixed`），对应 `StrategyConfigModel.scope`。

### 证据

- engine 侧 [StrategyConfigModel.scope](file:///d:/lcProject/stock-pulse/stock-engine/services/strategy/models.py) 描述明确写："范围提示：single / portfolio / mixed（**仅提示，不驱动分支**）"。
- engine 的 [validator.py](file:///d:/lcProject/stock-pulse/stock-engine/services/strategy/validator.py) **完全不校验 scope**，真正决定范式的是 `trading_config.signals` / `trading_config.rebalance` 的字段在场情况。
- watcher 的 [BacktestServiceImpl](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/BacktestServiceImpl.java) 回测路径**根本不读 scope**。

### 当前唯一真实作用

- 前端编辑器联动：`scope === 'single'` 时禁用 Tab7「调仓」面板（[strategy-editor.js applyScope](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/resources/static/js/strategy-editor.js)）。
- 列表页卡片徽章 + 筛选下拉（纯展示）。

### 歧义场景

用户选 `scope=single`，但选股范围选 `csi300`，系统不阻止这种自相矛盾的组合。

### 处置候选方案

- **A · 移除 scope**：前端 Tab 联动改为"是否填了 `rebalance`"自动判定。
- **B · 改为只读派生字段**：保存时由后端根据 `trading_config` 结构自动算（有 rebalance → portfolio；有 signals → single；都有 → mixed），用户不可编辑。
- **C · 加一致性校验**：保留 scope 但在 watcher 加约束（scope=single 时禁止 rebalance 段）。

## P2 · `trading_config.symbols`（买入标的）字段冗余 ⭐

### 现状

买入信号页顶部的「买入标的」文本框（`f-symbols`），对应 `trading_config.symbols`。

### 证据（决定性）

1. **回测完全不读这个字段**。[runner.py:163-165](file:///d:/lcProject/stock-pulse/stock-engine/services/backtest/runner.py)：

   ```python
   # 6. 推断 symbols
   symbols = list(data_map.keys())   # ← 只取 watcher 传入的 kline_data 的 key
   ```

   `data_map` 由 watcher 传入的 `kline_data` 构造，**`trading_config.symbols` 在整个回测路径上没有任何代码读取**。

2. **akquant 的工作方式**：`run_backtest(data=..., symbols=..., strategy=...)` 的 `symbols` + `data` 的 key 决定回测哪些标的；`on_bar(bar)` 里 `bar.symbol` 就是当前标的，`buy()` 默认买 `bar.symbol`。**不存在"读 buy_symbol 字段指定买哪个"的逻辑**。

3. **compiler 也不读**：[compiler.py](file:///d:/lcProject/stock-pulse/stock-engine/services/backtest/compiler.py) 的 `compile_strategy()` 完全没有读取 `tc.symbols`。

4. **Schema 设计与实现脱节**：[统一策略配置Schema.md §3.3](file:///d:/lcProject/stock-pulse/sdlc/prd/004-策略管理/统一策略配置Schema.md) 说"缺省继承 screen，→ run_backtest(symbols=)"，但两���都没实现。

### 当前唯一真实作用

- 编辑器摘要展示文案（"标的 510300.SH"）。
- 落库配置里留个标记。

### 静默不一致风险（最危险）

用户在「买入标的」填 `510300.SH`，但 watcher 实际传入的 `kline_data` 是 `{"000001.SZ": [...]}`，结果回测偷偷交易 `000001.SZ`，**不报错、不警告**，用户却以为在交易 510300。

### 处置候选方案

- **A · 删除字段**（phase-1 推荐）：从 editor.html / strategy-editor.js / TradingConfigModel / Schema 一并移除；回测标的永远由 watcher 的 `kline_data` key 决定。
- **B · 落实 Schema**：runner.py 改为 `symbols` 非空时过滤 `data_map`，空则用 `data_map.keys()`；validator 加 "symbols 必须在 kline_data 里" 校验。（适合 phase-2 portfolio 场景）
- **C · 标注为展示字段**：Schema 明确标"不参与回测"，placeholder 同步改写。

## P4 · `csi300` / `csi500` 选股时静默降级为全市场 ⭐（已确认可对接）

### 现状

选���范围选「沪深300」或「中证500」，实际跑的是全 A 股。

### 证据

- [ScreenerServiceImpl.resolveUniverse](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/ScreenerServiceImpl.java)（约 589-594 行）：`csi300` / `csi500` 分支当前降级为 `all_a_shares` 处理，仅打 warn 日志。
- 原因：成分股表未建（`index_weight` 链路整条缺失）。

### 影响

用户选了沪深300，实际选股范围是全市场，**运行时行为与用户预期不符**，且只有 warn 日志，前端无感知。

### 数据来源确认

- **tushare 提供 `index_weight` 接口**：按交易日发布沪深300/中证500 等指数的完整成分股清单 + 权重（每日全量快照）。
- **积分门槛 2000 分**：已确认当前 token 满足（项目已成功调用 `daily`/`adj_factor`/`daily_basic` 等同需 2000 分的接口）。
- **成分股会定期变化**：沪深300/中证500 每年 **6 月、12 月**定期调样，遇退市/合并/分拆也会临时调整 → 必须**定期拉取最新交易日快照**，不能一次性永久存储。
- **相关指数代码**：`000300.SH`(沪深300) / `000905.SH`(中证500) / `000016.SH`(上证50) / `000852.SH`(中证1000)。代码与股票不会冲突，A 股沪市 `000` 段为指数保留段。

### 当前缺失的完整链路

```
tushare index_weight 接口  ← 数据可得（积分够）
        │
        ▼  ❌ 缺 TushareApiEnum.INDEX_WEIGHT 枚举
TushareClient.indexWeight()  ← ❌ 缺这个方法
        │
        ▼  ❌ 缺 IndexWeightDTO / IndexWeightQueryDTO
index_weight 表 + DO / Mapper  ← ❌ schema 里没有这张表
        │
        ▼  ❌ 缺 IndexWeightService（含定期拉取）
ScreenerServiceImpl.resolveUniverse()  ← 现在直接降级全市场 + warn
```

### 数据流转架构（正常量化系统的标准设计）

成分股数据从数据源到应用的完整链路：

```
┌───────────────────────────────────────────────┐
│ 应用层                                         │
│   实时选股: getLatestConstituents() 取最新     │
│   回测:     getConstituentsAt(date) 按日期取   │
│   指数增强: 用 weight 列算跟踪误差(可选)       │
│   ↑ 按需查询                                   │
├───────────────────────────────────────────────┤
│ 服务层: IndexWeightService                     │
│   ↑ 提供按日期查询的 API                       │
├───────────────────────────────────────────────┤
│ 存储层: index_weight 表（每日快照）            │
│   索引: (index_code, trade_date)               │
│   ↑ 定时任务每日同步                           │
├───────────────────────────────────────────────┤
│ 采集层: TushareClient.indexWeight()            │
│   ↑ 从 tushare 拉                             │
├───────────────────────────────────────────────┤
│ 数据源: tushare（搬运自中证指数公司 CSI）      │
│   ↑ 中证指数公司官方判定成分股                 │
└───────────────────────────────────────────────┘
```

**关键认知**：成分股是中证指数公司官方发布的，不需要你"计算"。你要做的是**同步存储 + 按场景查询**。

### ⚠️ 幸存者偏差（回测最大的坑）

回测沪深300 策略时，**必须按每个 bar 的日期取对应的历史快照**，不能全程用"当前最新"的成分股。

**错误做法（幸存者偏差）**：
```
回测 2024-01 到 2024-12 的沪深300 策略
    │
    ▼  全程用 2026-07 的最新成分股
某股票 X 在 2024-06 被调入沪深300，2024-01 时还不在中国
    │
    ▼
错误地用 X 参与了 2024-01 ~ 2024-05 的回测（当时它根本不是成分股）
    │
    ▼
回测结果失真（幸存者偏差：用"后来才入选"的股票回测历史）
```

**正确做法（按日期查询）**：
```
回测 2024-01 到 2024-12 的沪深300 策略
    │
    ▼  遍历每个交易日 bar
2024-06-10（假设这天调样）
    │  查 WHERE index_code='000300.SH' AND trade_date <= '2024-06-10'
    │  ORDER BY trade_date DESC LIMIT 1
    │  → 得到调样后的新成分股
    ▼
2024-06-09（调样前）
    │  查 WHERE index_code='000300.SH' AND trade_date <= '2024-06-09'
    │  ORDER BY trade_date DESC LIMIT 1
    │  → 得到调样前的老成分股
    ▼
回测结果准确反映"当年真实能买到的股票"
```

> 这是补齐 P4 时**最容易遗漏的深层坑**：即使建了 `index_weight` 表，如果 [ScreenerServiceImpl.resolveUniverse](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/ScreenerServiceImpl.java) 回测时只查"最新"快照而不按 bar 日期查历史快照，回测结果仍然是错的。Checklist 第 9 步已把"回测场景按日期查询"列为强制要求。

### 风险提示

- **成分股有滞后性**：tushare 的 `index_weight` 一般 T 日盘后才有 T 日的成分快照，回测时要确认用的快照日期 ≤ 当前 bar 日期，避免未来函数。
- **权重的处理**：`index_weight` 返回的 `weight` 是成分股权重（百分比），选股池若只取成分股列表则忽略 weight；若要做指数增强/跟踪误差，才需要用 weight。
- **历史回补**：首次对接时，用 tushare 的 `start_date`/`end_date` 参数拉取历史全量成分股快照（覆盖回测所需的全部时间段），否则历史回测无数据可查。

## P5 · `BacktestServiceImpl.extractSymbols` 兜底分支路径疑似 bug

### 现状

回测提取标的的兜底分支读了不合 Schema 的路径。

### 证据

- [BacktestServiceImpl.extractSymbols](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/BacktestServiceImpl.java)（约 558-571 行）：

  ```java
  if (out.isEmpty()) {
      JSONObject universe = configJson.getJSONObject("universe");
      if (universe != null) {
          addAllStrings(universe.get("stocks"), out);  // ← 路径疑似错误
      }
  }
  ```

- 按 Schema，`universe` 是 `screen_config` 下的**字符串字段**，`stocks` 在 `screen_config.stocks` 下。这段代码读的是顶层 `universe.stocks`，路径不对。

### 影响

兜底分支几乎不会命中，实际回测只能依赖 `trading_config.symbols`。与 P2 叠加，回测标的来源更脆弱。

### 处置方向

- 确认正确路径后修正（应为 `screen_config.stocks`）；或
- 若 P2 采纳"删除 symbols"，则本兜底分支也需重新设计。

## P7 · 调仓页面英文文案 + 枚举选项未走常量机制

### 现状

### ⚠️ 核心原则：展示层改中文，数据层保持英文 code

| 层 | 例子 | 改不改 |
|---|---|---|
| **展示层**（`<option>`文本、`<label>`文案） | 用户看到的 `monthly` / `frequency` | ✅ 改中文 |
| **数据层**（`<option value="...">`、JS 默认值、后端 JSON） | 存进配置的 `monthly` / `frequency` | ❌ 保持英文 code |

> 关键：`<option value="monthly">每月</option>` —— value 保持 `monthly`（给后端存），展示文本改成"每月"（给用户看）。混在一起改会导致后端收到中文，engine 的 Pydantic `Literal["daily","weekly","monthly","quarterly"]` 校验直接失败。

## P8 · 回测参数页面：日期默认值含义不明 + 字段名中英混杂

### 现状

策略编辑器 Tab8「回测参数」页面存在两个问题：① 起始/结束日期进入页面时显示 `yyyy/mm/日` 占位符，用户不明含义；② 多个字段 label 是"中文+英文变量名"的混合形式。

### 证据

- Tab8 完整 HTML：[editor.html L398-442](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/resources/templates/quant/strategies/editor.html#L398-L442)。
- 日期字段：[L404-L405](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/resources/templates/quant/strategies/editor.html#L404-L405) 的 `<input type="date" id="f-bt-start">` / `<input type="date" id="f-bt-end">`，浏览器原生 date 控件未赋值时会显示 `yyyy/mm/日` 灰色占位。
- JS 默认值逻辑：[strategy-editor.js L813-L814](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/resources/static/js/strategy-editor.js#L813-L814) 用 `bt.start_date || ''` 回填，空字符串不会清除原生占位符。
- DEFAULT_CONFIG（[L69-L72](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/resources/static/js/strategy-editor.js#L69-L72)）的 `backtest_config` 没有设 `start_date`/`end_date` 默认值，所以新建策略时日期字段始终是空的 → 显示原生 `yyyy/mm/日` 占位。

### 问题 1：日期默认值含义不明

`<input type="date">` 在未赋值时，浏览器会显示灰色的 `yyyy/mm/日`（或类似格式，取决于浏览器语言）。这个占位符：
- 不说明"留空表示什么"（实际上留空 = 用数据全集，但用户不知道）；
- 格式不统一（有的浏览器显示 `yyyy/mm/dd`，有的 `年/月/日`）；
- 对非技术用户完全不明含义。

**处置建议**：
- 保持默认值为空（符合用户诉求"默认值为空就行"），但**加 placeholder 说明**或**加 help text** 解释"留空 = 使用数据全集起止"。
- 技术方案：`<input type="date">` 原生不支持 placeholder，可改为：
  - 方案 A：在 label 下方加一行小字说明"留空表示使用全部可用数据"；
  - 方案 B：改用 `<input type="text">` + 日期选择器组件（如 Bootstrap Datepicker），支持自定义 placeholder。
