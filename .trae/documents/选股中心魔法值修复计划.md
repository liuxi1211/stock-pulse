### 4. 新建 `FactorKey.java`（基本面因子 key 常量类，**非枚举**）

15 个因子 key（PE_TTM/PB/PS_TTM/DV_RATIO/TOTAL_MV/CIRC_MV/TURNOVER_RATE/ROE_TTM/ROA_TTM/GROSS_MARGIN/NETPROFIT_MARGIN/REVENUE_YOY/PROFIT_YOY/DEBT_TO_ASSETS/EPS_YOY）。

**为什么用常量类而不是 DisplayableEnum**：这些 key 是 engine Python Schema 的 `Literal[...]`（见规则 §四），watcher 仅作为 Map 的 key 透传给 engine，不需要 code↔label 双值映射。用 `public static final String` 常量类即可，避免过度设计。

```java
public final class FactorKey {
    private FactorKey() {}
    public static final String PE_TTM = "PE_TTM";
    public static final String PB = "PB";
    public static final String PS_TTM = "PS_TTM";
    // ... 其余 12 个
}
```

用法：`putIfNotNull(m, FactorKey.PE_TTM, db.getPeTtm());`

## 三、Assumptions & Decisions（假设与决策）

- **D1**：8 个新 DisplayableEnum 枚举的 `label` 用中文（符合现有 `BoardEnum` 范式，供前端 `/constants` 展示）。
- **D2**：`ScreenLockStatusEnum` 的 code 保持大写 `TRACKING`/`DONE`（与现有 DB 数据一致，避免数据迁移）。`UniverseEnum` 等 code 保持原小写值（与 engine Python Schema 一致）。
- **D3**：15 个因子 key 用常量类 `FactorKey`（非枚举），因为它们是 engine 透传的 Map key，无需 code↔label 映射。若后续前端要展示「因子中文名」，再升级为 DisplayableEnum。
- **D4**：JSON 字段名抽 `ScreenConfigFields`。这层是「防御性抽取」，避免 schema 漂移。**如果你倾向最小改动，可只抽跨文件复用的 `universe`/`stocks`，其余保留字面量**（JSON key 与业务魔法值性质不同）。
- **D5**：错误消息文案（如 `"manual 候选池未配置 stocks 列表"`）**不抽常量**，它们是用户可见的中文描述，频繁调整，抽常量反而降低可读性。
- **D6**：`switch case 5/10/20` 保留字面量（Java 语法要求 case 为编译期常量）。
- **D7**：`MAX_CONDITION_DEPTH=32` 移到 `ScreenerConstants`（它是业务约束阈值，非纯工具）。
- **D8**：前端 Thymeleaf/JS 的改造**不在本计划范围**（仅后端 Java）。但新增的 8 个枚举会自动出现在 `/constants`，前端可后续切换。
- **D9**：不动 DO/DTO 的字段类型（`status`/`universe` 仍是 String），仅替换字面量引用为枚举的 `getCode()`。避免 MyBatis 映射和前端契约回归风险。
- **D10**：范围限定在 `ScreenerServiceImpl` 与 `ScreenConfigValidator` 两个文件 + 新建常量文件。不触及 `ScreenerClient`/DTO/VO/Mapper。

## 四、Verification（验证步骤）

1. **编译**：`mvnw compile -pl stock-watcher`（或项目实际构建命令），确保无编译错误。

3. **接口验证**：`GET /constants` 返回里应包含这 8 个新 key，每个有正确的 code/label 列表。

   - 运行方案：确认 candidates 拼装、基本面 factorKey、涨跌停判定、收益计算精度无变化。
   - 锁定结果：确认 `status` 落库为 `"TRACKING"`，追踪后变 `"DONE"`。
   - 条件树校验：传嵌套 AND/OR + compare 叶子，确认递归校验正常。
5. **静态扫描**：在两个改造文件里 grep 裸字符串/数字，确认无残留（除错误消息文案、switch case、JSON 序列化场景）。
