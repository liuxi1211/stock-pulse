#### A4. 数据兼容性处理（关键）

旧策略的 `config_json` 里可能仍含 `scope` 和 `trading_config.symbols`。由于 engine 的 `StrategyConfigModel` 用 `extra="forbid"`，直接反序列化会报错。

## 五、阶段 D · index_weight 全链路对接（P4）⭐ 核心工程

### 决策
- **指数范围**：沪深300 (000300.SH) / 中证500 (000905.SH) / 上证50 (000016.SH) / 中证1000 (000852.SH)，共 4 个指数。
- **回测集成**：**按 bar 日期查历史快照**（完全防幸存者偏差）。
- **实时选股**：取最新交易日快照。

### D1. 数据采集层

- `IndexWeightQueryDTO`：查询参数 `indexCode` / `tradeDate` / `startDate` / `endDate`。

- 新增方法 `public List<IndexWeightDTO> indexWeight(IndexWeightQueryDTO query)`，采用风格 A（QueryDTO + 私有 build 方法），参考 L50-53 的 `daily` 方法范式。

**[application.yml](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/resources/application.yml)** L44-58

### D2. 存储层

**[schema-sqlite.sql](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/resources/schema-sqlite.sql)** + **schema-mysql.sql**
- 新增 `index_weight` 表：
  ```sql
  CREATE TABLE IF NOT EXISTS index_weight (
      ts_code    TEXT NOT NULL,   -- 指数代码
      trade_date TEXT NOT NULL,   -- 快照日期 yyyyMMdd
      con_code   TEXT NOT NULL,   -- 成分股代码
      weight     REAL,
      PRIMARY KEY (ts_code, trade_date, con_code)
  );
  CREATE INDEX IF NOT EXISTS idx_index_weight_date ON index_weight (ts_code, trade_date);
  ```

**新增 DO + Mapper**（`entity/` + `mapper/` 包）
- `IndexWeightDO`：字段对齐表结构。
- `IndexWeightMapper`：
  - 基础 CRUD（批量 insert）。
  - `selectLatestConstituents(tsCode)`：取该指数最新交易日的成分股代码列表（实时选股用）。
  - `selectConstituentsAt(tsCode, tradeDate)`：`WHERE ts_code=? AND trade_date <= ? ORDER BY trade_date DESC LIMIT 1`（回测防幸存者偏差���）。

### D6. 业务层改造（核心）

**[ScreenerServiceImpl.resolveUniverse](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/ScreenerServiceImpl.java)** L588-595
- 删除"降级为全市场 + warn"的 else 分支。
- 新增 csi300/csi500 分支：
  ```java
  String indexCode = mapUniverseToIndexCode(universe); // csi300→000300.SH 等
  List<String> constituents = indexWeightService.getLatestConstituents(indexCode);
  stocks = stockBasicMapper.selectList(new LambdaQueryWrapper<StockBasicDO>()
          .in(StockBasicDO::getTsCode, constituents)
          .eq(StockBasicDO::getListStatus, ListStatusEnum.LISTED));
  ```
- 新增 universe→indexCode 映射方法（或扩展 `UniverseEnum` 加 indexCode 属性）。
- 上证50/中证1000 若要支持，需扩展 `UniverseEnum` 新增 `csi50`/`csi1000` 枚举值（可选，本期 universe 下拉暂只暴露 csi300/csi500，但数据采集覆盖 4 个指数备用）。

**[BacktestServiceImpl.java](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/BacktestServiceImpl.java)** 回测路径
- 构造 `kline_data` 时，若 universe 为 csi300/csi500：**按回测起止日期区间，逐 bar 日期查历史成分股快照**，拼装每个 bar 对应的可交易标的集合。
- ⚠️ 这是防幸存者偏差的关键。实现思路：
  1. 取回测区间内所有交易日列表（从 trade_cal 表）。
  2. 对每个交易日，调 `indexWeightService.getConstituentsAt(indexCode, date)` 得当日成分股。
  3. 批量查这些成分股的当日日线，拼进 `kline_data`（按 bar 日期组织）。
- 由于 akquant 的回测是"每根 bar 一个 symbol"的事件模型，多标的池回测的 kline_data 组织方式需确认（可能需要 watcher 侧做"每日标的集合变化"的处理）。

> ⚠️ **D6 回测集成是最复杂的部分**，可能需要单独的技术设计。建议先完成 D1-D5（数据采集入库 + 实时选股改造），验证数据正确性后，再单独迭代 D6 回测集成。可在执行阶段拆分为 D6a（实时选股）+ D6b（回测按日期查）两个子任务。

1. **D6 回测集成复杂度高**：多标的池的 kline_data 组织方式需深入确认 akquant 的多标的回测机制。建议拆为 D6a（实时选股）+ D6b（回测按日期查）两步交付。
2. **旧数据兼容**：删除字段后，存量策略的 config_json 必须能被正确加载，白名单过滤逻辑需覆盖所有反序列化入口。
3. **D5 定时任务幂等**：index_weight 批量 insert 需处理重复主键（upsert 或 INSERT OR IGNORE）。
4. **前端联动改造**：删除 scope 后，Tab7 的启用逻辑需重新设计，避免 portfolio 策略无法配置 rebalance。
5. **P4 历史回补数据量**：4 个指数 × 5 年 × 约 1200 个交易日 × 约 300-500 成分股，数据量较大，初始化可能耗时，需评估限流和分批策略。
