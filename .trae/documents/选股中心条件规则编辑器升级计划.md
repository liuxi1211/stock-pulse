# 选股中心条件规则编辑器升级计划

## 一、摘要 (Summary)

升级多因子选股中心的**条件规则编辑器**，补齐三项缺失能力：① 因子 vs 因子比较；② 因子参数选择（MA5/MA10）；③ 条件嵌套分组（任意深度 AND/OR）。

三个需求的本质都是**前端 UI 落地问题**——后端 Pydantic Schema（`condition.py` 的 `ExpressionNode` + `ConditionTree`）与 `ConditionEngine` 已完整支持因子vs因子、`params`、递归嵌套，无需改动求值逻辑。

附带修复核查中发现的 **watcher→engine camelCase/snake_case 不匹配 bug**（`topN`/`filters` 全套/`candidates.*.meta` 静默失效）。

参照原型：`.trae/specs/003-multi-factor-screener/prototype/app.js`，UI 保持现有 Bootstrap 5 风格。

### 2.2 后端 Schema（已完整，无需改求值逻辑）

[condition.py](file:///d:/lcProject/stock-pulse/stock-engine/models/schemas/condition.py)：
- `ExpressionNode`：4 形态 union（`value`/`factor`+`params`+`inputs`+`outputIndex`/`op`+`left`+`right`/`ref`）
- `CompareLeaf`：`{type:"compare", left:ExpressionNode, comparator, right:ExpressionNode}` —— 左右对称，天然支持因子vs因子
- `ConditionTree`：`{operator, conditions:[ConditionTree|CompareLeaf]}` —— 递归，天然支持任意嵌套

[engine.py](file:///d:/lcProject/stock-pulse/stock-engine/services/screener/engine.py)：
- `_eval_compare` 左右对称处理 ExpressionNode
- `_eval_node`/`_eval_logic` 递归 AND(all)/OR(any)
- `validate_cross_section` 已禁用时序比较器（选股场景）

### 2.3 已存在的命名映射 Bug（顺带修复）

watcher→engine 全链路零 camelCase↔snake_case 转换，Pydantic v2 默认严格按字段名匹配 → 未匹配字段静默忽略用默认值：

| 层级 | watcher(camelCase) | engine(snake_case) | 影响 |
|---|---|---|---|
| 顶层 | `topN` / `verboseExcluded` | `top_n` / `verbose_excluded` | topN=30 失效；不返回 excluded |
| filters | `excludeSt`/`excludeSuspended`/`excludeLimitUp`/`excludeLimitDown`/`minListDays`/`excludeIndustries` | `exclude_st`/... | 用户自定义过滤全失效，用默认 |
| candidates | `ohlcvHistory` | `ohlcv_history` | 技术面因子算不出 |
| candidates.meta | `isSt`/`isSuspended`/`isLimitUp`/`isLimitDown`/`listDate` | `is_st`/...（filters.py 用 .get） | 过滤判定全失效 |

> `conditions` 树内的 `outputIndex` 是 engine 唯一 camelCase 字段，与前端一致，**不出问题**；`params` 内部 key（timeperiod/fastperiod 等）全是单词，无歧义。

#### 4.2.1 内部状态改为持有 ConditionTree

```javascript
const state = {
    factors: [], categories: [], plans: [], currentPlanId: null, lastResult: null,
    activeTab: 'conditions',
    tree: { operator: 'AND', conditions: [] },   // 根 ConditionTree（无 type 字段，engine 根节点不要求 type）
```

> engine `ConditionTree` 不要求 `type` 字段；`ConditionNode = Union[ConditionTree, CompareLeaf]`，靠 `operator`/`comparator` 字段区分。前端内部状态树：分组节点 `{operator, conditions:[...]}`，叶子节点 `{type:'compare', left, comparator, right}`。

#### 4.2.2 ExpressionNode 构造/收集辅助

```javascript
// 默认叶子：左因子 vs 右数值
function defaultLeaf() {
    return {
        type: 'compare',
        left: { factor: 'PE_TTM', params: {}, outputIndex: 0 },
        comparator: '<',
        right: { value: 30 },
    };
}

// 从 DOM 表单收集单个操作数（side = 'left' | 'right'）
function collectOperand(blockEl) {
    const type = blockEl.querySelector('.op-type').value;   // 'factor' | 'value'
    if (type === 'value') {
        const raw = blockEl.querySelector('.op-value').value;
        return { value: parseFloat(raw) };
    }
    const factor = blockEl.querySelector('.op-factor').value;
    const params = {};
    blockEl.querySelectorAll('.op-param').forEach(inp => {
        params[inp.dataset.name] = inp.value === '' ? null : Number(inp.value);
    });
    const outIdxSel = blockEl.querySelector('.op-output');
    const outputIndex = outIdxSel ? Number(outIdxSel.value) : 0;
    return { factor, params, outputIndex };
}

// 回填操作数到 DOM
function applyOperand(blockEl, node) { /* 反向，用于 applyConfig */ }
```

#### 4.2.3 递归渲染 `renderNode(node, path)`

- 分组节点：渲染 `.group-node`（带 `operator-badge` 点击切换 AND/OR、+条件/+分组/删除分组按钮），递归渲染 `children`
- 叶子节点：渲染 `.leaf-node`（左操作数块 + 比较器 + 右操作数块 + 删除按钮）

#### 4.2.4 操作数块 `renderExpression(node, side, pathStr)`

参照原型图 `renderExpression`，左右对称：
- `type-select`（因子/数值）
- factor 形态：因子下拉（按 category 分组 optgroup，复用 `buildFactorOptions`）+ 参数输入区（按所选 factor 的 `params` 元数据动态渲染 number input）+ 多输出因子 outputIndex 下拉（仅 `multiOutput=true` 显示）
- value 形态：单个 number input

#### 4.2.5 因子元数据辅助

复用现有 `loadFactors()` 拉到的 `state.factors`（每项含 `factorKey`/`displayName`/`category`/`params`/`multiOutput`/`outputLabels`/`defaultOutputIndex`）。新增：

```javascript
function getFactorDef(key) { return state.factors.find(f => f.factorKey === key); }
function getDefaultParams(def) {
    const p = {};
    (def && def.params || []).forEach(x => { p[x.name] = x.defaultValue; });
    return p;
}
```

> ⚠️ 需核对 `/factors` 返回的字段名是否为 `factorKey`（现有 `buildFactorOptions` 用 `f.factorKey`），以及是否含 `params`/`multiOutput`/`outputLabels`/`defaultOutputIndex`。若字段名与原型图（`key`/`params`/`multiOutput`/`outputLabels`/`defaultOutputIndex`）有出入，在辅助函数里做一次映射。

#### 4.2.6 节点寻址与操作（path 数组）

`nodeAt(path)` / `parentAt(path)` / `toggleOperator(path)` / `addLeaf(path)` / `addGroup(path)` / `removeNode(path)` / `updateLeaf(path, side, field, value)` —— 直接移植原型图同款函数，状态源从 `getActivePlan().tree` 改为 `state.tree`。

#### 4.2.7 事件委托（在 `conditionTree` 容器上）

- `click [data-action]` → toggleOperator/addLeaf/addGroup/removeNode
- `change [data-field]` → 更新对应字段；其中 `type` 切换需重渲染该叶子（参数区/数值输入互换）；`key` 切换需重置 params 为该因子默认值

每次变更后调 `renderTree()` 重绘（与原型图一致，简单可靠；树规模小无性能问题）。

#### 4.2.8 `collectConfig()` 改造

```javascript
function collectConfig() {
    // ... universe 不变
    const conditions = collectNode(state.tree);   // 递归收集为 engine ConditionTree 结构
    // filters / ranking / topN 收集逻辑不变（但注意 topN 仍发 camelCase，靠 4.3 修复映射）
    return { universe, conditions, ranking, filters, topN };
}

function collectNode(node) {
    if ('operator' in node) {
        return {
            operator: node.operator,
            conditions: (node.conditions || []).map(collectNode).filter(Boolean),
        };
    }
    // 叶子
    const leftEl = document.querySelector(`.leaf-node[data-path="${node.__path}"] .op-block.left`);
    const rightEl = ...;
    return {
        type: 'compare',
        left: collectOperandFromDom(node.__path, 'left'),
        comparator: node.comparator,
        right: collectOperandFromDom(node.__path, 'right'),
    };
}
```

> 实现细节：为避免 DOM 与状态双向同步复杂度，采用「状态优先 + 每次变更写回 state + 收集时直接读 state」策略（原型图即此模式）。`change` 事件把表单值写回 `state.tree` 对应节点；`collectConfig` 直接序列化 `state.tree`。

#### 4.2.9 `applyConfig(cfg)` 改造

收到 `cfg.conditions`（engine ConditionTree 结构）后，递归构建 `state.tree`，再 `renderTree()`。需兼容旧数据（若 `conditions` 是旧的扁平叶子数组或 `{operator, conditions:[{type:'compare',left:{factor},comparator,right:{value}]}`）。

#### 4.2.10 暴露 API

`window.Screener` 增加：`addLeaf`、`addGroup`、`removeNode`、`toggleOperator`（供 HTML onclick 调用）；保留 `addCondition`（内部转发 `addLeaf([])` 以向后兼容，若有其他地方调用）。

## 七、风险与备注

- **fastjson2 注解兼容性**：fastjson2 自 2.0.x 起兼容 Jackson 注解，但需以冒烟测试为准；不兼容则改 `@JSONField`。
- **`/factors` 返回字段名**：需核对 `state.factors` 元素是否含 `params`/`multiOutput`/`outputLabels`/`defaultOutputIndex`；若缺失或不一致，参数 UI 无法驱动——实施时先 `console.log(state.factors)` 确认，必要时在 engine `/factors` 端点补字段（但这属于 engine 改动，需重新评估范围）。
- **旧方案数据兼容**：`applyConfig` 需处理历史扁平条件结构，避免已有方案加载崩溃。
- **嵌套深度**：UI 不设硬性层级上限（engine 也无限制），但建议 CSS 上限制缩进避免极深嵌套布局崩坏。
