# 代码评审器

本技能用于指导智能体针对本地开发代码、远程拉取请求开展专业且全面的代码评审工作。

## 工作流程

### 1. 确定评审对象
*   **远程PR**：若用户提供PR编号或URL（例如“评审 PR #123”），则以该远程拉取请求为评审目标。
*   **本地变更**：若未指定具体PR，或用户要求“评审我的变更”，则以当前本地文件系统状态（暂存区与未暂存变更）为评审目标。

### 2. 评审准备

#### 远程PR场景：
1.  **检出代码**：使用 GitHub CLI 工具检出对应PR分支。
    ```bash
    gh pr checkout <PR编号>
    ```
2.  **前置校验**：执行项目标准校验套件，提前排查自动化检测可发现的故障。
    ```bash
    npm run preflight
    ```
3.  **明确上下文**：阅读PR描述与已有评论，理解变更目标与历史评审记录。

#### 本地变更场景：
1.  **识别变更范围**：
    *   查看状态：`git status`
    *   读取差异：`git diff`（工作区变更）和/或 `git diff --staged`（暂存区变更）。
2.  **前置校验（可选）**：若变更规模较大，可询问用户是否需要在评审前执行 `npm run preflight`。

### 3. 深度分析
围绕以下核心维度对代码变更展开分析：

*   **正确性**：代码是否能实现预期目标，不存在缺陷或逻辑错误？
*   **可维护性**：代码是否整洁、结构清晰，便于后续理解与迭代修改？需综合考量代码清晰度、模块化程度，以及是否遵循既定设计模式等因素。
*   **可读性**：代码是否在必要位置补充了注释，格式是否统一规范，是否符合项目编码风格要求？
*   **执行效率**：本次变更是否引入了明显的性能瓶颈或资源浪费问题？
*   **安全性**：是否存在潜在安全漏洞，或采用了不安全的编码实践？
*   **边界场景与错误处理**：代码是否妥善覆盖了边界场景，对潜在异常是否有合理的处理逻辑？
*   **可测试性**：新增或修改的代码是否具备充足的测试覆盖（即便前置校验已通过）？可提出补充测试用例的建议，以提升测试覆盖度与代码健壮性。

### 4. 输出评审反馈

#### 内容结构
*   **评审总结**：对本次评审的高层级概览。
*   **评审发现**：
    *   **严重问题**：代码缺陷、安全隐患或破坏性变更。
    *   **优化建议**：提升代码质量、性能的改进方向。
    *   **细微问题**：格式或次要风格问题（可选输出）。
*   **评审结论**：明确给出评审意见（评审通过 / 要求变更）。

#### 表达语气
*   保持建设性、专业与友善的态度。
*   解释提出修改建议的原因与依据。
*   对于评审通过的内容，明确认可本次贡献的具体价值。

### 5. 收尾操作（仅远程PR场景）
*   评审完成后，询问用户是否需要切回项目默认分支（如 `main` 或 `master`）。
