## §0 背景与问题定性

### 0.1 触发原因

对「策略管理 - 轮动范式」进行未来数据核查，发现三类系统性设计缺陷。前两类（订单叠加、资金争抢）已由 [006-策略范式治理](../006-策略范式治理/策略范式互斥改造PRD.md) 治理；**本报告聚焦 006 未覆盖的第三类缺陷：成分股身份泄露与选股模型分层缺失**。

### 0.2 三大缺陷概述

| # | 缺陷 | 严重度 | 根因层级 |
|---|---|---|---|

| **C** | 「选股范围」与「选股条件」���念边界模糊，缺中间层 | 🟡 中 | Schema 设计层（004 统一 Schema） |

### 0.3 akquant 责任 vs 封装层责任

**核查结论：akquant 0.2.47 框架本身设计完备，问题全部出在 stock-pulse 封装层。**

| akquant 提供 | stock-pulse 现状 |
|---|---|

## §1 缺陷 A：成分股 point-in-time 泄露（核心问题）

### 1.1 现象

watcher 的 `resolveBacktestSymbols`（[BacktestServiceImpl.java:582-621](../../../stock-watcher/src/main/java/com/arthur/stock/service/impl/BacktestServiceImpl.java#L582-L621)）对 `universe="csi300"` 使用 `getConstituentsInRange(indexCode, startDate, endDate)`，取的是**回测区间内所有曾入选成分股的并集**。

```java
List<String> constituents = indexWeightService.getConstituentsInRange(indexCode, startDate, endDate);
```

对应 SQL（[IndexWeightMapper.java:36-43](../../../stock-watcher/src/main/java/com/arthur/stock/mapper/IndexWeightMapper.java#L36-L43)）：

```sql
SELECT DISTINCT con_code FROM index_weight
WHERE ts_code = #{tsCode} AND trade_date BETWEEN #{startDate} AND #{endDate}
```

**后果**：假设回测区间 2022-01-01 ~ 2024-12-31，股票 X 在 2024-06 才新入选 csi300。按并集逻辑，X 会进入整个回测的 universe，engine 在 2022-01-01 的调仓日就能对 X 打分买入——**这是用「2024-06 才知道的成分股身份」做 2022 年的决策**，即 lookahead bias。

### 1.2 影响

| 影响 | 机制 |
|---|---|
| 系统性高估收益 | 新入选成分股往往近期表现强势（动量效应/指数纳入效应），回测早期就能买入这些"事后赢家" |
| 幸存者偏差的逆向 | 退市/被剔除的股票因"区间内曾入选"留在 universe，但其被剔除本身就是负面信号 |
| 回测不可复现 | 不同回测区间会产生不同的并集，同一策略不同区间结果无法横向对比 |

### 1.3 讽刺之处：基础设施已具备，回测路径未启用

watcher **已有** point-in-time 查询能力：

- `index_weight` 表（[schema-mysql.sql:269-277](../../../stock-watcher/src/main/resources/schema-mysql.sql#L269-L277)）按 `(ts_code, trade_date, con_code)` 存储每日成分股快照
- [IndexWeightTask.java](../../../stock-watcher/src/main/java/com/arthur/stock/task/IndexWeightTask.java) 每交易日 20:00 增量同步
- `selectConstituentsAt(tsCode, tradeDate)`（[IndexWeightMapper.java:27-30](../../../stock-watcher/src/main/java/com/arthur/stock/mapper/IndexWeightMapper.java#L27-L30)）可取「≤ 指定日期的最新快照」

**但回测路径调用的是 `getConstituentsInRange`（并集），而非 `selectConstituentsAt`（point-in-time）。**

### 1.4 架构冲突的根因

engine 有硬约束：**不触库、不回调 watcher**（[rebalance_engine.py:14-17](../../../stock-engine/services/backtest/rebalance_engine.py#L14-L17)）。而 point-in-time 过滤需要「每个调仓日动态查询当日成分股」，这恰恰需要 engine 在回测运行中获取外部数据——与硬约束冲突。

当前架构下，universe 在 watcher 侧 HTTP 请求时一次性构造完成，engine 拿到的 `{symbol: OHLCV}` 字典是扁平的、无时间维度的标的池。

## §2 缺陷 B：基本面因子静默退化

### 2.1 现象

watcher 的 `buildKlineData`（[BacktestServiceImpl.java:539-570](../../../stock-watcher/src/main/java/com/arthur/stock/service/impl/BacktestServiceImpl.java#L539-L570)）传给 engine 的每根 bar 只含 `{date, open, high, low, close, volume}`，**不含 PE/PB/ROE/市值等基本面字段**。

engine 的 [rebalance_engine.py:367-370](../../../stock-engine/services/backtest/rebalance_engine.py#L367-L370)：

```python
if fd.source == "TUSHARE":
    fundamentals = candidate.get("fundamentals") or {}  # 回测路径默认空
    return RebalanceEngine._to_float(fundamentals.get(factor_key))  # → NaN
```

### 2.2 影响

用户在 `screen_config.conditions` 或 `ranking.weights` 中引用基本面因子（如 `PE_TTM < 20`、`ROE_TTM > 15`），这些因子在回测路径全部算出 NaN：

- 在 `_filter_valid_symbols`（[rebalance_engine.py:386-401](../../../stock-engine/services/backtest/rebalance_engine.py#L386-L401)）阶段，含 NaN 的 symbol 被剔除
- 极端情况下整个 universe 被过滤光，rebalance 静默变成「不调仓」
- **仅日志 warning，用户无感知**

### 2.3 与缺陷 A 的关联

两者同源：都是「watcher → engine 的数据下发不完整」导致的轮动路径功能缺失。缺陷 A 缺的是成分股身份信息，缺陷 B 缺的是基本面截面数据。

## §3 缺陷 C：选股模型分层不足

### 3.1 现状：004 Schema 的两层模型

[统一策略配置 Schema §3.2](../004-策略管理/统一策略配置Schema.md) 的 `screen_config`：

```
screen_config
├── universe      ← 选股范围
├── conditions    ← 选股条件（混合了因子打分与硬筛选）
├── ranking       ← 排序
├── filters       ← 静态过滤（ST/停牌/行业）
└── top_n
```

### 3.2 问题：「选股条件」语义过载

akquant 的 `rebalance_to_topn` 实际执行链路是：

```
scores 字典（用户算好的因子分）
    → min_score / long_only 硬筛选（框架内置）
    → 降序排序选 TopN（框架内置）
    → weight_mode 权重分配（框架内置）
    → 下单（框架内置）
```

对照 004 Schema，「选股条件」（`conditions`）实际承载了两种不同性质的逻辑：

| 子概念 | 代码形态 | 性质 | akquant 对应 |
|---|---|---|---|
| 因子打分公式 | `ranking.weights: {PE_TTM: -0.3}` | 连续数值计算 | scores 字典的构造 |
| 硬性筛选阈值 | `conditions: {PE_TTM < 20}` | 离散布尔判断 | `rebalance_to_topn(min_score=)` |

二者**变化频率、计算方式、akquant 对应点都不同**，混在 `conditions` 一个字段里导致：
- 用户无法区分「这个条件是用来过滤的」还是「参与打分的」
- conditions 既支持 `{factor} < {value}` 又支持 `ranking.weights`，语义重叠
- akquant 的 `min_score`（分数阈值）与 conditions 的硬过滤是两套机制，容易混淆

### 3.3 akquant 视角的正确分层

基于 [rebalance_to_topn 完整签名](../../../akquant-0.2.47/python/akquant/strategy.py#L982-L1075) 与官方横截面策略文档，标准的轮动选股链路应是 **4 层**：

```
① Universe（选股范围）    → 决定数据订阅范围 + point-in-time 成分股过滤
    ↓
② Factor Scoring（因子打分）→ 连续数值计算，对应 scores 字典
    ↓
③ Filter（筛选条件）      → 离散布尔判断，剔除不达标标的（ST/停牌/PE阈值）
    ↓
④ Portfolio（组合构建）   → TopN + 权重模式，对应 rebalance_to_topn
```

akquant 原生覆盖 ③④（通过 `min_score` + `rebalance_to_topn`），①②留给封装层。

## §4 目标与非目标

### 4.1 目标

1. **消除成分股 point-in-time 泄露**：轮动范式的每个调仓日，候选池严格限定为「该日实际入选成分股」
2. **补齐基本面因子数据链路**：PE/PB/ROE/市值等截面因子在轮动路径可用且 point-in-time 正确
3. **选股模型重构为 4 层**：universe / factor / filter / portfolio，清晰对齐 akquant 执行链路
4. **保持 akquant 框架不变**：所有改动在 watcher/engine 封装层 + Schema 层

### 4.2 非目标

- **不改 akquant 源码**
- **不做混合范式**（006 已明确禁止）
- **不改择时范式（signals）链路**（signals 范式 universe ≤ 10 只，无成分股泄露问题）
- **不改选股中心（003）的独立选股 API**（`POST /api/screener/run` 消费 `screen_config` 的链路不变，仅 Schema 字段调整）

## §5 详细改造方案

### 5.1 缺陷 A 修复：放松 engine 边界 + watcher 成分股查询服务

> **决策**（用户确认）：放松 engine 边界约束，允许 engine 在回测期间通过 HTTP 反向查询 watcher 的 point-in-time 成分股快照。

#### 5.1.1 watcher 新增 point-in-time 成分股查询接口（P0）

**新增端点**：`POST /api/internal/constituents/query`

**Request**：
```json
{
  "index_code": "000300.SH",
  "trade_date": "2022-06-15"
}
```

**Response**：
```json
{
  "index_code": "000300.SH",
  "trade_date": "2022-06-15",
  "effective_date": "2022-06-13",
  "constituents": ["000001.SZ", "000002.SZ", ...]
}
```

**实现**：复用现有 `selectConstituentsAt`（[IndexWeightMapper.java:27-30](../../../stock-watcher/src/main/java/com/arthur/stock/mapper/IndexWeightMapper.java#L27-L30)），取 `≤ trade_date` 的最新快照。

**关键设计——生效日对齐**：
- 沪深300/中证500 成分股调整有「公告日」与「生效日」（通常 6 月、12 月第二个星期五的下一交易日生效）
- 接口返回的 `effective_date` 必须是**生效日**，engine 据此判断「今天该用哪期成分股」
- watcher 侧的 `index_weight` 表的 `trade_date` 应存生效日（采集任务 [IndexWeightTask.java](../../../stock-watcher/src/main/java/com/arthur/stock/task/IndexWeightTask.java) 需确认是否已对齐，未对齐则回补）

#### 5.1.2 engine 边界约束修订（P0）

**现状**：[rebalance_engine.py:14-17](../../../stock-engine/services/backtest/rebalance_engine.py#L14-L17) 注释「engine 不触库、不回调 watcher」。

**修订为分层约束**：

| 约束层级 | 内容 | 例外 |
|---|---|---|
| 数据单源性（强） | engine 永远不直接读写业务数据库（SQLite/MySQL） | 无例外 |
| 行情/基本面（强） | 行情与基本面数据由 watcher 预传，engine 不反向拉取 | 无例��� |
| **参考数据（弱，新增例外）** | **成分股身份、指数成分等「参考数据」（reference data）允许 engine 在回测期间按需查询 watcher 的只读内部接口** | 仅限 `/api/internal/*` 只读端点；查询必须幂等、无副作用 |

**新增 engine 侧 watcher client**：

**文件**：`stock-engine/services/backtest/watcher_client.py`（新建）

```python
class WatcherClient:
    """engine → watcher 的只读内部接口客户端（仅限参考数据查询）。"""

    def __init__(self, base_url: str, timeout: float = 5.0):
        self.base_url = base_url.rstrip("/")
        self.timeout = timeout

    def get_constituents_at(self, index_code: str, trade_date: str) -> set[str]:
        """查询 point-in-time 成分股快照。"""
        resp = requests.post(
            f"{self.base_url}/api/internal/constituents/query",
            json={"index_code": index_code, "trade_date": trade_date},
            timeout=self.timeout,
        )
        resp.raise_for_status()
        return set(resp.json().get("constituents", []))
```

    df = pd.DataFrame(records)
    time_col = next((c for c in ("date","trade_date","datetime") if c in df.columns), None)

        if fld in df.columns:
            extra[fld] = df[fld].astype("float64")
    return ohlcv, extra

### 5.3 缺陷 C 修复：选股模型重构为 4 层

#### 5.3.1 新 screen_config 结构（P0）

**文件**：[sdlc/prd/004-策略管理/统一策略配置Schema.md](../004-策略管理/统一策略配置Schema.md) §3.2

将 `screen_config` 重构为 4 层：

```jsonc
{
  "screen_config": {
    "universe": {
      "pool": "csi300",
      "point_in_time": true,
      "stocks": null
    },
    "factor": {
      "method": "composite",
      "weights": { "MOM_20D": 0.4, "ROE_TTM": 0.3, "TOTAL_MV": -0.3 }
    },
    "filter": {
      "conditions": { /* ConditionTree，仅 >, <, == 等截面比较 */ },
      "exclude_st": true,
      "exclude_suspended": true,
      "exclude_limit_up": true,
      "min_list_days": 60
    },
    "portfolio": {
      "top_n": 30
    }
  }
}
```

#### 5.3.2 4 层职责定义

| 层 | 字段 | 职责 | akquant 对应 | 变化频率 |
|---|---|---|---|---|
| ① Universe | `universe` | 选股范围 + point-in-time 成分股过滤 | 决定 `get_history_map` 的 symbols | 低（指数调整半年一次） |
| ② Factor Scoring | `factor` | 因子打分公式，输出连续 score | 构造 `scores` 字典 | 高（每调仓日重算） |
| ③ Filter | `filter` | 硬性筛选（布尔判断），剔除不达标 | 打分前置过滤（scores 只含达标标的） | 中（条件较稳定） |
| ④ Portfolio | `portfolio` | TopN + 权重模式 | `rebalance_to_topn(top_n, weight_mode)` | 低 |

#### 5.3.3 与 trading_config.rebalance 的边界

`trading_config.rebalance` 保持原职责（调仓频率、weight_mode、long_only），但 `weight_mode` 与 `factor` 的关系明确化：

- `rebalance.weight_mode="equal"` → 等权，忽略 factor 分数
- `rebalance.weight_mode="score"` → 按 `factor` 计算的 score 加权

`rebalance.long_only` / `min_score` 作为 factor 之后的**最终防线**（正分过滤），对应 `rebalance_to_topn(long_only=True, min_score=0)`。

#### 5.3.4 旧 Schema 兼容迁移

**决策**（与 006 一致）：不兼容，存量清零。

- `screen_config.conditions` → `screen_config.filter.conditions`
- `screen_config.ranking` → `screen_config.factor`
- `screen_config.filters` → `screen_config.filter`（合并）
- `screen_config.top_n` → `screen_config.portfolio.top_n`
- `screen_config.universe`（string）→ `screen_config.universe.pool`（object）

## §6 数据流向（改造后）

### 6.1 轮动范式完整链路

```
用户配置轮动策略（4 层 screen_config）
    │
    ▼
watcher resolveBacktestSymbols
    │  查 index_weight 区间并集 → 全量候选标的（含历史曾入选）
    ▼
watcher buildKlineData（改造后）
    │  对每只标的：OHLCV + daily_basic（PE/PB/市值等）按日期 join
    │  HTTP 传 engine，header 带 X-Watcher-Base-Url
    ▼
engine run_backtest_engine
    │  kline_to_df_with_extra：OHLCV → DataFrame，基本面 → extra map
    │  构造 WatcherClient(base_url)
    ▼
aq.run_backtest → on_daily_rebalance(trading_date)
    │
    ├─ ① Universe 过滤（新增）
    │    WatcherClient.get_constituents_at("000300.SH", str(trading_date))
    │    → 当日实际入选成分股集合
    │    → 从 kline_map 剔除非当日成分股
    │
    ├─ ② Factor 打分
    │    对当日候选池：技术面用 akquant.talib，基本面从 extra 取
    │    → scores: {symbol: float}
    │
    ├─ ③ Filter 筛选
    │    评估 filter.conditions（截面布尔树）
    │    + exclude_st / exclude_suspended / min_list_days
    │    → scores 只保留达标标的
    │
    └─ ④ Portfolio 构建
         rebalance_to_topn(scores, top_n, weight_mode, long_only, liquidate_unmentioned=True)
         → akquant 默认 T+1 开盘成交
```

### 6.2 关键时序保证（akquant 已具备）

| 环节 | 保证 | 机制 |
|---|---|---|
| `on_daily_rebalance` 取数 | 只见 T-1 及更早 | `_framework_history_cutoff_ns` 截断 |
| `on_daily_rebalance` 时机 | 在 `on_bar` 之前，当日 bar 不可见 | `hide_current_event=True` |
| 成分股查询时点 | 查询的是 `trading_date` 的 point-in-time 快照 | watcher `selectConstituentsAt` |
| 订单成交 | T+1 开盘 | 默认 `fill_policy: open + bar_offset=1` |

## §9 风险与权衡

### 9.1 风险

| 风险 | 等级 | 缓解 |
|---|---|---|
| engine 边界放松后滥用，engine 开始反向拉行情/基本面 | 中 | 分层约束明确「仅限参考数据」；watcher 内部接口加 `/internal/*` 前缀 + 鉴权 |
| point-in-time 查询拖慢回测（每次调仓一次 HTTP） | 低 | 月频调仓可接受；watcher 侧可加缓存（同一 trade_date 的成分股快照不变） |
| 成分股历史快照不完整（早期数据缺失） | 中 | 缺失日降级为「最近可用快照」并打日志；watcher 侧回补 |
| Schema 重构导致前端大量改动 | 中 | 与 006 同批进行；前端已需改范式切换，一并改 4 层 |

### 9.2 关键设计权衡

**权衡 1：放松 engine 边界 vs 预计算下发**

| 方案 | 优点 | 缺点 | 决策 |
|---|---|---|---|
| 放松边界（engine 反查 watcher） | engine 逻辑简单；成分股查询实时 | 增加 engine→watcher 耦合；回测依赖 watcher 可用 | ✅ 采用（用户确认） |
| 预计算下发（watcher 随 kline_data 下发每日成分股集合） | engine 保持无状态 | payload 膨胀（每日×300只×回测天数）；engine 解析复杂 | 备选 |

**权衡 2：4 层 vs 2 层选股模型**

| 方案 | 优点 | 缺点 | 决策 |
|---|---|---|---|
| 4 层（universe/factor/filter/portfolio） | 语义清晰，对齐 akquant 链路 | Schema 改动大；前端需适配 | ✅ 采用（用户确认） |
| 2 层保持 | 改动小 | 「选股条件」语义过载，因子打分与硬筛选混淆 | 否决 |

**权衡 3：weight_mode 的归属**

`weight_mode`（equal/score）放在 `trading_config.rebalance` 还是 `screen_config.portfolio`？
- 决策：**留在 `trading_config.rebalance.weight_mode`**。理由：权重分配是「组合执行」动作，属交易层；`portfolio.top_n` 只决定选几只，不决定怎么分配。

## §11 落地建议

### 11.1 分批策略

3. **降级安全性**：断开 watcher，engine 应降级执行（跳过 point-in-time 过滤 + warning），不崩溃
