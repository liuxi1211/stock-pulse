## 0. 拆分说明（为什么单独立项）

总计划文档把自选股补齐混排在现有 3 模块补齐清单里，与仪表盘（D1-D5）、行情中心（S1-S8）同列。本文把 **自选股 10 项功能补齐**（W1-W10）单独拎出独立成稿，因为：

- **自选股定位独特**：路由 `/page/watchlist` 是项目「用户专属管理」入口，区别于行情中心的全市场列表视角--自选股是老股民私有的关注股池，围绕「我的自选」做分组、提醒、备注、批量管理等**个性化管理**操作，10 项补齐围绕同一用户场景（管理自选股池 + 价格提醒 + 下钻诊断），天然是一组。
- **10 项补齐相互关联**：W1 表结构扩展是 W2 价格提醒 / W3 分组管理 / W7 批量改分组 的共同前置；W3 分组与 W6 排序、W7 批量操作 共享同一张自选股表格与行交互；W8 mini K 线 / W9 列扩展 共享同一组数据列定义；W10 脚本独立化是所有前端改造的基础工程。单独立项便于统一表格结构与交互。
- **数据依赖集中**：W1 扩展 `sys_watchlist` 表（项目自有表）；W2/W6/W8 共同依赖 `daily_quote` 表（行情 + 30 日收盘序列）；W9 依赖 `daily_basic` 表（PE/PB/换手率/市值）；W3 分组可复用 `sw_industry`（按行业自动归类）+ 自定义分组；W4/W5 依赖个股诊断页面路由与 `SearchSuggest` 组件。10 项在数据层面高度耦合。

> **不在本 PRD 范围**：仪表盘补齐（D1-D5）、行情中心补齐（S1-S8）、板块行情页面本身（015）、个股诊断页面本身（017）--各自独立排期；本 PRD 只描述自选股页面 `/page/watchlist` 的功能补齐。

## 1. 背景与目标

### 1.1 自选股定位

- **用户专属管理入口**：路由 `/page/watchlist`，侧边栏「主要功能」第 3 项，老股民管理个人关注股池的唯一入口。
- **老股民持仓决策的中枢环节**：老股民习惯「把关注的股票加入自选 -> 按行业/主题分组 -> 设价格提醒 -> 盘中提醒触发时查看 -> 点击下钻个股诊断做决策」；自选股是这个使用路径的中枢，上游是行情中心/个股诊断（加入自选），下游是个股诊断（下钻深入分析）。
- **现有页面已实现但功能单薄**：自选股页面骨架已存在（`watchlist.html` + 内联脚本），但仅展示基础行情列表 + 移除按钮，缺 10 项关键能力（详见 §1.2）。当前 `sys_watchlist` 表仅有 `id` / `user_id` / `stock_code` / `created_at` 四个字段，无法承载分组、备注、价格提醒等个性化管理需求。

### 1.2 现状与功能缺口

| # | 现状 | 缺口 |
|---|---|---|
| W1 | `sys_watchlist` 表仅 4 个字段（id/user_id/stock_code/created_at） | 需扩展 5 个字段：`group_id` / `note` / `target_price_high` / `target_price_low` / `sort_order` |
| W2 | 无价格提醒能力 | 需支持设置目标价/涨跌幅阈值，达到时 toast + 顶部通知 |
| W3 | 无分组管理，自选股平铺展示 | 需支持按行业或自定义分组归类，支持拖拽排序 |
| W4 | 股票代码是纯文本，不可点击 | 需改为链接，点击跳转 `/page/stock-detail/{code}` |
| W5 | 添加自选股需切到其他页面 | 需顶部加搜索框，集成 `SearchSuggest` 直接添加 |
| W6 | 列表不可排序 | 需支持按 涨跌幅 / 添加时间 / 字母（股票名称拼音）排序 |
| W7 | 无批量操作，只能逐个移除 | 需支持批量移除 / 批量改分组 |
| W8 | 列表无趋势可视化 | 需在表格右侧加 mini K 线（30 日收盘价 sparkline 折线） |
| W9 | 列表只有基础行情列（代码/名称/价/涨跌额/涨跌幅/量） | 需增加 行业 / 市值 / PE / PB / 换手率 5 列 |
| W10 | 前端脚本内联在 `watchlist.html` 的 `<script>` 标签 | 需抽出独立 `static/js/watchlist.js` 文件 |

### 1.3 非目标（明确不做）

| 非目标项 | 原因 | 备注 |
|---|---|---|
| **实时盘中推送** | WebSocket 推送不在本期范围 | 价格提醒采用前端轮询（每日收盘后数据更新，盘中按固定间隔轮询最新价） |
| **跨账户自选股共享** | 自选股是用户私有数据 | 不做共享/协作/导出给他人能力 |
| **自选股策略回测** | 策略回测归回测中心 | 自选股只做股池管理，不做"自选股组合回测"（那是 005 回测中心的能力） |
| **自选股收益统计** | 组合收益分析归仪表盘 | 不做"自选股组合累计收益曲线"，那是 D 系列仪表盘的能力 |
| **自定义预警条件组合** | 一期只做目标价/涨跌幅阈值 | MACD 金叉提醒、量比突增提醒等复杂条件预警在二期评估 |
| **自选股数据反向写入行情库** | 数据单源约束 | 自选股只读写 `sys_watchlist` 表（项目自有），不写入任何 tushare 行情表 |
| **移动端自选股管理** | 不在本期范围 | 自选股页面是 PC 端专业工具，不做响应式适配 |
| **自选股历史快照** | 不记录每日自选股池快照 | 自选股是当前状态，不做历史回溯（那是轮动回测 PIT 的范畴） |

### 1.4 总体目标

- **目标一**：让老股民能按行业或自定义主题对自选股分组管理，支持拖拽排序、批量调整，管理几十只甚至上百只自选股不混乱。
- **目标二**：让老股民能为每只自选股设置价格提醒（目标价/涨跌幅阈值），盘中价格触发时主动 toast + 顶部通知，不用一直盯盘。
- **目标三**：让老股民在自选股列表一眼看到多维指标（行业/市值/PE/PB/换手率）+ 30 日 mini K 线趋势，不用切到其他页面即可判断个股状态。
- **目标四**：让老股民从自选股列表无缝下钻个股诊断，形成「管理自选 -> 提醒触发 -> 下钻诊断」的完整使用闭环。
- **目标五**：让自选股页面脚本独立化，为后续功能迭代（如复杂预警、组合收益）打下可维护的前端工程基础。
- **质量准绳**：`.trae/rules/系统设计理念(用法建议).md` + A 股实战派认可（不接受"纸面能跑、实战不能用"）。

## 2. 典型用户场景

### 2.1 主场景：老股民管理自选股池

**时间**：交易日盘前 9:00 / 盘中 10:30 / 收盘后 15:30 三个时点
**用户**：3 年以上经验的 A 股散户/老股民，自选股池里有 30-80 只关注的股票，习惯按行业分组管理，需要价格提醒辅助决策

**使用路径**：

1. 登录后点击侧边栏「自选股」进入 `/page/watchlist`
2. **按分组浏览（W3）**：
   - 左侧分组栏显示「全部」「银行」「新能源」「医药」「自定义1」等分组
   - 点击「银行」分组 -> 列表只显示银行股自选
   - 老股民判断「今日银行板块有异动，先看银行自选」
3. **直接添加新股票（W5）**：
   - 顶部搜索框输入「招商」-> SearchSuggest 联想出「招商银行 600036.SH」
   - 点击联想项 -> 直接加入自选股（默认归入「全部」或当前选中分组）
   - toast 提示「已添加招商银行到自选股」
4. **设置价格提醒（W2）**：
   - 列表中某只股票行点击「设提醒」-> 弹窗设置目标价上限 15 元 / 下限 12 元
   - 或选「按涨跌幅」模式：涨幅超 5% / 跌幅超 3% 提醒
   - 保存后，列表该行显示提醒图标（小铃铛）
5. **批量整理（W7）**：
   - 勾选多只新能源股票 -> 点击「批量改分组」-> 选「新能源」分组 -> 批量归类
   - 或勾选多只已不关注的股票 -> 点击「批量移除」-> 确认后批量删除
6. **按维度排序（W6）**：
   - 切换排序方式「按涨跌幅降序」-> 自选股按今日涨幅从高到低排列
   - 或切「按添加时间」-> 看最近加入的自选股
7. **看多维指标 + mini K 线（W9 + W8）**：
   - 列表每行显示 行业 / 市值 / PE / PB / 换手率 5 列
   - 右侧 mini K 线显示 30 日收盘价折线，一眼判断趋势
   - 老股民经验：PE < 10 且 PB < 1 且 mini K 线向上 = 低估值上升趋势股（可能是机会）
8. **价格触发提醒（W2）**：
   - 盘中 10:30，某只自选股涨幅达 5%，触发涨跌幅阈值
   - 页面顶部弹出通知条 + toast 提醒「招商银行 涨幅达 5%，当前价 15.20 元」
   - 老股民点击通知 -> 跳转该股票的个股诊断页
9. **下钻个股诊断（W4）**：
   - 列表中点击「招商银行 600036.SH」股票代码
   - 跳转 `/page/stock-detail/600036.SH` 进入个股诊断做深入分析
10. **收盘后复盘**：
    - 收盘后回到自选股页面，刷新列表
    - 按「换手率」排序看哪些自选股今日活跃
    - 删除今日已卖出的自选股

### 2.2 异常场景：数据未拉取或依赖模块未上线

- **`daily_basic` 表无数据**：行业/市值/PE/PB/换手率 5 列显示「-」占位，不影响基础行情列（代码/名称/价/涨跌额/涨跌幅/量）展示
- **`daily_quote` 表无数据**：mini K 线区域显示「暂无数据」占位，涨跌幅列显示「-」，价格提醒不触发（无行情可比对）
- **个股诊断模块（017）未上线**：股票代码列显示为纯文本（不可点击），不报错；017 上线后自动变为可点击链接
- **`sw_industry` / `sw_industry_member` 表无数据**：按行业自动分组功能降级--分组栏不显示「按行业」分组，只显示「全部」+ 自定义分组
- **`sys_watchlist` 表扩展未完成（W1 未上线）**：W2 价格提醒 / W3 分组管理 / W7 批量改分组 三项功能不可用，其他 7 项功能正常（W4/W5/W6/W8/W9/W10 不依赖 W1）
- **价格提醒轮询失败**：单次轮询失败静默重试，不 toast 干扰用户；连续 3 次失败才 toast 提示「行情刷新异常，请稍后重试」

## 3. 功能需求

### FR-1：扩展 sys_watchlist 表结构（对应 W1）

#### 3.1.1 功能描述

扩展项目自有表 `sys_watchlist`，在现有 4 个字段（`id` / `user_id` / `stock_code` / `created_at`）基础上**新增 5 个字段**，支撑分组管理、备注、价格提醒、排序：

| 新增字段 | 类型 | 说明 | 用途 |
|---|---|---|---|
| `group_id` | BIGINT / INTEGER | 所属分组 ID，NULL 表示「未分组」（默认归入「全部」） | W3 分组管理 |
| `note` | VARCHAR(255) | 用户备注（如「破净关注」「业绩预告看好」） | 个性化备注 |
| `target_price_high` | DECIMAL(20,4) | 目标价上限，NULL 表示未设上限 | W2 价格提醒 |
| `target_price_low` | DECIMAL(20,4) | 目标价下限，NULL 表示未设下限 | W2 价格提醒 |
| `sort_order` | INT | 分组内排序序号（从 1 递增），默认 0 | W3 拖拽排序、W6 列表排序 |

**约束**：
- 同步改 `schema-mysql.sql` + `schema-sqlite.sql` + `schema-mysql-comments.sql` 三份 schema 文件。
- `UNIQUE(user_id, stock_code)` 唯一约束保留不变。
- 新增字段均允许 NULL（向后兼容现有数据，已有自选股记录默认 `group_id=NULL` / `note=NULL` / 提醒阈值 NULL / `sort_order=0`）。
- 新增 `INDEX idx_sys_watchlist_group (user_id, group_id)` 索引，支撑按分组查询。

#### 3.1.2 用户价值

- **为分组管理奠基**：`group_id` 字段让自选股可以按分组归类，老股民管理几十只自选股不混乱。
- **为价格提醒奠基**：`target_price_high` / `target_price_low` 字段让每只自选股可设置独立的目标价阈值，不用一直盯盘。
- **为个性化备注奠基**：`note` 字段让老股民记录关注理由（如「等破净买入」「业绩预增」），避免时间久遗忘。
- **为拖拽排序奠基**：`sort_order` 字段让自选股可按用户自定义顺序排列，把重点关注股排在前面。

#### 3.1.3 数据依赖

- **项目自有表 `sys_watchlist`**：无 tushare 接口依赖（该表由项目自身的 `WatchlistController` 管理，非行情数据）。
- **MySQL 与 SQLite 双库同步**：必须同步改 `schema-mysql.sql` + `schema-sqlite.sql` 两份 schema 文件（项目支持 MySQL 生产 + SQLite 测试双部署模式）。
- 同步改 `schema-mysql-comments.sql` 加 ALTER TABLE 注释（项目规范要求）。

#### 3.1.4 前端改造点

- 无直接前端改造（表结构扩展是后端基础）。
- 前端在 W2/W3/W6/W7 的接口调用中消费新字段。

#### 3.1.5 后端改造点

- `schema-mysql.sql` + `schema-sqlite.sql` + `schema-mysql-comments.sql` 三份 schema 同步加 5 个字段。
- `WatchlistDO`（DO 对象）加 5 个字段对应 Java 属性。
- `WatchlistMapper` 无需改（继承 BaseMapper，自动支持新字段）。
- `WatchlistService` 的 `addToWatchlist` 方法扩展接受可选参数 `groupId` / `note` / `targetPriceHigh` / `targetPriceLow`。
- 现有 `GET /watchlist` 接口返回的 VO 扩展含 5 个新字段。

### FR-2：价格提醒（对应 W2）

#### 3.2.1 功能描述

为自选股列表中的每只股票提供**价格提醒**能力，支持两种设置模式：

| 模式 | 设置方式 | 触发条件 |
|---|---|---|
| 目标价模式 | 设置目标价上限 `target_price_high` / 下限 `target_price_low`（如上限 15 元 / 下限 12 元） | 最新价 ≥ 上限 或 ≤ 下限 时触发 |
| 涨跌幅模式 | 设置涨幅阈值 / 跌幅阈值（如涨幅超 5% / 跌幅超 3%） | 当日 `pct_chg` ≥ 涨幅阈值 或 ≤ -跌幅阈值 时触发 |

**设置入口**：
- 列表中每行右侧「设提醒」按钮（小铃铛图标）-> 弹窗设置提醒参数。
- 弹窗内可选「按目标价」或「按涨跌幅」模式，二选一。
- 「按涨跌幅」模式保存时，前端根据当前最新价换算为等价的目标价上下限存入 `target_price_high` / `target_price_low`（如当前价 10 元 + 涨幅 5% 阈值 -> 上限 10.5 元），复用 W1 扩展字段。

**提醒触发方式**：
- 前端轮询：页面打开时，每 60 秒轮询一次自选股最新行情（复用 `/watchlist` 接口返回的最新价 + `pct_chg`），比对阈值。
- 触发时：
  1. 页面顶部弹出通知条（固定在页面顶部，红色/绿色背景区分涨/跌方向，显示「股票名 + 代码 + 当前价 + 触发条件」），5 秒后自动消失或用户手动关闭。
  2. 同步 toast 提醒（右下角，`StockApp.toast`）。
  3. 列表对应行的高亮闪烁 2 秒（背景色脉冲动画）。
  4. 通知条点击 -> 跳转该股票的个股诊断页（`/page/stock-detail/{code}`，依赖 W4）。
- 同一股票同一阈值在 5 分钟内只提醒一次（避免反复触发）。

**提醒状态展示**：
- 列表中已设提醒的股票行，右侧显示「小铃铛」图标，鼠标悬停 tooltip 显示「上限 15 元 / 下限 12 元」。
- 点击小铃铛图标 -> 弹窗编辑或清除提醒。

#### 3.2.2 用户价值

- **不用一直盯盘**：老股民关注几十只自选股，不可能全程盯盘，价格提醒让系统主动通知「哪只股票到价了」，解放盯盘时间。
- **目标价 + 涨跌幅双模式**：老股民有不同习惯--价值派习惯设目标价（如「破净 0.9 倍 PB 时买入，对应 12 元」），趋势派习惯设涨跌幅（如「涨幅超 5% 就止盈」）；双模式覆盖两类习惯。
- **顶部通知 + toast + 行高亮三重提醒**：避免漏看，顶部通知最显眼，行高亮定位具体股票。

#### 3.2.3 数据依赖

- **依赖 `sys_watchlist` 表的 `target_price_high` / `target_price_low` 字段**（W1 扩展）：存储每只自选股的提醒阈值。
- **依赖 `daily_quote` 表**（已有数据，由 tushare `daily` 接口采集，120 起积分）：提供最新价 `close` + 当日涨跌幅 `pct_chg`，用于触发判断。
- **依赖现有 `/watchlist` 接口**：返回的自选股 VO 扩展含 `targetPriceHigh` / `targetPriceLow` 字段 + 最新价 + 涨跌幅（已有 VO 已含行情字段）。
- 无需新建表。

#### 3.2.4 前端改造点

- `watchlist.html`：列表每行右侧加「设提醒」按钮 + 提醒状态小铃铛图标；页面顶部预留通知条容器。
- `watchlist.js`（W10 抽出后）：

### FR-3：分组管理（对应 W3）

#### 3.3.1 功能描述

为自选股提供**分组管理**能力，支持两种分组来源：

| 分组类型 | 来源 | 说明 |
|---|---|---|
| 行业分组（自动） | `sw_industry_member` + `sw_industry` 表 | 按申万一级行业自动归类（如「银行」「新能源」「医药」），用户无需手动分组 |

- 左侧加分组栏（侧边导航样式），显示「全部」+ 行业分组列表 + 自定义分组列表 + 「未分组」。
- 中间是自选股表格，显示当前选中分组下的股票。

- 表格行支持拖拽（HTML5 drag-and-drop），拖动某行到新位置 -> 更新 `sort_order` 字段。

#### 3.3.4 前端改造点

- `watchlist.js`：

#### 3.3.5 后端改造点

## 4. 非功能需求

### 4.1 价格提醒实时性

- **轮询频率**：页面打开时每 60 秒轮询一次自选股最新行情，比对价格提醒阈值。
- **触发时效**：从价格触达阈值到 toast 提醒弹出，延迟 ≤ 60 秒（受轮询频率限制）。
- **去重**：同一股票同一阈值在 5 分钟内只提醒一次，避免阈值附近价格抖动反复触发。
- **轮询失败处理**：单次轮询失败静默重试（下次轮询），不 toast 干扰用户；连续 3 次失败才 toast 提示「行情刷新异常，请稍后重试」。
- **页面不可见时**：浏览器 tab 切到后台时，轮询降频为每 5 分钟一次（用 `document.visibilitychange` 事件监听），节省资源。
- **提醒记录**：提醒触发后，可选在 localStorage 记录最近提醒历史（时间/股票/触发条件），用户可查看「今日提醒历史」（一期可选，非必须）。

### 4.2 mini K 线性能

- **单页 mini 图数量**：自选股列表默认每页 50 条，即每页最多 50 个 ECharts 实例。
- **渲染性能**：50 个 mini K 线图首屏渲染时间 ≤ 1 秒（含数据加载 + ECharts 初始化 + 渲染）。
- **内存占用**：每个 ECharts mini 实例内存 ≤ 100KB，50 个实例总内存 ≤ 5MB。
- **释放**：翻页或刷新列表时，旧实例必须 `echarts.dispose()` 释放，避免内存泄漏。
- **懒加载**：mini K 线图在表格行滚动到可视区时才渲染（IntersectionObserver），不在可视区的行延迟渲染（可选优化，一期若性能达标可不实现）。
- **数据量**：每只股票 30 个收盘价数据点，50 只共 1500 个数据点，单次接口返回数据量 ≤ 30KB。

### 4.3 schema 双库同步

- **MySQL 与 SQLite 同步**：`sys_watchlist` 表扩展的 5 个字段必须同步改 `schema-mysql.sql` + `schema-sqlite.sql` 两份 schema 文件（项目支持 MySQL 生产 + SQLite 测试双部署模式）。
- **新建 `sys_watchlist_group` 表**：同步建两份 schema + `schema-mysql-comments.sql` 注释。
- **字段类型对齐**：MySQL 用 `BIGINT` / `DECIMAL(20,4)` / `INT` / `VARCHAR(255)`；SQLite 用 `INTEGER` / `REAL` / `INTEGER` / `TEXT`（SQLite 动态类型，但建议对齐 MySQL 语义）。
- **索引同步**：新增 `INDEX idx_sys_watchlist_group (user_id, group_id)` 索引，两份 schema 都加。
- **向后兼容**：扩展字段均允许 NULL，已有自选股记录默认 NULL / 0，不破坏现有数据。

### 4.4 主题统一

- 列表所有数字、涨跌色、链接色必须复用 `theme.css` 与 `ChartsTheme` 主题变量。
- 三主题（azure / mist / cyber）切换后，列表颜色 + mini K 线颜色 + 分组栏样式同步变化，不允许出现「切换主题后表格颜色不更新」的 bug。
- 涨跌色统一用 `theme.css` 的 `.stock-up`（红）/ `.stock-down`（绿）类名，不硬编码颜色值。
- mini K 线的折线 + 填充色用 `ChartsTheme.upColor` / `ChartsTheme.downColor` 主题变量。
- 分组栏样式、复选框样式、拖拽样式与全局组件样式一致（复用 Bootstrap 主题变量）。

### 4.5 数据降级

- **`daily_basic` 表无数据**：行业/市值/PE/PB/换手率 5 列显示「-」占位，基础行情列正常展示。
- **`daily_quote` 表无数据**：mini K 线区域显示「暂无数据」占位，涨跌幅列显示「-」，价格提醒不触发（无行情可比对）。
- **`sw_industry_member` 表无数据**：行业列显示「-」，按行业自动分组功能降级（分组栏不显示「按行业」分组，只显示「全部」+ 自定义分组）。
- **个股诊断模块（017）未上线**：股票代码列降级为纯文本（不可点击），其他功能不受影响。
- **`sys_watchlist` 表扩展未完成（W1 未上线）**：W2 价格提醒 / W3 分组管理 / W7 批量改分组 三项功能不可用，其他 7 项功能正常。
- **数据时效提示**：列表顶部显示「数据日期 YYYY-MM-DD」，让用户清楚看到数据是何日的（避免周末看到周五数据以为今日数据）。

### 4.6 兼容性

- 浏览器：Chrome / Edge / Firefox 最新版（不兼容 IE）。
- 分辨率：最低 1366x768，推荐 1920x1080。
- 移动端：不做响应式适配（自选股页面是 PC 端专业工具，不在手机使用）。

## 7. 风险与注意事项

### 7.1 价格提醒轮询的浏览器限制

- **后台 tab 轮询降频**：浏览器对后台 tab 的 `setInterval` 会自动降频（通常降至 1 次/秒），但 60 秒轮询不受影响；切到后台时主动降为 5 分钟一次更稳妥。
- **多 tab 一致性**：用户若开多个自选股 tab，多个 tab 都会轮询 + 触发提醒，可能重复 toast。一期不处理多 tab 协调（复杂度高），提示用户「不要开多个自选股 tab」即可。
- **轮询接口压力**：50 只自选股每 60 秒轮询一次，单次接口 JOIN `daily_quote` + `daily_basic` 查询，需确保 `(ts_code, trade_date)` 索引存在，避免慢查询。

### 7.2 mini K 线的 ECharts 实例管理

- **实例泄漏风险**：翻页 / 刷新列表 / 切换分组时，旧 ECharts 实例必须 `echarts.dispose()` 释放，否则内存持续增长。
- **实例数量上限**：若自选股超过 100 只，单页 50 个实例已是上限，建议分页每页 ≤ 50 条，不做虚拟滚动（复杂度高）。
- **渲染时机**：ECharts 实例化需在 DOM 元素插入后再 `echarts.init(dom)`，注意异步数据加载后 DOM 已存在再初始化。

### 7.3 sys_watchlist_group 表的设计取舍

- **为何新建表而非复用 group_id 字段**：分组名也需要存储 + 用户私有，若只靠 `sys_watchlist.group_id` 无法表达分组元信息（名称/创建时间/排序），必须新建 `sys_watchlist_group` 表。
- **行业分组 vs 自定义分组的区分**：行业分组是虚拟分组（不写 `group_id`，展示时动态 JOIN `sw_industry_member` 取），自定义分组是实际分组（写 `group_id`）。这种混合设计需前端清楚区分两类分组的查询逻辑。
- **分组排序**：`sys_watchlist_group.sort_order` 字段控制分组栏中分组的显示顺序，用户可拖拽分组调整顺序。

### 7.4 PE/PB 为 NULL 或负值的处理

- **亏损股 PE 为负或 NULL**：`daily_basic.pe_ttm` 对亏损股为负值或 NULL，展示时 NULL 显示「-」，负值原值显示（老股民习惯看亏损股 PE 为负，是风险信号）。
- **破净股 PB < 1**：PB 可能为负（资不抵债）或 NULL，展示时 NULL 显示「-」，负值原值显示（老股民习惯看破净股，PB < 1 是机会信号，不能简单隐藏）。

### 7.5 股票代码跳转的降级方案

- **个股诊断模块（017）未上线时**：FR-4 股票代码跳转需做降级处理，避免点击后 404 影响体验。
- **建议方案**：前端读取全局配置 `window.STOCK_DETAIL_ENABLED`（或调用 `/api/stock-detail/health` 健康检查），017 未上线时渲染为纯文本 `<span>`，017 上线后渲染为 `<a>` 链接。具体降级方案落地 spec 定。

### 7.6 数据时效提示

- **行情数据延迟**：`daily` / `daily_basic` 接口每日收盘后采集，盘中自选股页面显示的是前一日收盘数据，需在列表顶部明确标注「数据日期」避免误导。
- **价格提醒盘中触发**：因数据是前一日收盘数据，盘中价格提醒实际比对的是「昨日收盘价 + 昨日涨跌幅」，而非实时价。若需盘中实时提醒，需引入实时行情接口（一期不做）。
