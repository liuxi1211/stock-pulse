## §1 P1-6：FactorNode transform 滚动窗口聚合（二期）

### 1.1 现象

`filter.conditions` 与 `factor` 的 `FactorNode` 只取**当日因子快照值**。`FactorNode.params: {"timeperiod": 20}` 是指标计算周期（MA20 = 20 日均线当天的点），**不是「对因子值再做 N 日聚合」**。

老股民常用条件当前**无法表达**：

| 用例 | 需要的聚合 | 当前能否表达 |
|---|---|---|
| 「过去 20 日均价 > MA60」（均线多头排列） | CLOSE 的 20 日均值 | ❌（只能用当天 MA20 > MA60 等价绕开） |
| 「过去 5 日涨幅 > 10%」 | 5 日 pct_change | ❌ |
| 「过去 20 日波动率 < 0.3」 | 20 日 std | ❌ |
| 「过去 20 日 PE 均值 < 30」 | PE_TTM 的 20 日均值 | ❌（基本面因子完全无滚动窗口） |

### 1.2 改造方案

#### 1.2.0 方案选型：即时聚合（采用）而非预计算快照

曾考虑「注册派生因子（如 `PE_TTM_MA20`）并预计算到 `factor_snapshot` 表」的方案，**否决，采用 engine 即时聚合**。核校代码后理由：

| 维度 | 即时聚合（采用） | 预计算到 factor_snapshot（否决） |
|---|---|---|
| 历史覆盖 | ✅ 候选 60 bar 窗口已在内存，任意历史调仓日可算 | ❌ `factor_snapshot` 现为「最新日 + 末值 `[-1]` + 仅技术面」的预留缓存（`FactorSnapshotService` 注释明示 Phase 2 性能优化、选股仍走 engine 实时算），回测需每个历史日的值 |
| point-in-time | ✅ 窗口截至调仓日 T，天然无未来数据 | ❌ 需每个 (T, 股票) 都有快照行 |
| engine 不触库 | ✅ 纯内存 | ❌ engine 不能直读 DB，回测逐调仓日 HTTP 查不可接受 |
| 基本面因子（PE_TTM 均值是头号用例） | ✅ `extra` 逐 bar 已有 | ❌ 预计算管线只喂 OHLCV，MA(PE_TTM) 算不出 |
| 用户可配（任意 factor×type×window） | ✅ transform 字段随配随算 | ❌ 白名单驱动，无法预判组合 |

> **结论**：因子值的「计算能力」本就在 engine（`batchComputeFactor` / `compute_single` 返回完整序列），即时聚合只是**保留序列**而非取 `[-1]` 再做 ma/std 聚合。`factor_snapshot` 是选股中心实时截面的未来性能优化，与回测 transform 正交，本需求不改动它。
>
> **范围限定**：transform **仅轮动范式**（`rebalance_engine` 选股条件）。择时范式（`on_bar`）虽可经 `get_history_df` 取历史 bar 实现同样聚合，但本期不做——聚合内核抽成共享函数（见 §1.2.2），留择时扩展点。

#### 1.2.1 schema 扩展（engine `services/strategy/models.py`）

`FactorNode` 增加 `transform` 字段：

```python
class TransformConfig(BaseModel):
    """对因子值再做滚动窗口聚合（选股条件用）。"""
    model_config = ConfigDict(extra="forbid")

    type: Literal["ma", "std", "pct_change", "max", "min"] = Field(
        ..., description="聚合类型：ma=均值 std=标准差 pct_change=(末-首)/首 max=最大 min=最小"
    )
    window: int = Field(..., ge=1, description="窗口天数（交易日）")


class FactorNode(BaseModel):
    model_config = ConfigDict(extra="forbid")

    factor: str = Field(...)
    params: Optional[Dict[str, Any]] = Field(None, ...)
    inputs: Optional[Dict[str, List[str]]] = Field(None, ...)
    output_index: Optional[int] = Field(None, ...)
    transform: Optional[TransformConfig] = Field(None, description="对因子值再做聚合；仅选股条件生效")
```

示例（条件树叶子）——「过去 20 日 PE 均值 < 30」：

```json
{"factor": "PE_TTM", "transform": {"type": "ma", "window": 20}, "comparator": "<", "right": {"value": 30}}
```

#### 1.2.2 engine 因子计算改造（`rebalance_engine._compute_one_factor`）

当前对技术因子 `kline_to_arrays → compute_single → [-1]`（末值）、基本面取 `candidate["extra"][trading_date]`（单值）。改造为**当 transform 存在时返回窗口末 N 日序列聚合后的标量**：

```python
def _compute_one_factor(factor_key, ref, candidate, trading_date=None) -> float:
    transform = ref.get("transform")  # {"type","window"} 或 None
    series = _factor_series(factor_key, ref, candidate, trading_date)  # 窗口内每日因子序列
    if series is None or len(series) == 0:
        return float("nan")
    if transform is None:
        return float(series[-1])  # 现状：末值
    window = transform["window"]
    if len(series) < window:
        return float("nan")  # 窗口不足，该 symbol 此因子无效
    seg = series[-window:]
    t = transform["type"]
    if t == "ma":         return float(np.mean(seg))
    if t == "std":        return float(np.std(seg, ddof=1)) if len(seg) > 1 else float("nan")
    if t == "pct_change": return float((seg[-1] - seg[0]) / seg[0]) if seg[0] != 0 else float("nan")
    if t == "max":        return float(np.max(seg))
    if t == "min":        return float(np.min(seg))
```

`_factor_series` 实现：
- **技术/价格因子**：`kline_to_arrays(candidate kline) → compute_single → 整条序列`（不再只取 `[-1]`）。
- **基本面因子**：从 `candidate["extra"]` 按窗口内每个 `trade_date_str` 取 `pe_ttm / pb / ...` 组成序列（基本面已逐 bar 下发，见 §0.3-①）。
- **多输出因子**（MACD/BOLL）：先按 `output_index` 降维成单序列，再 transform。

> **NaN 安全**：窗口内任何一日因子值为 NaN，按「跳过该日」处理（`seg = [v for v in series[-window:] if not nan]`），跳过后不足 window 仍返回 NaN。
>
> **共享聚合内核**：ma/std/pct_change/max/min 聚合逻辑抽成独立函数（如 `services/shared/factor_pipeline.py` 的 `aggregate_series(series, transform)`），`_compute_one_factor` 调用它。择时范式未来接入时复用同一函数（取 `get_history_df(window)` 序列 → 同一聚合），避免逻辑分叉。

#### 1.2.3 因子预计算去重（`_collect_all_factor_refs` / `_compute_factors`）

含 transform 的 FactorNode 与同名的无 transform 节点**语义不同**（一个是序列聚合值，一个是末值），预计算 key 需区分，避免互相覆盖。约定 key 规则：

```
key = factorKey                              # 无 transform
key = f"{factorKey}__{type}{window}"         # 有 transform，如 PE_TTM__ma20
```

`_collect_all_factor_refs` 把 transform 透传进 ref dict；`_compute_factors` 用上述 key 落入 `factor_values_by_symbol[sym]`；条件树求值（`_eval_node`）按相同规则查 key。

#### 1.2.4 validator（`services/strategy/validator.py`）

- `transform.window` 范围：`1 ≤ window ≤ 60`（history_window 上限），超出报 `INVALID_TRANSFORM_WINDOW`；
- `transform.type` 必须在枚举内，否则报 `INVALID_TRANSFORM_TYPE`；
- **作用域约束**：transform 仅在 `filter.conditions` 内合法；出现在 `factor.weights`（ranking）时报 `TRANSFORM_NOT_ALLOWED_IN_RANKING`（本期不支持 ranking transform，见 §1.4）。

新增错误码到 `errors.py`：`INVALID_TRANSFORM_WINDOW` / `INVALID_TRANSFORM_TYPE` / `TRANSFORM_NOT_ALLOWED_IN_RANKING`。

#### 1.2.5 前端（`strategy-editor.js` 条件树叶子）

条件树叶子增加可选「滚动窗口」配置：聚合类型下拉（均值/标准差/涨跌幅/最大/最小）+ 窗口天数。未配则等价现状（当日值）。collect/refill 透传 `transform`。

### 1.3 关键决策：仅选股条件

transform **仅作用于 `filter.conditions`**，`factor.weights`（ranking 打分）不支持。理由：
- 008 列出的全部用例都是选股条件；
- ranking 用滚动聚合语义复杂（综合分用窗口均值还是末值？截面排序基准如何定义？），无明确需求，留扩展点。

### 1.4 边界

- **窗口数据已具备**：技术因子有 60 bar 历史，基本面经 `appendBasicFields` 逐 bar 下发（§0.3-①）。**无新 watcher 数据依赖**。
- **窗口不足**：候选历史 < window（如回测初期或次新股）→ 该因子返回 NaN → 后续走 `_filter_valid_symbols` 剔除或条件不命中（与现有 NaN 处理一致）。
- **warmup 联动**：含大 window transform 的策略，compiler 的 warmup 自动推断应把 max(window) 纳入考量（避免回测初期大面积 NaN）；当前 warmup 推断基于因子 lookbackHint，需确认 transform.window 是否被纳入——若否，补一条「transform.window 计入 warmup」。

### 1.5 验收

- [ ] `FactorNode` 支持 `transform`，`PE_TTM + transform(ma,20)` 能算出 20 日 PE 均值；
- [ ] 条件树可用 `transform(ma/std/pct_change/max/min)` 表达「过去 N 日 X 聚合 < 阈值」；
- [ ] 同一因子「当日值」与「20 日均值」可在同一条件树共存（key 不冲突）；
- [ ] `window > 60` 报 `INVALID_TRANSFORM_WINDOW`；
- [ ] transform 出现在 ranking 报 `TRANSFORM_NOT_ALLOWED_IN_RANKING`；
- [ ] 前端条件树可配滚动窗口，collect/refill 正确。

## §2 P2-9：分批调仓 + 冲击成本建模（三期）

### 2.1 现象

轮动调仓一次性全量切换（一期 `order_target_weights` 一次性下单）。大资金（百万级以上）一次性吃单的**冲击成本未建模**，回测高估收益；且一次性大单对流动性差的标的成交不真实。

### 2.2 改造方案

#### 2.2.1 schema 扩展（engine `services/strategy/models.py`）

`RebalanceModel` 增加 `execution`：

```python
class ExecutionConfig(BaseModel):
    """分批调仓 + 冲击成本配置。"""
    model_config = ConfigDict(extra="forbid")

    split_days: int = Field(1, ge=1, le=5, description="分批天数（1=一次性，3=分 3 天）")
    impact_cost_bps: Optional[float] = Field(
        None, ge=0.0, description="冲击成本(bps)，按成交量线性建模；空=不建模"
    )


class RebalanceModel(BaseModel):
    ...
    execution: Optional[ExecutionConfig] = Field(None, description="分批调仓 + 冲击成本")
```

#### 2.2.2 分批执行：冻结法状态机（engine `compiler.on_daily_rebalance`）

一期 `on_daily_rebalance`：触发日一次性算 target_weights → `order_target_weights`。改造为**冻结法分批**：

```
触发日（_is_rebalance_day=True）：
  1. select_at_rebalance_date 算出完整 plan（ranking + score 排序后的 target_weights）
  2. 经一期后处理（cash_reserve / 单标的上限 / 行业暴露 / buffer / 涨跌停拒单）
  3. 若 split_days==1：直接 order_target_weights(plan)（现状）
  4. 若 split_days>1：把 plan 切成 N 份（per_day = plan / N），存入状态机
     self._pending_split = {plan, total_days=N, current_day=1, per_day_target}
     当天执行第 1 份（order_target_weights(per_day_1)）

非触发日（且 _pending_split 未耗尽）：
  - current_day += 1；执行第 current_day 份增量
  - current_day == total_days 时清空 _pending_split
```

**边界——新触发日撞上未完成分批**：若新调仓日到来时 `_pending_split` 仍在，**作废剩余分批，按新 plan 重新开始**（不合并，避免状态纠缠）。日志记录「分批被打断：原 plan 剩 X 天未执行」。

**与一期机制协同**：
- score 排序、cash_reserve、buffer、涨跌停拒单都在**触发日一次性算定**（冻结 plan），分批日只执行固定增量，不重算；
- `reject_limit_up` 在每个分批日**当日**判定（涨跌停是逐日事实，不能冻结）：分批日若某买入标的当日涨停，该份增量跳过该标的。

#### 2.2.3 冲击成本：volume-linear 建模

每笔成交在 fill price 上叠加冲击成本：

```python
# 本笔成交额 / 当日成交额（participation rate），封顶防除零/异常
participation = min(order_value / bar_volume, PARTICIPATION_CAP)  # CAP 默认 1.0
impact_bps = impact_cost_bps * participation
fill_price_adj = fill_price * (1 + sign * impact_bps / 10000)  # 买入加价/卖出折价
```

注入路径（akquant 硬约束：`slippage` 必须用 dict，见 CLAUDE.md 防坑表）：
- 优先经 `order_target_weights` 的 `price_map`：engine 为每个标的算 `fill = close × (1 ± impact)` 传入 `price_map`，让 akquant 按该价成交；
- 或经 `run_backtest` 的 `slippage={"type":"percent","value":...}` ——但 slippage 是固定比例，无法表达「按成交量线性」，故**优先用 price_map**。

> `bar_volume` 来自 kline_data 的 `volume`（已下发）；`order_value` = 目标股数 × 价格。`PARTICIPATION_CAP` 抽常量（默认 1.0，防小成交量标的算出 absurd impact）。

#### 2.2.4 结果回传（`result_serializer`）

新增 `execution_diagnosis`：

```python
"execution_diagnosis": {
    "split_days": 3,
    "splits_completed": 3,        # 实际完成分批数
    "splits_interrupted": 0,      # 被新触发日打断的次数
    "total_impact_cost": 1234.56, # 累计冲击成本（元）
    "avg_participation": 0.012,   # 平均参与率
}
```

#### 2.2.5 validator

- `split_days ∈ [1,5]`，`impact_cost_bps ≥ 0`，否则报 `INVALID_EXECUTION_CONFIG`；
- `execution` 仅轮动范式（has_rebalance）合法，择时范式报 `EXECUTION_REQUIRES_REBALANCE`。

### 2.3 验收

- [ ] `split_days=1` 行为与现状一致（一次性调仓）；
- [ ] `split_days=3`：触发日 + 后 2 个交易日各执行约 1/3 增量，trades 分布在 3 天；
- [ ] 新触发日撞未完成分批 → 作废重来 + 日志；
- [ ] 分批日某标的当日涨停 → 该份增量跳过该标的；
- [ ] `impact_cost_bps=10` 对低成交量标的的成交价叠加更高冲击；
- [ ] 结果含 `execution_diagnosis`；`split_days>5` 报 `INVALID_EXECUTION_CONFIG`。

## §3 遗留#1：watcher 元数据下发（Tier 2 完整 Tushare，收口）

### 3.1 现状

`BacktestServiceImpl.buildKlineData`（L623-625 TODO）未下发 `is_st / is_suspended / is_limit_up / is_limit_down / list_date`。engine 侧 `_filter_by_static_rules`（P0-1）与涨跌停拒单（P0-4）**逻辑已就绪**，缺数据即静默跳过 + warning。采用 **Tier 2 完整 Tushare**：端到端对接 3 个接口，逐日准确。

### 3.2 改造方案

按 [`.trae/rules/stock-watcher/business/02-tushare-integration-guide.md`](../../../.trae/rules/stock-watcher/business/02-tushare-integration-guide.md) 端到端对接（参照 P0-0 申万行业 10 处改动的范例）：

#### 3.2.1 对接的 Tushare 接口（3 个）

| 接口 | doc_id | 用途 | 权限 | 限量 |
|---|---|---|---|---|
| `namechange` | [160](https://tushare.pro/document/2?doc_id=160) | 股票更名历史（含 ST/*ST 戴帽摘帽） | 2000 积分 | 单次 5000 行 |

## §5 遗留#3：行业归属 point-in-time（收口）

### 5.1 现状

`buildKlineData` 用 `getLatestL1Industries`（最新归属，全 bar 相同，L605-607 注释明示）。跨年度回测中，标的若期间换行业（申万每年 6/12 月调整），用最新归属会产生轻微 lookahead bias。

### 5.2 改造方案

#### 5.2.1 批量 point-in-time 查询

`SwIndustryService` 新增批量接口（避免逐 bar 逐股查 N×M 次）：

```java
/**
 * 批量 point-in-time：取多只股票在指定区间内每日生效的一级行业。
 * @return key=tsCode, value={trade_date(yyyyMMdd) -> index_code}
 */
Map<String, Map<String, String>> getL1IndustriesPit(List<String> tsCodes, String startDate, String endDate);
```

实现：一次性查 `sw_industry_member` 在区间内的全部记录（按 ts_code + update_date），内存按 ts_code 排序、对每个 trade_date 用「≤ 该日最新 update_date」的前向填充（forward-fill）建索引。

#### 5.2.2 buildKlineData 改用 PIT

```java
Map<String, Map<String, String>> swL1PitMap = swIndustryService.getL1IndustriesPit(symbols, startDate, endDate);
// ...
for (DailyQuoteDO q : quotes) {
    // ...
    Map<String, String> perDay = swL1PitMap.get(code);
    if (perDay != null) {
        String swL1 = perDay.get(q.getTradeDate());  // 当日归属
        if (swL1 != null) bar.put("sw_industry_l1", swL1);
    }
}
```

#### 5.2.3 engine 侧（无改动）

`data_adapter` 仍按 bar 取 `sw_industry_l1`，`_get_industry` 已就绪。PIT 由 watcher 侧保证（per-bar 注入当时归属）。

### 5.3 性能与边界

- 申万行业半年调一次，区间内多数标的归属不变，forward-fill 命中率高；
- 早期回测（早于最早 `update_date`）的标的：`getL1IndustriesPit` 返回 null，bar 不下发 `sw_industry_l1`，engine 行业暴露项静默跳过 + warning（与现状一致）。

### 5.4 验收

- [ ] `getL1IndustriesPit` 返回各标的区间内每日归属；
- [ ] 跨年度回测中，期间换行业的标的按当时归属（非最新）下发；
- [ ] engine 行业暴露/静态过滤用 PIT 归属，消除 lookahead bias。

## §8 风险与回滚

### 8.1 P1-6 warmup 联动风险
含大 window transform 的策略若 warmup 未覆盖 window，回测初期大面积 NaN。**缓解**：compiler warmup 推断把 `transform.window` 纳入（§1.4）；前端配置大 window 时提示所需历史长度。

### 8.2 P2-9 分批状态机风险
跨天状态机（`_pending_split`）在 akquant warm-start / 快照恢复时可能丢失。**缓解**：确认 akquant 策略对象状态在回测内持久（非跨进程）；单测覆盖「分批中」「被打断」「分批日涨停跳过」三态。**回滚**：`execution=None`（默认）即退回一次性调仓。

### 8.3 遗留#1 三接口限流与分页
namechange/suspend_d 单次有行数上限，全量回补需分页循环。**缓解**：参照 sw_industry `fetchAndSaveAllMembers` 分页范式；限流配置 200/min。`stk_limit` 精确涨跌停判定依赖该表完整，缺数据日 bar 不下发 is_limit_*（engine 静默跳过）。

### 8.4 遗留#3 forward-fill 一致性
forward-fill 基准是 `sw_industry_member.update_date`（数据快照日），若快照日与申万实际调整日（6/12 月生效）有偏差，PIT 可能有数日误差。**缓解**：可接受（行业调整低频，影响有限）；若需精确，未来用 `in_date/out_date` 区间替代 update_date。
