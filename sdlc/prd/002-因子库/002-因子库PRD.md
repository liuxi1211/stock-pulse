## 1. 产品定位

把量化投资拆成**三个可独立操作的闭环**，每个闭环都是一条"想法 → 验证 → 复用"的链路：

| 闭环 | 入口 | 产出 |
|---|---|---|
| 因子库 | 指标/因子定义 | 可复用的计算单元，前端可配置参数 |
| 策略管理 | 组合因子构建策略 | 规则树 JSON，版本化管理 |
| 回测中心 | 策略 + 历史数据 | 绩效报告 / 参数优化 / Walk-forward 分析 |
| 选股模块 | 策略规则 + 预计算指标 | 符合条件的股票列表 |

核心价值用一句话总结：**让技术指标可配置，让策略规则可回测，让选股条件可复用。**

## 3. 基础设施约定（跨模块共享）

### 3.1 Java ↔ Python 通信协议

#### 3.1.1 服务地址

| 环境 | 地址 |
|---|---|
| 开发 / 默认 | http://127.0.0.1:8000 |
| 生产可配置 | 通过 application.properties 配置 `python.compute.url` |

#### 3.1.2 统一请求格式

所有 POST 请求遵循以下外层结构：

```json
{
  "task_id": "uuid-generated-by-java",
  "timestamp": 1780000000,
  "payload": {
    "_comment": "具体参数由各模块定义"
  }
}
```

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `task_id` | 是 | Java 生成的任务唯一标识，用于幂等/追踪 |
| `timestamp` | 否 | 请求时间戳（秒），用于日志追溯 |
| `payload` | 是 | 具体业务参数，由各模块定义 |

#### 3.1.3 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "task_id": "uuid-same-as-request",
    "compute_ms": 125,
    "_comment": "具体业务数据由各模块定义"
  }
}
```

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `code` | 是 | 0 = 成功，非 0 = 错误 |
| `message` | 是 | 成功 = "success"，失败 = 错误描述 |
| `data.task_id` | 是 | 回传请求中的 task_id，便于追踪 |
| `data.compute_ms` | 否 | 本次计算耗时（毫秒），用于性能监控 |
| `data.*` | 子模块定义 | 具体业务结果 |

> **非 0 code 的统一约定**：code = 1xx 为参数错误，2xx 为计算引擎错误，3xx 为数据错误。具体子码在各模块中定义。

#### 3.1.4 本模块涉及的接口清单

| 接口 | 说明 |
|---|---|
| `GET /api/compute/factors/registry` | 获取因子元数据（供前端渲染配置器） |
| `POST /api/compute/factors` | 批量计算因子值（单股多因子） |
| `POST /api/compute/indicators` | 计算日线技术指标（单股全指标，选股模块使用） |
| `GET /api/compute/health` | 健康检查 |

#### 3.1.5 数据内嵌模式 vs 文件传递模式

由于去掉了信号中心（全市场扫描场景移除），**本阶段默认全部使用「数据内嵌」模式**：Java 调用 Python 时，OHLCV 数据直接放在请求 Body 的 JSON 中。

文件传递模式（大体积数据）保留为**后续扩展**，当前阶段不实现。

## 4. 跨模块关键决策（开发前必须拍板）

以下决策点影响多个模块的表结构和代码逻辑，在 Phase 1 开始前必须确认：

| 决策项 | 结论 | 影响模块 |
|--------|------|----------|
| 止损止盈的判断方式 | **止损用 `low`（日内最低），止盈用 `high`（日内最高），条件卖出用 `close`（收盘），`max_holding_days` 不判断价格直接平仓** | 策略管理、回测中心 |
| `rebalance_frequency_days` 的归属 | **从策略 JSON 的 `position_sizing` 中移除，仅保留在回测配置的 `rebalance` 中**（这是回测参数，不是策略的固定属性） | 策略管理、回测中心 |
| `quant_strategy_saved` 表是否保留 | **移除**。用 `quant_strategy` 表的 DRAFT 状态策略统一承载选股草稿场景 | 策略管理 |
| 因子结果 key 的命名规范 | **统一规则**：单输出因子 = `factorKey_paramValue`（如 `MA_5`、`RSI_14`）；多输出因子 = `factorKey_outputLabel`（如 `MACD_DIF`、`MACD_HIST`）。详见下文 §7 | 因子库、策略管理、选股 |
| INDEX_300 / INDEX_500 回测是否放行 | **放行**。300 只 × 750 交易日 ≈ 1.35 MB，HTTP 可承载 | 回测中心 |

## 7. 需求概述

因子是策略的积木。系统需要一个标准化的因子定义体系：

- 每类因子（如 MA、MACD、RSI）有**完整的元数据描述**（名称、参数、返回值数量、输入数据源）
- 因子元数据通过 HTTP API 暴露给 Java 端，用于**前端动态渲染参数配置表单**
- 新增因子只需在 Python 端注册，无需修改 Java 代码，前端自动适配
- 因子计算结果统一格式，上层（策略/回测/选股）消费时无需区分因子类型

## 9. 因子元数据规范

### 9.1 核心字段定义

每个因子在注册表中必须返回以下字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `factorKey` | string | 是 | 唯一标识，全大写，如 "MA"、"MACD" |
| `displayName` | string | 是 | 前端显示名，如 "简单移动均线" |
| `category` | string | 是 | 分类：`TREND` / `MOMENTUM` / `VOLATILITY` / `VOLUME` / `PRICE` |
| `description` | string | 是 | 因子含义描述，用于前端 tooltip |
| `params` | array | 是 | 参数定义列表（可为空数组） |
| `inputs` | array[string] | 是 | 输入数据源：从 `["close","high","low","volume"]` 中选择 |
| `multiOutput` | boolean | 是 | 是否多输出（如 MACD 返回 DIF/DEA/HIST 三条线） |
| `outputLabels` | array[string] | 条件 | `multiOutput=true` 时必填，各输出的中文标签 |
| `defaultOutputIndex` | number | 否 | `multiOutput=true` 时，默认选中第几个输出；不填则为 0 |
| `lookbackHint` | string | 否 | 预热 bar 估算表达式，供 UI 提示用户，如 "timeperiod - 1" |
| `lookbackDefault` | number | 否 | 默认参数下所需历史 bar 数（给回测/选股做数据量检查用） |

### 9.2 params 单项定义

```json
{
  "name": "timeperiod",
  "displayName": "周期",
  "type": "INT",
  "defaultValue": 5,
  "min": 1,
  "max": 500,
  "step": 1
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `name` | string | 参数代码名（透传给 talib 函数） |
| `displayName` | string | 前端标签 |
| `type` | string | `INT` / `FLOAT` / `ENUM` |
| `defaultValue` | number | 默认值 |
| `min` / `max` | number | 数字范围约束 |
| `step` | number | 步进（INT 默认为 1） |
| `enumValues` | array | `type=ENUM` 时必填：`[{value:0, label:"SMA"}, ...]` |

### 9.3 因子结果 key 的统一命名规范

**该规范由因子库、策略管理、选股三个模块共同遵守。** 所有模块在内部缓存、SQL 列名、JSON 字段命名时必须使用同一套规则。

| 因子类型 | 规则 | 示例 |
|----------|------|------|
| 单输出因子（`multiOutput=false`） | `factorKey` + `_` + `paramValue` | `MA_5`、`RSI_14`、`EMA_20` |
| 多输出因子（`multiOutput=true`） | `factorKey` + `_` + `outputLabel` | `MACD_DIF`、`MACD_DEA`、`MACD_HIST`、`KDJ_K`、`KDJ_D`、`KDJ_J` |
| 无参数因子 | `factorKey` | `SAR`、`ATR_14`（参数固定时也可以省略参数部分，如 SAR 即为 SAR） |
| 价格直取类（CLOSE / HIGH / LOW / VOLUME） | `factorKey`（全小写） | `close`、`high`、`low`、`volume` |

**详细规则说明**：

1. **factorKey 标准化**：统一大写，与注册表中 `factorKey` 一致。例如 `MA`、`MACD`、`RSI`。
2. **单输出参数值拼接**：取唯一参数（通常是 `timeperiod`）的整数值，用下划线拼接。例如 `MA` + `_` + `5` = `MA_5`。
3. **多输出 outputLabel 拼接**：取 `outputLabels` 数组中对应 `outputIndex` 的标签值（全大写），用下划线拼接。例如 `MACD` + `_` + `HIST` = `MACD_HIST`。
4. **价格直取类特殊处理**：CLOSE / HIGH / LOW / VOLUME 这四个"因子"本质上是原始行情数据，key 直接用小写的列名（`close` / `high` / `low` / `volume`），便于与 SQL 表中的同名字段直接映射。
5. **参数固定时省略参数**：SAR、ATR_14 等参数固定或已在 factorKey 中隐含周期的因子，省略参数部分。但为了一致性，建议统一保留参数版本（如 `ATR_14` 而非 `ATR`）。

## 11. 因子计算 API（POST）

### 11.1 用途

用于 **单股 + 多因子 + 指定范围** 的批量计算。典型场景：

- 策略配置器的"因子值预览"功能
- 前端手动调试某个指标的值
- **非**选股场景的大批量计算（选股使用独立的指标预计算接口，见选股 PRD）

### 11.2 请求

```
POST /api/compute/factors
Content-Type: application/json
```

```json
{
  "task_id": "uuid",
  "payload": {
    "stock_code": "000001.SZ",
    "ohlcv": [
      { "date": "2026-05-20", "open": 12.5, "high": 13.0, "low": 12.3, "close": 12.8, "volume": 550000 },
      { "date": "2026-05-21", "open": 12.8, "high": 13.2, "low": 12.6, "close": 13.1, "volume": 620000 }
    ],
    "factors": [
      { "factor": "MA",   "params": {"timeperiod": 5} },
      { "factor": "RSI",  "params": {"timeperiod": 14} },
      { "factor": "MACD", "params": {"fastperiod": 12, "slowperiod": 26, "signalperiod": 9}, "output_index": 0 }
    ]
  }
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `stock_code` | string | 否 | 仅用于标识和日志，Python 不做查询 |
| `ohlcv` | array | 是 | 日线数据，按日期升序。每项含 `date / open / high / low / close / volume` |
| `factors` | array | 是 | 要计算的因子列表，每个元素为**因子引用对象**，格式与策略规则树 Schema 中 expression 节点的 `factor` 引用完全一致 |

**因子引用对象的字段（与策略 Schema 保持一致）**：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `factor` | string | 是 | 因子的唯一标识，对应注册表中 `factorKey` 的值（如 `"MA"`、`"MACD"`） |
| `params` | object | 条件必填 | 因子参数。无参数因子（如 CLOSE/HIGH/LOW/VOLUME）可省略；其余必须提供完整参数 |
| `output_index` | number | 条件必填 | 多输出因子（`multiOutput=true`）时必填，指定取第几个输出（0-based）；单输出因子可省略 |

> **统一设计意图**：`factors` 数组中的每个对象，其字段名和含义与策略规则树 Schema §8.4 中的 `factor` 表达式节点完全一致。策略模块从规则树中提取因子引用后，可以**零转换**地传给本 API 做批量计算。

## 12. 因子扩展机制（开发者视角）

**核心约束**：因子计算逻辑一律通过 `akquant.talib` 完成，不自行实现任何算法。注册表仅负责「元数据描述 + akquant.talib 的薄封装」，确保与 akquant 回测引擎口径一致。

### 12.1 新增一个内置因子的步骤

```python
# Step 1: 在 factor_registry.py 中注册
import akquant.talib as talib
from factor_registry import register_factor

@register_factor(
    factor_key="MY_INDICATOR",        # 全局唯一，全大写，对应注册表的 factorKey 和请求的 factor 字段
    display_name="我的自定义指标",
    category="MOMENTUM",              # TREND / MOMENTUM / VOLATILITY / VOLUME / PRICE
    description="指标描述",
    params=[
        {"name": "timeperiod", "displayName": "周期", "type": "INT",
         "defaultValue": 10, "min": 1, "max": 500}
    ],
    inputs=["close"],                 # 声明需要哪些行情列（Python 侧据此从 ohlcv 提取）
    multi_output=False,               # 单输出设 False；多输出时必须设 output_labels
    output_labels=["DIF", "DEA"],     # multi_output=true 时必填，数量需与返回 tuple 一致
    lookback_hint="timeperiod - 1",   # 预热 bar 的估算表达式，供 UI 提示
    lookback_default=9,               # 默认参数下所需历史 bar 数，供回测/选股做数据量检查
)
def compute_my_indicator(close_array, timeperiod=10):
    """
    输入：numpy array（Python 侧完成类型转换，数据按日期升序）
    输出：numpy array（单输出）或 tuple of numpy array（多输出）
    """
    # 一律使用 akquant.talib 完成计算
    result = talib.SOME_FUNC(close_array, timeperiod=timeperiod)
    return result

# 多输出因子示例
@register_factor(
    factor_key="MY_MULTI",
    display_name="多输出指标",
    category="MOMENTUM",
    description="返回两条线",
    params=[{"name": "timeperiod", "displayName": "周期", "type": "INT",
             "defaultValue": 20, "min": 2, "max": 500}],
    inputs=["close"],
    multi_output=True,
    output_labels=["LINE1", "LINE2"],
)
def compute_my_multi(close_array, timeperiod=20):
    line1 = talib.MA(close_array, timeperiod=timeperiod)
    line2 = talib.EMA(close_array, timeperiod=timeperiod)
    return line1, line2   # 返回 tuple，长度与 output_labels 一致
```

### 12.2 注册器约束

- `factor_key` 必须全局唯一，全大写；冲突时报 Warning 并覆盖旧定义
- 参数 `type` 仅支持 `INT` / `FLOAT` / `ENUM`；参数名需与 `akquant.talib` 函数签名一致
- `inputs` 必须是 `["close","high","low","volume"]` 的子集（对应 ohlcv 中的可提取列）
- `multi_output=true` 时必须设置 `output_labels`，且计算函数返回的 tuple 长度与之一致
- 计算函数必须接收 numpy array 输入并返回 numpy array 输出（或 tuple）
- 计算函数内部**不得**自行实现任何指标算法，一律调用 `akquant.talib.*` 函数

### 12.3 因子引用→计算→结果 key 的完整链路

策略规则树 Schema §8.4 中的因子引用 → 本 API 请求 → 结果 key，三者映射关系明确：

```
策略规则树中的因子引用节点：
  { "factor": "MACD", "params": {fastperiod:12, slowperiod:26, signalperiod:9}, "output_index": 0 }
                            ↓ 零转换传入本 API
POST /api/compute/factors 的 factors 数组中的元素：
  { "factor": "MACD", "params": {fastperiod:12, slowperiod:26, signalperiod:9}, "output_index": 0 }
                            ↓ Python 端按 factorKey 查注册表，用 params 调 akquant.talib
响应 results 中的 key（按 §9.3 命名规则）：
  MACD_DIF  → 由 output_index=0 映射到 outputLabels[0]="DIF"，拼接为 MACD_DIF
  MACD_DEA  → 多输出因子计算时自动返回所有输出
  MACD_HIST → （即使请求方只声明了 output_index=0，也返回全部输出）
```

### 13.2 需在开发阶段细化（不阻塞开发，但需写代码时决策）

| 项 | 影响 | 决策建议 |
|---|---|---|
| 因子引用→计算→结果 key 的映射实现 | ✅ **核心开发项** | Python 端需实现统一的 `result_key(factor, params, output_index)` 工具函数，输入是策略规则树中提取的因子引用对象（即本 API 的 `factors` 元素），输出是 `results` 中的 key（如 `MA_5`、`MACD_DIF`）。策略、选股、回测三方共享此实现的逻辑副本 |
| `ohlcv` 数据字段校验 | 低 | Python 收到 JSON 后需验证每个 bar 是否含 `open/high/low/close/volume`；缺 `high/low` 时 BOLL/ATR/SAR/KDJ/ADX/PLUS_DI/MINUS_DI/WILLR/CCI 无法计算，需有明确的错误响应（code=201） |
| 注册表缓存失效策略 | 低 | 本 PRD 定义 1 小时 TTL；Python 端更新因子后无主动通知机制。V2.1 不做热更新，部署重启后自然刷新，够用 |
| 因子计算函数的异常收敛 | 中 | `akquant.talib` 在输入数据长度不足时返回全 NaN；极端情况（1 根 bar）也需正常返回 `[null]` 而非抛异常。统一包裹 try/except，将异常转为 code=201 错误响应 |
| 多输出因子是否只返回指定输出 | 中 | 策略模块可能只关心 MACD_DIF，但为确保前端预览和其他场景可用，**建议始终返回全部输出**（如 MACD 永远返回 DIF/DEA/HIST 三条结果），`output_index` 仅用于调用方标识自己关心的列，不影响 Python 端的计算范围 |

### 13.3 本次范围外，需在后续版本明确

| 项 | 本版处理方式 | 后续版本需考虑 |
|---|---|---|
| 基本面因子（PE / PB / ROE 等） | ❌ 不做 | 需要确认数据源（Tushare 能否直接拿到？字段名？更新频率）；因子分类体系需扩展为 `FUNDAMENTAL` |
| 自定义因子（用户上传 Python 代码） | ❌ 不做 | 安全风险极高，建议永远不在生产环境开放 |
| 因子值持久化到数据库 | ❌ 不做 | 因子计算是 on-the-fly；选股有独立的指标预计算表（但预计算表的列名需遵循本 PRD §9.3 的命名规则） |
| 因子间依赖（如某因子的输入=另一因子的输出） | ❌ 不做 | 当前设计要求因子输入只能是原始行情字段 |
| 因子回测（单独评估单个因子的预测能力） | ❌ 不做 | 需要单独的因子回测模块 |

### 13.4 结论

**✅ 需求充分，可以进入开发。** 关键架构决策均已明确：

1. **因子定义即 akquant 薄封装** — 20 个因子的计算逻辑全部来自 `akquant.talib`，无自行维护算法的成本，也保证了与 akquant 回测引擎的计算口径一致
2. **因子引用格式已与策略 Schema 对齐** — 请求中 `factors` 数组的每个元素就是策略规则树中的 `factor` 引用节点，策略模块从规则树提取因子后可直接传入本 API，无需格式转换
3. **结果 key 命名规则统一** — §9.3 的命名规则是因子库、策略管理、选股三个模块共享的"契约"，任何对因子命名规则的修改都必须同时更新这三个模块的映射逻辑
