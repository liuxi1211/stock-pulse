## 1. 产品定位

把量化投资拆成**三个可独立操作的闭环**，每个闭环都是一条"想法 → 验证 → 复用"的链路：

| 闭环 | 入口 | 产出 |
|---|---|---|
| 因子库 | 指标/因子定义 | 可复用的计算单元，前端可配置参数 |
| 策略管理 | 组合因子构建策略 | 规则树 JSON，版本化管理 |
| 回测中心 | 策略 + 历史数据 | 绩效报告 / 参数优化 / Walk-forward 分析 |
| 选股模块 | 策略规则 + 预计算指标 | 符合条件的股票列表 |

核心价值用一句话总结：**让技术指标可配置，让策略规则可回测，让选股条件可复用。**

## 3. 基础设施约定（跨模块共享）

### 3.1 Java ↔ Python 通信协议

#### 3.1.1 服务地址

| 环境 | 地址 |
|---|---|
| 开发 / 默认 | http://127.0.0.1:8000 |
| 生产可配置 | 通过 application.properties 配置 `python.compute.url` |

#### 3.1.2 统一请求格式

所有 POST 请求遵循以下外层结构：

```json
{
  "task_id": "uuid-generated-by-java",
  "timestamp": 1780000000,
  "payload": {
    "_comment": "具体参数由各模块定义"
  }
}
```

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `task_id` | 是 | Java 生成的任务唯一标识，用于幂等/追踪 |
| `timestamp` | 否 | 请求时间戳（秒），用于日志追溯 |
| `payload` | 是 | 具体业务参数，由各模块定义 |

#### 3.1.3 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "task_id": "uuid-same-as-request",
    "compute_ms": 125,
    "_comment": "具体业务数据由各模块定义"
  }
}
```

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `code` | 是 | 0 = 成功，非 0 = 错误 |
| `message` | 是 | 成功 = "success"，失败 = 错误描述 |
| `data.task_id` | 是 | 回传请求中的 task_id，便于追踪 |
| `data.compute_ms` | 否 | 本次计算耗时（毫秒），用于性能监控 |
| `data.*` | 子模块定义 | 具体业务结果 |

> **非 0 code 的统一约定**：code = 1xx 为参数错误，2xx 为计算引擎错误，3xx 为数据错误。具体子码在各模块中定义。

#### 3.1.4 本模块涉及的接口清单

| 接口 | 说明 |
|---|---|
| `POST /api/compute/indicators` | 计算日线技术指标（单股全指标） |
| `GET /api/compute/health` | 健康检查 |

## 4. 跨模块关键决策（开发前必须拍板）

以下决策点影响多个模块的表结构和代码逻辑，在 Phase 1 开始前必须确认：

| 决策项 | 结论 | 影响模块 |
|--------|------|----------|
| 止损止盈的判断方式 | **止损用 `low`（日内最低），止盈用 `high`（日内最高），条件卖出用 `close`（收盘），`max_holding_days` 不判断价格直接平仓** | 策略管理、回测中心 |
| `rebalance_frequency_days` 的归属 | **从策略 JSON 的 `position_sizing` 中移除，仅保留在回测配置的 `rebalance` 中**（这是回测参数，不是策略的固定属性） | 策略管理、回测中心 |
| `quant_strategy_saved` 表是否保留 | **移除**。用 `quant_strategy` 表的 DRAFT 状态策略统一承载选股草稿场景 | 策略管理 |
| 因子结果 key 的命名规范 | **统一规则**：单输出因子 = `factorKey_paramValue`（如 `MA_5`、`RSI_14`）；多输出因子 = `factorKey_outputLabel`（如 `MACD_DIF`、`MACD_HIST`）。详见因子库 PRD | 因子库、策略管理、选股 |
| INDEX_300 / INDEX_500 回测是否放行 | **放行**。300 只 × 750 交易日 ≈ 1.35 MB，HTTP 可承载 | 回测中心 |

## 7. 需求概述

选股和回测的核心区别：

| 维度 | 回测 | 选股 |
|---|---|---|
| 数据来源 | 全量历史 bar 传入 Python（嵌入式数据） | 指标值预存在 Java 的 stock_indicator_daily 表 |
| 计算方式 | Python 端按需实时计算因子值 | Java 端 SQL 查询预计算结果 |
| 选股范围 | 通常 < 500 只（受限） | 可支持全市场（3000+ 只） |
| 时间维度 | 连续时间段（区间内每根 bar 都评估） | 选定的某个日期或区间 |
| 输出 | 净值曲线、交易流水、指标 | 股票代码列表 + 当天触发的条件说明 |

一句话：**回测是 Python 端做的"时间序列仿真"，选股是 Java 端做的"快照查询"。**

## 8. 技术指标预计算

### 8.1 stock_indicator_daily 表设计

这张表是选股的核心基础设施。每一行 = 某只股票 × 某一天 × 一组技术指标值。

```sql
CREATE TABLE stock_indicator_daily (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    stock_code      TEXT NOT NULL,
    trade_date      DATE NOT NULL,

    -- 基础行情（冗余存一份，SQL 不用 JOIN daily_quote）
    open            REAL,
    high            REAL,
    low             REAL,
    close           REAL,
    volume          REAL,

    -- 趋势类
    ma_5            REAL,
    ma_10           REAL,
    ma_20           REAL,
    ma_60           REAL,
    ma_120          REAL,
    ma_250          REAL,
    ema_5           REAL,
    ema_10          REAL,
    ema_20          REAL,
    ema_60          REAL,
    sar             REAL,

    -- 动量类
    rsi_6           REAL,
    rsi_14          REAL,
    rsi_28          REAL,
    macd_dif        REAL,   -- DIF = EMA(12) - EMA(26)
    macd_dea        REAL,   -- DEA = EMA(DIF, 9)
    macd_hist       REAL,   -- HIST = DIF - DEA
    kdj_k           REAL,   -- KDJ 的 K（9,3,3 标准参数）
    kdj_d           REAL,
    kdj_j           REAL,
    adx_14          REAL,
    plus_di_14      REAL,
    minus_di_14     REAL,
    willr_14        REAL,
    cci_14          REAL,

    -- 波动类
    atr_14          REAL,

    -- 成交量类
    vol_ma_5        REAL,
    vol_ma_20       REAL,
    vol_ma_60       REAL,

    -- 派生：方便直接写 SQL（例如直接比 "price_above_ma20 = 1"）
    -- 不建议在此表中存太多派生字段，让 SQL 构建器灵活处理

    UNIQUE(stock_code, trade_date)
);

CREATE INDEX idx_sid_date ON stock_indicator_daily(stock_code, trade_date);
CREATE INDEX idx_date ON stock_indicator_daily(trade_date);
```

### 8.2 预计算流程

```
Java 调度（每天收盘后 T+1 凌晨 1:00，或手动触发）
    ↓
1. 从 daily_quote 读取所有股票的最新日线数据
   （每只股票取最近 300 个交易日，供计算长周期指标如 MA250）
    ↓
2. 分批调用 Python /api/compute/indicators
   （每批 50 只股票，避免单次 payload 过大）
    ↓
3. 收到每只股票的指标值数组后，批量 UPSERT 到 stock_indicator_daily
    ↓
4. 记录本次预计算的日期范围和耗时（日志）
```

### 8.3 Python 指标预计算接口

```
POST /api/compute/indicators
```

请求：

```json
{
  "task_id": "uuid",
  "payload": {
    "stocks": [
      {
        "stock_code": "000001.SZ",
        "ohlcv": [ {date, open, high, low, close, volume}, ... ]
      },
      /* 一批最多 50 只 */
    ]
  }
}
```

响应：

```json
{
  "code": 0,
  "data": {
    "task_id": "uuid",
    "results": {
      "000001.SZ": [
        {
          "date": "2026-05-20",
          "ma_5": 12.80,
          "ma_20": 12.50,
          "macd_dif": 0.15,
          "macd_dea": 0.12,
          "rsi_14": 65.3,
          "...": "..."
        }
      ]
    }
  }
}
```

**关键设计决策**：返回的字段 key（`ma_5`、`macd_dif` 等）与 `stock_indicator_daily` 的列名**一一对应**。Java 端收到后可以直接批量 INSERT/UPDATE，无需做字段映射。

**缺点**：因子注册表中新增一个因子（比如 MA_30）时，需要同步在 stock_indicator_daily 表中增加列。这是"列式存储 + 静态表结构"方案的固有约束。

**替代方案**：把指标存为行式表（`stock_code, trade_date, indicator_key, indicator_value`），这样不需要修改表结构。但这样一张表会有 3000 只 × 750 天 × 约 30 个指标 = **约 6750 万行**，查询时需要自 JOIN 多次，SQL 复杂度和性能都会显著下降。

**最终选择**：列式表（本 PRD §8.1）。理由：SQL 简单、查询性能好、开发成本低。新增指标的场景不频繁（V2.1 初期一次性定义好所有常用指标即可）。

### 8.4 预计算支持的指标清单

| SQL 列名 | 对应因子 | 参数 | 说明 |
|---|---|---|---|
| `ma_5, ma_10, ma_20, ma_60, ma_120, ma_250` | MA | timeperiod | 简单移动均线 |
| `ema_5, ema_10, ema_20, ema_60` | EMA | timeperiod | 指数移动均线 |
| `sar` | SAR | 固定参数 | 抛物转向 |
| `rsi_6, rsi_14, rsi_28` | RSI | timeperiod | 相对强弱指数 |
| `macd_dif, macd_dea, macd_hist` | MACD | 12, 26, 9 | MACD 三线 |
| `kdj_k, kdj_d, kdj_j` | KDJ | 9, 3, 3 | KDJ 三线 |
| `adx_14, plus_di_14, minus_di_14` | ADX / +DI / -DI | 14 | 趋向指标 |
| `willr_14` | WILLR | 14 | 威廉指标 |

## 9. 条件 → SQL 翻译引擎

### 9.1 设计思路

把规则树翻译成 SQL 的过程，本质上是把**每棵表达式子树**映射为**一个 SQL 表达式**。

```
规则树节点                    →    SQL 片段
───────────────────────────────────────────────
静态值 value:70               →    70.0
因子引用 MA(5)                →    sid.ma_5
因子引用 MACD 取 HIST          →    sid.macd_hist
算术运算 MA(5) * 1.5          →    (sid.ma_5 * 1.5)
比较 MA(5) > MA(20)           →    (sid.ma_5 > sid.ma_20)
cross_up MA(5) vs MA(20)      →    (sid.ma_5 > sid.ma_20 AND sid_prev.ma_5 <= sid_prev.ma_20)
AND 组合                      →    (cond1 AND cond2 AND ...)
OR  组合                      →    (cond1 OR cond2 OR ...)
```

### 9.2 因子 → 列名映射器

```java
// Java 端的核心类
public class FactorColumnMapper {
    /**
     * 将一个因子引用节点映射为 stock_indicator_daily 的列名
     *
     * @param factorKey    如 "MA"
     * @param params       如 {"timeperiod": 5}
     * @param outputIndex  多输出时的下标
     * @return 列名（如 "ma_5"），如果不支持（如非标准参数）抛 UnsupportedParameterException
     */
    public static String mapToColumn(String factorKey, Map<String, Object> params, Integer outputIndex) {
```

### 9.3 完整 SQL 生成示例

**输入策略**：MA5 上穿 MA20 且成交量 > 1.5 × VOL_MA20

**规则树（简化后）**：

```
AND
├── compare: MA(5) cross_up MA(20)
└── compare: VOLUME > (VOL_MA(20) × 1.5)
```

**生成的 SQL（快照模式：某一天）**：

```sql
SELECT
    s.stock_code,
    s.name,
    sid.close         AS current_close,
    (sid.close / sid_prev.close - 1) * 100 AS daily_change_pct
FROM stock_indicator_daily sid
JOIN stock_indicator_daily sid_prev
  ON sid_prev.stock_code = sid.stock_code
 AND sid_prev.trade_date = (
        SELECT MAX(trade_date)
        FROM stock_indicator_daily
        WHERE stock_code = sid.stock_code
          AND trade_date < sid.trade_date
    )
JOIN stock_basic s
  ON s.ts_code = sid.stock_code
WHERE sid.trade_date = '2026-05-20'  -- 查询日期
  AND sid.ma_5 IS NOT NULL
  AND sid.ma_20 IS NOT NULL
  AND sid_prev.ma_5 IS NOT NULL
  AND sid_prev.ma_20 IS NOT NULL
  -- 条件 1: MA5 上穿 MA20
  AND (sid.ma_5 > sid.ma_20 AND sid_prev.ma_5 <= sid_prev.ma_20)
  -- 条件 2: VOLUME > 1.5 * VOL_MA20
  AND (sid.volume > (sid.vol_ma_20 * 1.5))
  -- 前置过滤：剔 ST + 剔次新 + 成交额过滤
  AND s.name NOT LIKE '%ST%'
  AND s.list_date <= date('2026-05-20', '-365 days')
  AND (sid.close * sid.volume) >= 50000000
ORDER BY (sid.close * sid.volume) DESC
LIMIT 100;
```

### 9.4 cross_up / cross_down 的 JOIN 策略

从上面 SQL 可以看到一个性能问题：每个 cross_up/cross_down 都需要 JOIN 一次 `sid_prev`。如果一个策略中有多个 cross_up，会产生多次自 JOIN。

**优化策略**：

1. 如果规则树中**有任何** cross_up / cross_down 节点，全局 JOIN 一次 `sid_prev`，所有穿越条件共享这一份前一交易日数据
2. 如果没有穿越条件，完全不 JOIN `sid_prev`，最简单的单表查询

**更极端的优化**：`stock_indicator_daily` 表中冗余 `prev_ma_5, prev_ma_20, ...` 列（预计算时写入）。这样完全不需要 JOIN。但列数会翻倍，维护成本高。V2.1 阶段用 JOIN 方案，性能不够再考虑冗余列。

### 9.5 区间模式（Range Mode）

默认是**快照模式**（查询某日满足条件的股票）。也支持**区间模式**（查询某个日期区间内**任一天**满足条件的股票，返回每只股票 + 首次触发日期）。

SQL 示例：

```sql
SELECT
    s.stock_code,
    s.name,
    MIN(sid.trade_date) AS first_match_date,
    COUNT(*) AS match_days_count
FROM stock_indicator_daily sid
-- 如果有 cross_up：
LEFT JOIN stock_indicator_daily sid_prev
  ON sid_prev.stock_code = sid.stock_code
 AND sid_prev.trade_date = (SELECT MAX(trade_date) FROM ...)
JOIN stock_basic s ON s.ts_code = sid.stock_code
WHERE sid.trade_date BETWEEN '2026-01-01' AND '2026-05-20'
  AND /* 同样的条件树 ... */
  AND /* 同样的前置过滤 ... */
GROUP BY s.stock_code, s.name
ORDER BY first_match_date DESC
LIMIT 200;
```

```
1. **四舍五入到最接近的标准参数**。例：MA(8) → MA(10)、RSI(9) → RSI(6)、RSI(18) → RSI(14)
2. **前端弹框提示**：「MA(8) 未预计算，已降级为 MA(10)，结果为近似值」
3. **SQL 注释记录**：生成的 SQL 中加入注释 `-- NOTE: MA(8) 降级为 ma_10`，便于排查和审计
4. **完全无法匹配的情况**（如 MA(2)、EMA(3)）抛 `UnsupportedParameterException`，前端展示「该因子参数未在选股中支持，请改用标准参数」
```
