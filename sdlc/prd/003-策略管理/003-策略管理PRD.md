## 1. 产品定位（背景参考）

把量化投资拆成**三个可独立操作的闭环**，每个闭环都是一条"想法 → 验证 → 复用"的链路：

| 闭环 | 入口 | 产出 |
|---|---|---|
| 因子库 | 指标/因子定义 | 可复用的计算单元，前端可配置参数 |
| 策略管理 | 组合因子构建策略 | 规则树 JSON，版本化管理 |
| 回测中心 | 策略 + 历史数据 | 绩效报告 / 参数优化 / Walk-forward 分析 |
| 选股模块 | 策略规则 + 预计算指标 | 符合条件的股票列表 |

核心价值用一句话总结：**让技术指标可配置，让策略规则可回测，让选股条件可复用。**

## 3. 基础设施约定（背景参考 · 跨模块共享）

### 3.1 Java ↔ Python 通信协议

#### 3.1.1 服务地址

| 环境 | 地址 |
|---|---|
| 开发 / 默认 | http://127.0.0.1:8000 |
| 生产可配置 | 通过 application.properties 配置 `python.compute.url` |

#### 3.1.2 统一请求格式

所有 POST 请求遵循以下外层结构：

```json
{
  "task_id": "uuid-generated-by-java",
  "timestamp": 1780000000,
  "payload": {
    "_comment": "具体参数由各模块定义"
  }
}
```

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `task_id` | 是 | Java 生成的任务唯一标识，用于幂等/追踪 |
| `timestamp` | 否 | 请求时间戳（秒），用于日志追溯 |
| `payload` | 是 | 具体业务参数，由各模块定义 |

#### 3.1.3 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "task_id": "uuid-same-as-request",
    "compute_ms": 125,
    "_comment": "具体业务数据由各模块定义"
  }
}
```

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `code` | 是 | 0 = 成功，非 0 = 错误 |
| `message` | 是 | 成功 = "success"，失败 = 错误描述 |
| `data.task_id` | 是 | 回传请求中的 task_id，便于追踪 |
| `data.compute_ms` | 否 | 本次计算耗时（毫秒），用于性能监控 |
| `data.*` | 子模块定义 | 具体业务结果 |

> **非 0 code 的统一约定**：code = 1xx 为参数错误，2xx 为计算引擎错误，3xx 为数据错误。具体子码在各模块中定义。

#### 3.1.4 本模块涉及的接口清单

| 接口 | 说明 |
|---|---|
| `POST /api/compute/factors` | 批量计算因子值（单股多因子） |
| `POST /api/compute/backtest` | 执行一次回测（使用本模块产出的策略 JSON） |
| `POST /api/compute/grid-search` | 执行参数网格搜索（使用本模块的参数提取结果） |
| `GET /api/compute/health` | 健康检查 |

## 4. 跨模块关键决策（开发前必须拍板）

以下决策点影响多个模块的表结构和代码逻辑，在 Phase 1 开始前必须确认：

| 决策项 | 结论 | 影响模块 |
|--------|------|----------|
| 止损止盈的判断方式 | **止损用 `low`（日内最低），止盈用 `high`（日内最高），条件卖出用 `close`（收盘），`max_holding_days` 不判断价格直接平仓** | 策略管理、回测中心 |
| `rebalance_frequency_days` 的归属 | **从策略 JSON 的 `position_sizing` 中移除，仅保留在回测配置的 `rebalance` 中**（这是回测参数，不是策略的固定属性） | 策略管理、回测中心 |
| `quant_strategy_saved` 表是否保留 | **移除**。用 `quant_strategy` 表的 DRAFT 状态策略统一承载选股草稿场景 | 策略管理 |
| 因子结果 key 的命名规范 | **统一规则**：单输出因子 = `factorKey_paramValue`（如 `MA_5`、`RSI_14`）；多输出因子 = `factorKey_outputLabel`（如 `MACD_DIF`、`MACD_HIST`）。详见因子库 PRD §9.3 | 因子库、策略管理、选股 |
| INDEX_300 / INDEX_500 回测是否放行 | **放行**。300 只 × 750 交易日 ≈ 1.35 MB，HTTP 可承载 | 回测中心 |

## 7. 需求概述

策略管理模块解决一个核心问题：**如何用结构化的 JSON 描述一个完整的量化策略**，使得这个 JSON 既可以被前端可视化编辑，也可以被 Python 解析后交给 akquant 执行。

每个策略由以下要素组成：

```
策略
├── 基本信息（名称、描述、分类）
├── 选股范围（全市场 / 沪深300 / 自选股 / 自定义代码列表）
├── 前置过滤（剔除 ST / 剔除次新股 / 流动性过滤）
├── 买入规则（AND/OR 嵌套条件树，叶子节点是"因子 vs 值"或"因子 vs 因子"比较）
├── 卖出规则（结构同买入，含止损止盈 + 条件卖出）
├── 仓位管理（单票最大仓位 / 最大持仓数 / 调仓频率）
├── 参数（均线天数、阈值等可调参数，从规则树中提取，用于参数优化）
└── 版本（每次修改自动快照，可追溯 / 可对比 / 可回滚）
```

## 8. 核心数据模型：规则树（Rule Tree）

规则树是整个系统最重要的数据结构。回测中心、选股模块都**直接复用这一套 Schema**，不在各自 PRD 中重新定义。

### 8.1 三层抽象

```
条件树（Condition Tree）← 顶层：AND / OR 组合 + 叶子 compare
    └── 比较节点（Compare Node） ← 左右各一个表达式节点 + 操作符
         └── 表达式节点（Expression Node） ← 静态值 / 因子引用 / 算术运算（递归）
```

### 8.2 顶层：逻辑组合节点（Condition Node）

```json
{
  "operator": "AND",
  "conditions": [
    { /* 子节点：可以是 compare 叶子节点，也可以是嵌套的 AND/OR */ }
  ]
}
```

| 字段 | 约束 | 说明 |
|---|---|---|
| `operator` | 必填，`"AND"` 或 `"OR"` | 子条件之间的逻辑关系 |
| `conditions` | 必填，数组，长度 ≥ 1 | 子条件列表，每个子项可以是组合节点或比较节点 |

### 8.3 叶子节点：比较节点（Compare Node）

```json
{
  "type": "compare",
  "left":   { /* 表达式节点 */ },
  "comparator": ">",
  "right":  { /* 表达式节点 */ }
}
```

### 8.4 表达式节点（Expression Node）

有且仅有三种形态（三选一，互斥）：

| 形态 | JSON 结构 | 用途 |
|---|---|---|
| 静态值 | `{"value": 70}` | 阈值比较 |
| 因子引用 | `{"factor": "RSI", "params": {"timeperiod": 14}}` | 取某因子当前值 |
| 算术运算 | `{"op": "*", "left": {...}, "right": {...}}` | 因子间 / 因子与值的四则运算 |

**因子引用的完整字段**：

```json
{
  "factor": "MACD",
  "params": {"fastperiod": 12, "slowperiod": 26, "signalperiod": 9},
  "output_index": 2
}
```

| 字段 | 约束 | 说明 |
|---|---|---|
| `factor` | 必填，因子注册表中存在的 `factorKey` | 引用因子 |
| `params` | 可选（无参数因子可省略），key-value | 透传给因子计算函数 |
| `output_index` | 条件必填，`multiOutput=true` 时必填 | 取第几个输出（0-indexed） |
| `inputs` | 可选，高级 | 覆盖默认输入源，如改用 volume 计算 MA |

### 8.5 比较操作符（Comparator）

| 值 | 含义 | 支持的表达式 |
|---|---|---|
| `>` | 大于 | 任意表达式 |
| `<` | 小于 | 任意表达式 |
| `>=` | 大于等于 | 任意表达式 |
| `<=` | 小于等于 | 任意表达式 |
| `==` | 等于 | 任意表达式（注意：浮点等于需谨慎使用） |
| `!=` | 不等于 | 任意表达式 |
| `cross_up` | 上穿（金叉） | left/right 必须都是因子引用（不支持算术表达式参与穿越） |
| `cross_down` | 下穿（死叉） | 同上 |

**cross_up 的语义**：

```
当前 bar:  left_value > right_value
上一 bar:  prev_left_value <= prev_right_value
两者同时满足 → cross_up = true
```

> 回测中需要维护上一根 bar 的因子值；选股中需要 JOIN 前一交易日的指标数据。

### 8.6 完整示例：双均线 + 成交量放大策略

```json
{
  "name": "双均线交叉策略",
  "description": "MA5 上穿 MA20 买入，MA5 下穿 MA20 卖出；成交量配合放大时加强信号",

  "universe": "INDEX_300",
  "pre_filters": [
    {"type": "exclude_st"},
    {"type": "exclude_recent_ipo", "days": 365},
    {"type": "min_turnover", "min_amount": 50000000}
  ],

  "buy_rules": {
    "operator": "AND",
    "conditions": [
      {
        "type": "compare",
        "left":  {"factor": "MA", "params": {"timeperiod": 5}},
        "comparator": "cross_up",
        "right": {"factor": "MA", "params": {"timeperiod": 20}}
      },
      {
        "type": "compare",
        "left":  {"factor": "VOLUME"},
        "comparator": ">",
        "right": {
          "op": "*",
          "left":  {"factor": "VOL_MA", "params": {"timeperiod": 20}},
          "right": {"value": 1.5}
        }
      }
    ]
  },

  "sell_rules": {
    "operator": "OR",
    "conditions": [
      {
        "type": "compare",
        "left":  {"factor": "MA", "params": {"timeperiod": 5}},
        "comparator": "cross_down",
        "right": {"factor": "MA", "params": {"timeperiod": 20}}
      },
      {
        "type": "stop_loss",
        "params": {"percent": 0.08}
      },
      {
        "type": "take_profit",
        "params": {"percent": 0.20}
      }
    ]
  },

  "position_sizing": {
    "max_single_position_pct": 0.10,
    "max_total_positions": 10,
    "min_holding_days": 1
  }
}
```

### 8.7 特殊卖出规则节点（非条件树）

卖出规则中除了通用的 compare 节点外，还支持以下特殊节点。**所有特殊节点放在 `sell_rules.conditions` 中与普通 compare 节点混用，`operator` 为 OR（任一条件满足即卖出）**。

| type | 参数 | 含义 | **执行逻辑（回测/实盘统一）** |
|---|---|---|---|
| `stop_loss` | `{"percent": 0.08}` | 持仓亏损 8% 强制卖出 | **用当日 `low` 判断**：若 `low <= entry_price × (1 − percent)`，触发止损，以当日 `close` 成交 |
| `take_profit` | `{"percent": 0.20}` | 持仓盈利 20% 止盈 | **用当日 `high` 判断**：若 `high >= entry_price × (1 + percent)`，触发止盈，以当日 `close` 成交 |
| `max_holding_days` | `{"days": 20}` | 持仓超过 20 天强制卖出 | **不判断价格**：持仓 bar 计数达到 days 时，以当日 `close` 直接平仓 |

**普通条件卖出（sell_rules 中的 compare 节点）**：用当日 `close` 判断条件是否满足，以当日 `close` 成交。

> 设计说明：这些规则不参与条件树的递归解析，在 Python 端作为独立的交易逻辑处理。`min_holding_days`（持仓最少天数，在 `position_sizing` 中定义）是一个全局约束：在持仓未达到该天数前，所有卖出规则（包括止损止盈）均不生效。

### 9.2 quant_strategy_version — 策略版本快照

```sql
CREATE TABLE quant_strategy_version (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    strategy_id     INTEGER NOT NULL,
    version         INTEGER NOT NULL,                       -- 版本号，从 1 开始
    buy_rules       TEXT NOT NULL,                          -- 本次的规则快照
    sell_rules      TEXT NOT NULL,
    position_sizing TEXT,
    pre_filters     TEXT,
    universe        TEXT,
    change_note     TEXT,                                   -- 用户填写的变更说明
    backtest_id     INTEGER,                                -- 关联的验证回测（可选）
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(strategy_id, version),
    FOREIGN KEY (strategy_id) REFERENCES quant_strategy(id) ON DELETE CASCADE
);
```

> **设计决策**：不做"细粒度 diff"存储（成本太高）。每次版本快照直接存完整规则 JSON。由于 JSON 体积不大（单策略通常 < 5KB），存 100 个版本也只有 500KB，完全可接受。

---

> **关于「选股草稿」场景的说明**：V2.1 不再单独建 `quant_strategy_saved` 表。选股页保存的条件草稿直接写入 `quant_strategy` 表（`status=DRAFT`，`sell_rules`/`position_sizing` 可为空或缺省值；`category=SCREEN_DRAFT`）。这样草稿和完整策略共用一套表结构，「从选股条件升级为完整策略」的流程也更自然。

## 10. 策略生命周期

```
  [新建] → DRAFT ────────编辑───────┐
              │                      │
              ├─ 回测验证（通过） ─→ ACTIVE
              │                        │
              └─ 手动归档            ├─ 编辑（自动建新版本，旧版本保留）
                 ↑                     │
                 └──────── 手动 ── ARCHIVED
```

| 状态 | 说明 | 允许的操作 |
|---|---|---|
| DRAFT | 草稿，未通过回测验证 | 编辑规则 / 保存版本 / 回测 |
| ACTIVE | 已通过回测，标记为可用策略 | 编辑规则（自动建新版本） / 回测 / 选股 |
| ARCHIVED | 归档，不再使用 | 只读查看 |

> **关于 ACTIVE 的定义**：ACTIVE 不代表"自动交易"或"信号推送"（V2.1 不做信号中心），只表示该策略已通过至少一次回测，标记为可复用策略，可在选股/回测界面直接选择使用。

### 11.2 条件行：三段式结构

每个 `compare` 节点渲染为一行，由**左表达式 + 操作符 + 右表达式**三个独立编辑器组成。

```
┌─ 条件行（完整宽度）───────────────────────────────────────────────────────────────────┐

│  │ [类型: 因子引用 ▼] │   │ [ >      ▼] │   │ [类型: 静态值 ▼] │                       │
│  │                  │   │ [ <      ]    │   │                 │                       │
│  │   因子：[MA ▼]   │   │ [ >=     ]    │   │   值：[70]      │                       │
│  │   周期：[5]      │   │ [ <=     ]    │   │                 │                       │
│  │                  │   │ [ ==     ]    │   │ [类型切换为：]   │                       │
│  │ [→ 类型切换为：]   │   │ [ !=     ]    │   │  · 因子引用     │                       │
```

### 11.9 前端实现建议（技术方案）

1. **单一 JSON 对象作为状态**：所有规则数据保存在 `window.strategyJson` 中。用户操作（修改因子、切换操作符、添加/删除条件）只修改此对象，然后触发局部重渲染。这样确保右侧预览区的 JSON 始终与编辑区同步。
2. **递归渲染函数**：写一个 `renderConditionNode(node, depth, container)` 函数，根据节点类型（`operator` + `conditions` = 嵌套组 / `type="compare"` = 条件行 / 因子/值/算术表达式 = 表达式）递归渲染。
3. **事件委托**：使用 `data-action` 属性（如 `data-action="add-condition"`、`data-action="delete-condition"`、`data-action="change-operator"`、`data-action="change-expression-type"`）标记按钮，在顶层 DOM 节点统一监听事件，避免为每个新节点重复绑定。
4. **CSS Grid + Flex 布局**：条件行用 Grid 实现「左表达式 40% / 操作符 15% / 右表达式 40% / 删除按钮 5%」的等宽布局；嵌套组用 padding-left + 左侧 border-left 实现层级视觉。
5. **深度限制**：嵌套组最大深度 3 层，算术表达式最大深度 3 层，防止表达式过于复杂难以维护。
6. **因子元数据缓存**：页面加载时从 Java 端（经 Java 代理获取 Python 注册表）拉取一次因子清单，缓存在 `window.factorRegistry`，所有表达式编辑器共享此缓存。

### 12.2 FactorSpec — 因子规格（可哈希、可去重）

```python
from dataclasses import dataclass, field
from typing import Optional, Dict, FrozenSet


@dataclass(frozen=True)  # 不可变 → 可放入 set() 自动去重
class FactorSpec:
    factor_key: str                      # "MA", "MACD", "VOLUME", "CLOSE"
    params: Dict[str, float]             # {"timeperiod": 5}，无参因子为 {}
    output_index: Optional[int] = None   # MACD=0(DIF)/1(DEA)/2(HIST); 单输出因子为 None

    @property
    def result_key(self) -> str:
        """生成因子计算结果的 key，与因子库 §9.3 命名规则一致。
        单输出: "MA_5", "RSI_14"
        多输出: "MACD_DIF", "KDJ_K"
        价格直取: "close", "high", "low", "volume"
        """
        # 价格直取类特殊处理
        if self.factor_key in {"CLOSE", "HIGH", "LOW", "VOLUME"}:
            return self.factor_key.lower()
        # 多输出因子 → 用 output_label
        if self.output_index is not None:
            registry_entry = FACTOR_REGISTRY[self.factor_key]
            label = registry_entry["outputLabels"][self.output_index]
            return f"{self.factor_key}_{label}"
        # 单输出因子 → 拼接参数值（取第一个参数的整数值）
        if self.params:
            first_param_value = next(iter(self.params.values()))
            return f"{self.factor_key}_{int(first_param_value)}"
        # 无参因子（如 SAR）
        return self.factor_key

    @classmethod
    def from_node(cls, expr_node: dict) -> "FactorSpec":
        """从规则树中的因子引用节点构造 FactorSpec"""
        return cls(
            factor_key=expr_node["factor"],
            params=dict(expr_node.get("params", {})),
            output_index=expr_node.get("output_index")
        )


# ── 提取逻辑 ─────────────────────────────────────────────────
class FactorSpecExtractor:
    """遍历规则树，收集所有 FactorSpec（自动去重）"""

    def extract(self, node: dict) -> FrozenSet[FactorSpec]:
        specs = set()
        self._collect(node, specs)
        return frozenset(specs)

    def _collect(self, node: dict, specs: set) -> None:
        # 因子引用节点 → 生成 FactorSpec
        if "factor" in node:
            specs.add(FactorSpec.from_node(node))
            return
        # 逻辑组合节点 → 递归遍历子条件
        if "operator" in node and "conditions" in node:
            for child in node["conditions"]:
                self._collect(child, specs)
            return
        # compare 节点 → 递归遍历左右表达式
        if node.get("type") == "compare":
            self._collect(node["left"], specs)
            self._collect(node["right"], specs)
            return
        # 算术表达式节点 → 递归遍历左右
        if "op" in node:
            self._collect(node["left"], specs)
            self._collect(node["right"], specs)
            return
        # value 静态值节点 → 无因子依赖，跳过
```

### 12.3 ConditionEvaluator — 递归条件评估器

```python
from typing import Callable, Dict, List, Optional


class ConditionEvaluator:
    """将规则树 JSON「编译」为可逐 bar 调用的评估函数。

    设计原则:
      · 编译阶段（__init__）执行一次，将 JSON 转换为嵌套的 Python lambda
      · 评估阶段（evaluate）对每根 bar 执行，O(1) 递归
      · factor_values 为 dict[str, np.array]，key 为 FactorSpec.result_key
    """

    def __init__(self, rule_tree: dict, factor_registry: dict = None):
        self._registry = factor_registry or {}
        self._root_evaluator = self._compile_node(rule_tree)

    # ── 对外 API ─────────────────────────────────────────

    def evaluate(self, factor_values: Dict[str, List[float]], bar_idx: int) -> bool:
        """评估当前 bar 是否满足整个条件树"""
        try:
            return self._root_evaluator(factor_values, bar_idx)
        except (IndexError, KeyError, TypeError):
            # 预热不足或因子值缺失 → 安全降级为 False
            return False

    def evaluate_all(self, factor_values: Dict[str, List[float]]) -> List[bool]:
        """对所有 bar 批量评估（向量化场景可选）"""
        if not factor_values:
            return []
        n = len(next(iter(factor_values.values())))
        return [self.evaluate(factor_values, i) for i in range(n)]

    # ── 编译逻辑（将 JSON 节点 → 函数）───────────────────

    def _compile_node(self, node: dict) -> Callable[[Dict, int], bool]:
        """根据节点类型分发到不同的编译函数"""

        # 情况 1: 逻辑组合节点 — AND / OR
        if "operator" in node and "conditions" in node:
            child_funcs = [self._compile_node(c) for c in node["conditions"]]
            op = node["operator"]
            if op == "AND":
                return lambda fv, idx: all(fn(fv, idx) for fn in child_funcs)
            elif op == "OR":
                return lambda fv, idx: any(fn(fv, idx) for fn in child_funcs)
            else:
                return lambda fv, idx: False  # 未知 operator

        # 情况 2: compare 节点
        if node.get("type") == "compare":
            left_fn = self._compile_expression(node["left"])
            right_fn = self._compile_expression(node["right"])
            comparator = node["comparator"]

            def compare(fv: Dict, idx: int) -> bool:
                lv = left_fn(fv, idx)
                rv = right_fn(fv, idx)
                if lv is None or rv is None:
                    return False

                if comparator == ">":  return lv > rv
                if comparator == "<":  return lv < rv
                if comparator == ">=": return lv >= rv
                if comparator == "<=": return lv <= rv
                if comparator == "==": return abs(lv - rv) < 0.001  # 浮点安全比较
                if comparator == "!=": return abs(lv - rv) >= 0.001
                if comparator in ("cross_up", "cross_down"):
                    # 穿越判断需要上一根 bar 的值
                    if idx == 0:
                        return False
                    prev_lv = left_fn(fv, idx - 1)
                    prev_rv = right_fn(fv, idx - 1)
                    if prev_lv is None or prev_rv is None:
                        return False
                    if comparator == "cross_up":
                        return prev_lv <= prev_rv and lv > rv
                    else:  # cross_down
                        return prev_lv >= prev_rv and lv < rv
                return False

            return compare

        # 无法识别的节点 → 恒 False
        return lambda fv, idx: False

    def _compile_expression(self, expr_node: dict) -> Callable[[Dict, int], Optional[float]]:
        """编译表达式节点 → 取值函数（返回当前 bar 的数值或 None）"""

        # 形态 A: 静态值
        if "value" in expr_node:
            v = float(expr_node["value"])
            return lambda fv, idx: v

        # 形态 B: 因子引用
        if "factor" in expr_node:
            spec = FactorSpec.from_node(expr_node)
            key = spec.result_key  # 如 "MA_5", "MACD_DIF", "close"

            def eval_factor(fv: Dict, idx: int) -> Optional[float]:
                arr = fv.get(key)
                if arr is None or idx >= len(arr):
                    return None
                val = arr[idx]
                if val is None or (isinstance(val, float) and val != val):  # NaN check
                    return None
                return float(val)

            return eval_factor

        # 形态 C: 算术表达式（op + left + right）
        if "op" in expr_node:
            left_fn = self._compile_expression(expr_node["left"])
            right_fn = self._compile_expression(expr_node["right"])
            op = expr_node["op"]

            def eval_arithmetic(fv: Dict, idx: int) -> Optional[float]:
                lv = left_fn(fv, idx)
                rv = right_fn(fv, idx)
                if lv is None or rv is None:
                    return None
                if op == "+": return lv + rv
                if op == "-": return lv - rv
                if op == "*": return lv * rv
                if op == "/": return lv / rv if rv != 0 else None
                return None

            return eval_arithmetic

        # 无法识别 → 返回 None
        return lambda fv, idx: None
```

> **关于 cross_up / cross_down 的关键说明**：
>
> 实现中**不需要单独维护 `prev_factors` 快照**。由于 `factor_values` 中保存了所有 bar 的完整因子序列，`left_fn(fv, idx-1)` 可以直接访问上一根 bar 的值。
>
> 对 `cross_up` 的语义检查：前端已限制操作符为 `cross_up/cross_down` 时左右表达式必须是因子引用（禁用算术表达式）。Python 端在 `_compile_node` 中也做一次兜底校验，如果左右表达式类型不是 `factor`，发出 Warning 并将其当作普通比较处理。

### 12.4 SellRules — 特殊卖出规则的统一封装

```python
from dataclasses import dataclass, field
from typing import Optional, Dict, List


@dataclass
class StopLossRule:
    percent: float  # 如 0.08 = 亏损 8%

    def should_sell(self, entry_price: float, current_low: float) -> bool:
        """用 low 判断：当前 bar 的最低价跌破阈值即止损"""
        if entry_price <= 0 or current_low <= 0:
            return False
        return current_low <= entry_price * (1 - self.percent)


@dataclass
class TakeProfitRule:
    percent: float  # 如 0.20 = 盈利 20%

    def should_sell(self, entry_price: float, current_high: float) -> bool:
        """用 high 判断：当前 bar 的最高价突破阈值即止盈"""
        if entry_price <= 0 or current_high <= 0:
            return False
        return current_high >= entry_price * (1 + self.percent)


@dataclass
class MaxHoldingDaysRule:
    days: int  # 持仓超过 N 个交易日强制平仓

    def should_sell(self, bars_since_entry: int) -> bool:
        """不判断价格，纯计数"""
        return bars_since_entry >= self.days


@dataclass
class SellRules:
    """集合所有卖出规则。任一规则触发即卖出（OR 语义）"""

    condition_evaluator: Optional[ConditionEvaluator] = None  # 普通条件卖出
    stop_loss: Optional[StopLossRule] = None
    take_profit: Optional[TakeProfitRule] = None
    max_holding_days: Optional[MaxHoldingDaysRule] = None
    min_holding_days: int = 1  # 全局约束：持仓未达此天数前所有卖出规则不生效

    def should_sell(self,
                    factor_values: Dict[str, List[float]],
                    bar_idx: int,
                    entry_price: float,
                    bars_since_entry: int,
                    current_high: float,
                    current_low: float,
                    current_close: float) -> bool:
        """综合判断当前 bar 是否需要卖出。

        参数来源（供调用方参考）:
          · factor_values: 批量计算的因子值 dict
          · bar_idx: 当前 bar 索引
          · entry_price: 买入成交价格（回测中记录的买入价）
          · bars_since_entry: 已持仓 bar 数
          · current_high/low/close: 当前 bar 的 OHLC 数据

        返回 True 表示应在当前 bar 以 close 卖出。
        """

        # 全局约束：持仓未达 min_holding_days 天数 → 禁止卖出
        if bars_since_entry < self.min_holding_days:
            return False

        # 1) 普通条件卖出（用 close 判断条件树）
        if self.condition_evaluator and self.condition_evaluator.evaluate(factor_values, bar_idx):
            return True

        # 2) 止损（用 low 判断）
        if self.stop_loss and self.stop_loss.should_sell(entry_price, current_low):
            return True

        # 3) 止盈（用 high 判断）
        if self.take_profit and self.take_profit.should_sell(entry_price, current_high):
            return True

        # 4) 最大持仓天数（不判断价格）
        if self.max_holding_days and self.max_holding_days.should_sell(bars_since_entry):
            return True

        return False


# ── SellRules 的构造函数（从 sell_rules JSON 解析）─────────────
def parse_sell_rules(sell_rules_json: dict,
                     factor_registry: dict = None,
                     position_sizing: dict = None) -> SellRules:
    """从策略 JSON 的 sell_rules 部分解析 SellRules 对象"""

    # 提取 min_holding_days（来自 position_sizing）
    min_holding_days = 1
    if position_sizing and "min_holding_days" in position_sizing:
        min_holding_days = int(position_sizing["min_holding_days"])

    # 解析 conditions 列表 → 区分普通条件和特殊规则
    condition_nodes = []
    stop_loss = None
    take_profit = None
    max_holding_days = None

    for condition in sell_rules_json.get("conditions", []):
        ctype = condition.get("type")
        if ctype == "stop_loss":
            stop_loss = StopLossRule(percent=float(condition["params"]["percent"]))
        elif ctype == "take_profit":
            take_profit = TakeProfitRule(percent=float(condition["params"]["percent"]))
        elif ctype == "max_holding_days":
            max_holding_days = MaxHoldingDaysRule(days=int(condition["params"]["days"]))
        else:
            # 普通 compare 节点或嵌套组 → 收集后整体编译为 ConditionEvaluator
            condition_nodes.append(condition)

    # 如果有普通条件卖出，将其包装为一个 AND/OR 节点
    condition_evaluator = None
    if condition_nodes:
        wrapper = {
            "operator": sell_rules_json.get("operator", "OR"),
            "conditions": condition_nodes
        }
        condition_evaluator = ConditionEvaluator(wrapper, factor_registry)

    return SellRules(
        condition_evaluator=condition_evaluator,
        stop_loss=stop_loss,
        take_profit=take_profit,
        max_holding_days=max_holding_days,
        min_holding_days=min_holding_days
    )
```

### 12.5 ParamExtractor — 参数路径提取（供参数优化）

```python
from dataclasses import dataclass
from typing import List, Dict, Optional


@dataclass
class ParamPath:
    path: str                    # "buy_rules.conditions[0].left.params.timeperiod"
    current_value: float         # 当前参数值
    param_type: str              # "INT" / "FLOAT"
    min_value: float             # 优化范围下界
    max_value: float             # 优化范围上界
    label: str                   # 前端可读标签，如"短期均线周期"
    factor_name: Optional[str] = None  # 关联的因子名（如果有）


class ParamExtractor:
    """递归遍历策略 JSON，提取所有可调整的数值参数。

    参数来源:
      1) 因子引用节点的 params 中所有数值字段（如 MA 的 timeperiod）
      2) 特殊卖出规则节点的 params 中所有数值字段（如 stop_loss 的 percent）
      3) 算术表达式中的静态值（可选，默认不提取以避免参数爆炸）
    """

    def __init__(self, factor_registry: dict = None):
        self._registry = factor_registry or {}

    def extract_all(self, strategy_json: dict) -> List[ParamPath]:
        results = []
        # 买入规则
        self._traverse(strategy_json.get("buy_rules", {}), "buy_rules", results)
        # 卖出规则（包括普通条件和特殊节点）
        self._traverse_sell(strategy_json.get("sell_rules", {}), "sell_rules", results)
        return results

    def _traverse(self, node: dict, path: str, results: List[ParamPath]) -> None:
        # 因子引用节点 → 提取 params 中的数值
        if "factor" in node and "params" in node:
            factor_key = node["factor"]
            registry_entry = self._registry.get(factor_key, {})
            registry_params = {p["name"]: p for p in registry_entry.get("params", [])}
            for pname, pvalue in node["params"].items():
                if isinstance(pvalue, (int, float)):
                    # 从注册表获取元数据
                    reg_param = registry_params.get(pname, {})
                    results.append(ParamPath(
                        path=f"{path}.params.{pname}",
                        current_value=float(pvalue),
                        param_type=reg_param.get("type", "INT" if isinstance(pvalue, int) else "FLOAT"),
                        min_value=reg_param.get("min", 2 if pname.lower().find("time") >= 0 else 0.01),
                        max_value=reg_param.get("max", 120 if pname.lower().find("time") >= 0 else 0.5),
                        label=f"{factor_key} 的 {reg_param.get('displayName', pname)}",
                        factor_name=factor_key
                    ))
            return

        # 逻辑组合节点 → 递归子条件
        if "operator" in node and "conditions" in node:
            for i, child in enumerate(node["conditions"]):
                self._traverse(child, f"{path}.conditions[{i}]", results)
            return

        # compare 节点 → 递归左右
        if node.get("type") == "compare":
            self._traverse(node["left"], f"{path}.left", results)
            self._traverse(node["right"], f"{path}.right", results)
            return

        # 算术表达式 → 递归左右（value 静态值默认不提取，避免参数爆炸）
        if "op" in node:
            self._traverse(node["left"], f"{path}.left", results)
            self._traverse(node["right"], f"{path}.right", results)
            return

    def _traverse_sell(self, node: dict, path: str, results: List[ParamPath]) -> None:
        """专门遍历 sell_rules.conditions，识别特殊类型节点"""
        for i, condition in enumerate(node.get("conditions", [])):
            cpath = f"{path}.conditions[{i}]"
            ctype = condition.get("type")
            if ctype in ("stop_loss", "take_profit"):
                pct = condition["params"]["percent"]
                label = "止损比例" if ctype == "stop_loss" else "止盈比例"
                results.append(ParamPath(
                    path=f"{cpath}.params.percent",
                    current_value=float(pct),
                    param_type="FLOAT",
                    min_value=0.02, max_value=0.30,
                    label=label
                ))
            elif ctype == "max_holding_days":
                days = condition["params"]["days"]
                results.append(ParamPath(
                    path=f"{cpath}.params.days",
                    current_value=float(days),
                    param_type="INT",
                    min_value=5, max_value=120,
                    label="最大持仓天数"
                ))
            else:
                # 普通条件节点 → 走通用遍历
                self._traverse(condition, cpath, results)


# ── 参数优化中的使用方式 ─────────────────────────────────────
```

### 12.6 RuleParser 的对外统一入口

```python
from dataclasses import dataclass
from typing import FrozenSet, Optional


@dataclass
class ParsedStrategy:
    """一次 parse 的完整产出，调用方按需取用各字段"""

    # 买入规则
    buy_evaluator: ConditionEvaluator
    buy_factor_specs: FrozenSet[FactorSpec]

    # 卖出规则
    sell_rules: SellRules
    sell_factor_specs: FrozenSet[FactorSpec]

    # 全量因子规格 = 买入用 + 卖出用（去重）
    @property
    def all_factor_specs(self) -> FrozenSet[FactorSpec]:
        return self.buy_factor_specs | self.sell_factor_specs

    # 参数路径列表
    param_paths: List[ParamPath]

    # 仓位管理配置
    max_single_position_pct: float
    max_total_positions: int
    min_holding_days: int


class RuleParser:
    """对外的统一解析器入口"""

    def __init__(self, factor_registry: dict = None):
        self._registry = factor_registry or {}
        self._spec_extractor = FactorSpecExtractor()
        self._param_extractor = ParamExtractor(self._registry)

    def parse(self, strategy_json: dict) -> ParsedStrategy:
        """完整解析一个策略"""

        # 1) 买入规则
        buy_tree = strategy_json.get("buy_rules", {"operator": "AND", "conditions": []})
        buy_evaluator = ConditionEvaluator(buy_tree, self._registry)
        buy_specs = self._spec_extractor.extract(buy_tree)

        # 2) 卖出规则（含特殊规则 + 普通条件）
        sell_tree = strategy_json.get("sell_rules", {"operator": "OR", "conditions": []})
        position_sizing = strategy_json.get("position_sizing", {})
        sell_rules = parse_sell_rules(sell_tree, self._registry, position_sizing)
        sell_specs = self._spec_extractor.extract(sell_tree)

        # 3) 参数路径提取
        param_paths = self._param_extractor.extract_all(strategy_json)

        # 4) 仓位管理
        return ParsedStrategy(
            buy_evaluator=buy_evaluator,
            buy_factor_specs=buy_specs,
            sell_rules=sell_rules,
            sell_factor_specs=sell_specs,
            param_paths=param_paths,
            max_single_position_pct=float(position_sizing.get("max_single_position_pct", 0.10)),
            max_total_positions=int(position_sizing.get("max_total_positions", 10)),
            min_holding_days=int(position_sizing.get("min_holding_days", 1)),
        )
```

## 13. 参数提取与参数优化接口（补充说明）

本节补充 §12.5 未覆盖的一个设计决策：**参数优化是回测中心的职责，本模块只提供 ParamPath 提取能力**。

```
数据流:
  用户在前端配置策略 → 策略 JSON 保存到 Java
  Java 保存时调用 RuleParser.extract_params() → 生成 ParamPath 列表
      ↓ 保存到 quant_strategy_param 表（或直接嵌入 JSON）
  回测中心页面加载策略 → 读取 ParamPath 列表，展示可调参数的滑杆
  用户设置参数范围 → 回测中心循环调用 Python RuleParser + 回测引擎
  输出最优参数组合 → 用户可选择应用该参数值回存为新版本
```

| 能力 | 本模块（策略管理）提供 | 回测中心负责 |

| 参数路径提取 | ✅ `ParamExtractor.extract_all()` | — |
| 参数范围推断 | ✅ 从因子注册表 params.min/max 推断 | — |
| 参数替换 | — | 在 JSON 中替换指定 path 的值 |
| 网格搜索 | — | 对每个参数的候选值跑回测 |
| 绩效统计 | — | Sharpe / 年化收益 / 最大回撤 |
| 应用最优参数 | — | 调用 Java 的「保存为新版本」接口 |
