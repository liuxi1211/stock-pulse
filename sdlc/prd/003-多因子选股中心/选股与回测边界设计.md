## 1. 背景与目标

### 1.1 背景

StockPulse 量化研究模块包含两个核心子模块：
- **多因子选股**：根据因子条件从股票池中筛选出符合要求的股票
- **策略回测**：在选定的股票上运行交易策略，评估策略绩效

两个模块共享同一套因子体系和条件表达式，但各自的使用场景、输入输出、技术实现都有显著差异。如果边界不清，会导致：
- 重复建设：两套条件解析逻辑，维护成本高
- 数据不一致：选股用的因子值和回测用的因子值对不上
- 扩展困难：新增一个因子要改两个地方
- 用户体验差：选股和回测的操作逻辑不统一

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| **职责清晰** | 每个模块做什么、不做什么，一目了然 |
| **内核共用** | 条件表达式引擎、因子计算逻辑只写一遍，两边都能用 |
| **数据一致** | 选股和回测使用的因子值计算方式一致，结果可对比 |
| **独立演进** | 两个模块可以独立开发、独立部署，互不影响 |
| **无缝配合** | 从选股到回测是一条顺畅的链路，用户体验连贯 |

## 2. 核心概念定义

### 2.1 多因子选股（Stock Screening）

**定义**：在某个时间点，从一个股票池中，根据一组因子条件，筛选出符合条件的股票，并按某种规则排序，返回股票列表。

**本质**：**截面判断**——在同一个时间截面上，对多只股票做横向比较。

```
时间点：2024-06-20
股票池：全市场 5000+ 只
条件：PE < 20 AND ROE > 15% AND MA5 > MA20
排序：按总市值升序
输出：满足条件的前 30 只股票代码列表
```

### 2.2 策略回测（Strategy Backtesting）

**定义**：在一段时间内，对一只或多只股票，按照既定的买卖规则进行模拟交易，计算收益曲线和绩效指标。

**本质**：**时序交易**——在时间序列上，对每只股票做纵向判断和交易。

```
时间范围：2023-01-01 ~ 2024-06-01
标的：30 只股票
策略：MA5 上穿 MA20 买入，下穿卖出，ATR 止损
调仓：每月调一次仓
输出：权益曲线、交易记录、绩效指标（收益率、夏普率、最大回撤等）
```

### 2.3 条件表达式（Condition Expression）

**定义**：描述「满足什么条件」的标准化 JSON 结构，由因子引用、静态值、算术运算、逻辑组合构成。

**本质**：**可复用的判断逻辑**——选股和回测都用同一套表达式描述条件。

```json
{
  "operator": "AND",
  "conditions": [
    {
      "type": "compare",
      "left": { "factor": "PE_TTM" },
      "comparator": "<",
      "right": { "value": 20 }
    },
    {
      "type": "compare",
      "left": { "factor": "MA", "params": { "timeperiod": 5 } },
      "comparator": "cross_up",
      "right": { "factor": "MA", "params": { "timeperiod": 20 } }
    }
  ]
}
```

### 3.2 多因子选股模块的职责

| 职责 | 说明 | 不做什么 |
|------|------|---------|
| **股票池管理** | 维护股票池（全市场、沪深300、自选股等） | 不管理持仓 |
| **条件选股** | 根据条件表达式筛选股票 | 不做买卖决策 |
| **排序打分** | 按因子排序或多因子综合打分 | 不计算收益 |
| **过滤规则** | ST 过滤、停牌过滤、涨跌停过滤、行业过滤 | 不止损止盈 |
| **选股结果** | 返回股票列表 + 因子值 + 排名 | 不输出权益曲线 |
| **回测入口** | 选股结果可一键发起回测 | 不执行回测 |

### 3.3 策略回测模块的职责

| 职责 | 说明 | 不做什么 |
|------|------|---------|
| **信号生成** | 根据买卖条件生成买入/卖出信号 | 不选股（用别人选好的股票） |
| **仓位管理** | 买多少、怎么买（等权重、市值加权等） | 不做股票池管理 |
| **止损止盈** | 持仓期间的退出规则 | 不做行业/板块过滤 |
| **调仓管理** | 定期重新选股换仓 | 不保存选股结果 |
| **绩效计算** | 收益率、夏普率、最大回撤、胜率等 | 不输出股票列表 |
| **回测报告** | 权益曲线、交易明细、归因分析等 | 不做选股排序 |
| **参数优化** | 网格搜索、Walk-forward 分析 | 不做因子计算（用因子服务） |

### 3.4 共用层的职责

| 模块 | 职责 | 两边怎么用 |
|------|------|-----------|
| **因子库定义** | 因子元数据（参数、输入输出、分类等） | 选股和回测都读取同一份因子定义 |
| **条件表达式引擎** | 解析条件树、计算表达式、执行比较 | 选股用来筛选股票，回测用来生成信号 |
| **因子计算服务** | 统一计算因子值（技术面走 akquant，基本面读数据库） | 选股时批量计算，回测时实时计算 |

## 4. 交集部分设计

### 4.1 交集总览

| 交集点 | 说明 | 归属 | 怎么共用 |
|--------|------|------|---------|
| 因子定义 | 有哪些因子、参数是什么、怎么算 | 共用层 | 同一份 JSON 定义文件 |
| 条件表达式 Schema | 条件怎么描述、表达式怎么写 | 共用层 | 同一份 JSON Schema |
| 条件解析引擎 | 怎么把条件解析成可执行的判断 | 共用层 | 同一套代码（**仅 Python 实现**，Java 选股经 HTTP 调用 engine） |
| 因子计算 | 因子值怎么算出来 | 共用层 | 同一个计算服务（Python + akquant） |
| 股票列表 | 选股的输出 = 回测的输入 | 数据交互 | 通过接口/数据库传递 |
| 调仓日选股 | 回测调仓时需要重新选股 | 回测内本地选股 | engine 用 watcher 预传的候选池数据本地跑条件引擎（不回调 watcher） |

### 4.2 因子定义：完全共用

**设计**：因子库只有一份定义（`标准因子库-v2.json`），Java 和 Python 都读取这份定义。

```
因子库定义（JSON）
    │
    ├── Java 端读取 → 选股模块用 → 前端条件编辑器用
    │
    └── Python 端读取 → 回测模块用 → 因子计算服务用
```

**确保一致的手段**：
- 因子定义是唯一真相源（Single Source of Truth）
- 新增/修改因子只改这一份文件
- 两边都有校验逻辑，确保因子定义合法

### 4.3 条件表达式：完全共用

**设计**：条件表达式的 JSON 结构完全一致，选股的「入选条件」和回测的「买入条件」用同一套 Schema。

```json
// 选股条件
{
  "operator": "AND",
  "conditions": [
    { "type": "compare", "left": {"factor":"PE_TTM"}, "comparator":"<", "right": {"value":20} },
    { "type": "compare", "left": {"factor":"ROE_TTM"}, "comparator":">", "right": {"value":15} }
  ]
}

// 回测买入条件
{
  "operator": "AND",
  "conditions": [
    { "type": "compare", "left": {"factor":"MA","params":{"timeperiod":5}}, "comparator":"cross_up", "right": {"factor":"MA","params":{"timeperiod":20}} },
    { "type": "compare", "left": {"factor":"RSI","params":{"timeperiod":14}}, "comparator":">", "right": {"value":50} }
  ]
}
```

两者结构完全一样，只是里面的因子不同。

### 4.4 条件解析引擎：单一 Python 实现

> ⚠️ 与硬约束对齐（CLAUDE.md）：engine（Python）不触库、交互单向（watcher → engine）。条件引擎**只在 Python 写一遍**，不维护 Java 版——既避免 §1.1 说的"两套条件解析逻辑"重复建设，又因为条件求值天然依赖因子计算（akquant.talib），Python 是唯一合理落点。

**两边怎么用同一套**：
- **选股**（截面）：watcher 把候选池数据 + 条件树经 HTTP 传给 engine → engine 跑条件引擎 → 返回命中列表（单向，engine 不回调 watcher）
- **回测**（时序）：engine 在 `on_bar` / `on_daily_rebalance` 内直接调条件引擎（数据已由 watcher 预传）

**一致性保证**：同一份代码、同一套测试用例（JSON 格式），覆盖基本运算、边界值、NaN 处理、穿越判断。

```
条件表达式（JSON）
    │
    └── Python ConditionEngine（engine 内唯一实现）
            ├── 选股：watcher 经 HTTP 调用
            └── 回测：on_bar / on_daily_rebalance 本地调用
```

### 4.5 因子计算：统一走 Python + akquant

**设计原则**：**因子计算只写一遍，两边都调用**。

**技术方案**：
- 技术面因子：统一用 `akquant.talib` 计算（Python 端）
- 基本面因子：统一从数据库读取（Java 端存库，Python 端读 extra）
- 选股需要批量计算时，Java 调用 Python 的因子计算 HTTP 接口
- 回测需要实时计算时，Python 直接用 akquant.talib

```
                    ┌───────────────────────┐
                    │  因子计算服务（Python）  │
                    │  FactorCalculator     │
                    │  (akquant.talib)      │
                    └───────────┬───────────┘
                                │
              ┌─────────────────┴─────────────────┐
              ▼                                   ▼
    ┌───────────────────┐              ┌───────────────────┐
    │  Java 选股服务     │              │  Python 回测服务   │
    │  批量计算因子值     │              │  on_bar 实时计算   │
    │  存 stock_indicato│              │  直接调 talib      │
    │  r_daily 表       │              │                   │
    └───────────────────┘              └───────────────────┘
```

**预计算 vs 实时计算**：

| 场景 | 计算方式 | 说明 |
|------|---------|------|
| 选股（全市场批量） | **预计算 + 数据库** | 每日收盘后批量算完存库，选股时直接查 |
| 回测（少量股票时序） | **实时计算** | on_bar 里直接调 akquant.talib，不需要预计算 |
| 回测里的调仓日选股 | **实时计算 / 预计算** | 股票少就实时算，股票多就查预计算表 |

**一致性保证**：
- 两种方式用的是同一个 `akquant.talib` 函数，计算结果一致
- 定期做一致性校验：随机抽几只股票，对比预计算值和实时计算值

### 4.6 股票列表：选股的输出 = 回测的输入

**交互方式**：

| 场景 | 交互方式 | 说明 |
|------|---------|------|
| 用户选股后一键回测 | 选股结果通过前端传递给回测页面 | 最简单，用户手动触发 |
| 回测时定期调仓选股 | 回测服务调用选股服务接口 | 程序化，需要 API |
| 选股结果保存后回测 | 选股结果存数据库，回测时读取 | 可追溯，适合研究 |

**数据格式**：

```json
// 选股结果（传递给回测）
{
  "strategy_id": "value_001",
  "screen_date": "2024-06-20",
  "universe": "all_a_shares",
  "conditions": { /* 条件表达式 */ },
  "ranking": { "factor": "TOTAL_MV", "order": "asc" },
  "stocks": [
    { "symbol": "000001.SZ", "rank": 1, "factor_values": { "pe_ttm": 8.5, "roe_ttm": 18.2, ... } },
    { "symbol": "600000.SH", "rank": 2, "factor_values": { ... } },
    ...
  ]
}
```

## 5. 数据流向全链路

### 5.1 场景一：选股 → 一键回测（最简单的链路）

```
用户在选股页面配置条件
    │
    ▼
Java 选股服务
    │
    ├── 1. 从数据库读因子值（预计算好的）
    ├── 2. 用条件引擎筛选
    ├── 3. 排序，取前 N 只
    └── 4. 返回股票列表
    │
    ▼
用户点击「回测这个组合」
    │
    ▼
跳转到回测页面，带上股票列表
    │
    ▼
用户配置买卖规则、调仓频率、回测参数
    │
    ▼
Python 回测服务
    │
    ├── 1. 接收 watcher 经 HTTP 传入的行情 + 基本面数据
    ├── 2. 组装成 akquant 格式（DataFrame + extra）
    ├── 3. 构建动态策略类
    ├── 4. aq.run_backtest() 执行回测
    └── 5. 返回回测报告
    │
    ▼
前端展示回测结果
```

### 5.2 场景二：多因子策略回测（含定期调仓）

```
用户配置多因子策略
    ├── 选股条件（哪些股票能买）
    ├── 调仓频率（每月/每周调一次）
    ├── 持仓数量（比如 30 只）
    ├── 排序规则（按什么选股）
    ├── 仓位管理（等权重 / 市值加权）
    └── 止损止盈规则
    │
    ▼
Python 回测服务
    │
    ├── on_start()：初始化，加载数据
    │
    ├── on_daily_rebalance()：调仓日执行
    │   │   ⚠️ engine 不回调 watcher（硬约束）：调仓选股须在 engine 本地完成
    │   │
    │   ├── 用 watcher 预传的候选池数据（全段 OHLCV + 基本面）
    │   │   本地跑条件引擎筛选 + ranking 取前 N 只
    │   │
    │   └── 执行调仓：order_target_weights()
    │
    ├── on_bar()：每个交易日执行
    │   ├── 检查止损止盈
    │   └── 其他日内逻辑
    │
    └── on_stop()：计算绩效，生成报告
    │
    ▼
返回回测结果
```

## 6. Schema 复用设计

### 6.1 分层结构

为了实现「一套 Schema，两头复用」，我们把策略配置分成三层：

```
┌──────────────────────────────────────────────┐
│  策略配置（Strategy Config）                  │
│  ┌────────────────────────────────────────┐  │
│  │  选股配置（Screen Config）             │  │
│  │  - 股票池                              │  │
│  │  - 选股条件（共用条件表达式）          │  │← 选股模块只需要这一层
│  │  - 排序规则                            │  │
│  │  - 过滤规则（ST/涨跌停/行业等）        │  │
│  └────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────┐  │
│  │  交易配置（Trading Config）            │  │
│  │  - 买入信号条件（共用条件表达式）      │  │
│  │  - 卖出信号条件（共用条件表达式）      │  │
│  │  - 仓位管理                            │  │
│  │  - 止损止盈                            │  │
│  └────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────┐  │
│  │  回测配置（Backtest Config）          │  │← 回测模块需要全部三层
│  │  - 调仓频率                            │  │
│  │  - 初始资金                            │  │
│  │  - 手续费参数                          │  │
│  │  - 时间范围                            │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

### 6.2 Schema 字段定义（见统一 Schema）

> ⚠️ 策略配置的完整字段树、条件表达式模型、akquant 映射的**唯一权威**是
> [`004-策略管理/统一策略配置Schema.md`](../004-策略管理/统一策略配置Schema.md)。
> 本节早期的「完整 JSON 结构」示例**已删除**（存在 4 处致命缺陷，见下），不再作为字段参考。

本节只保留**三段分区的职责边界**说明（选股/回测模块各读哪层），字段细节请查统一 Schema：

| 分区 | 选股模块 | 回测模块 |
|---|---|---|
| `screen_config` | ✅ 全量读取（`POST /api/screener/run` 只接收这层） | ✅ 调仓日读取（重新选股换仓） |
| `trading_config` | ❌ 不感知 | ✅ 全量（`signals` 或 `rebalance` 至少一个） |
| `backtest_config` | ❌ 不感知 | ✅ 全量（映射 `run_backtest` 参数） |

**已删除的旧 JSON 的 4 处致命缺陷**（统一 Schema 已修正）：

1. `trading_config` 里**没有买卖信号条件树**（`signals`），却号称与选股条件共用同一套 Schema → 统一 Schema §3.3.1 补回。
2. 表达式节点只有 `{factor}/{value}`，**丢了 `op` 算术与 `output_index`**（无法表达 MACD hist / 布林轨） → 统一 Schema §4.3 恢复 4 形态。
3. `position_sizing.method="equal_weight"`（组合权重语义）与下单方法名撞义 → 组合权重移至 `rebalance.weight_mode`，见统一 Schema §3.3.2 / §3.3.4。
4. `screen_config.conditions` 里误用 `MA5 cross_up MA20` —— 选股是截面操作，`cross_up` 无定义 → 统一 Schema §7.2 硬性禁止。

> 多因子价值策略的**修正后完整示例**见统一 Schema §5.2。

### 6.2-bis 选股条件表达式的特殊约束

选股（`screen_config.conditions`）与回测信号（`trading_config.signals`）**共用同一套条件表达式模型**，但选股是**截面操作**（某日全市场快照），有时序依赖的节点在此无定义。统一 Schema §7.2 规定：

| 禁止在 `screen_config.conditions` 出现 | 原因 |
|---|---|
| `comparator: cross_up` / `cross_down` | 穿越判断需要「上一根 bar」的因子值，截面场景无时序上下文；若实现层偷偷用个股历史 bar 计算，选股结果会依赖隐藏状态，对停牌/新股产生诡异结果。趋势过滤应移入 `trading_config.signals`（调仓日按标的逐个评估）。 |
| `ref` 节点（如 `entry_price`） | ref 引用持仓状态，选股时无持仓上下文。 |

> 截面场景允许：`>` `<` `>=` `<=` `==` `!=` + `{value}`/`{factor}`/`{op}` 节点（基本面因子如 `PE_TTM`/`ROE_TTM`/`TOTAL_MV` 在此合法）。

### 6.3 各模块使用哪部分

| 模块 | 使用哪些部分 | 忽略哪些部分 |
|------|-------------|-------------|
| **选股模块** | `screen_config`（全部） | `trading_config` + `backtest_config` |
| **回测模块** | `screen_config`（调仓日用）+ `trading_config` + `backtest_config` | 没有，全部都用 |
| **条件编辑器** | 只关心 `conditions` 里的表达式结构 | 其他都不关心 |

## 7. 典型场景说明

### 7.1 场景 A：单标的策略回测（最简单）

**特点**：人工指定一只股票，不需要选股模块参与。

```
股票来源：用户手动输入（如 510300.SH 沪深300ETF）
选股模块：不参与
回测模块：全流程
适用场景：新手入门、单只个股研究、ETF 策略
```

**配置中 screen_config 的处理**：
```json
{
  "screen_config": {
    "universe": "manual",
    "stocks": ["510300.SH"]
  }
}
```

### 7.2 场景 B：固定股票池 + 技术指标策略

**特点**：股票池固定（如沪深300成分股），用技术指标做买卖判断，不调仓。

```
股票来源：用户选择股票池（沪深300），或者直接给列表
选股模块：只提供股票池，不做筛选（或者简单筛选）
回测模块：每只股票独立跑策略，或者组合跑
调仓：不调仓（买入后持有，直到卖出信号）
适用场景：行业轮动、指数增强
```

### 7.3 场景 C：多因子选股 + 定期调仓（最复杂）

**特点**：定期重新选股换仓，选股和回测深度配合。

```
股票来源：选股模块动态选出
选股模块：每个调仓日都要选一次
回测模块：调仓日调用选股，执行换仓
调仓：每月/每周调一次
适用场景：多因子策略、量化选股策略
```

## 8. 技术实现建议

### 8.1 实现顺序建议

| 阶段 | 做什么 | 说明 |
|------|--------|------|
| **P0** | 因子库定义 + 因子计算服务 | 基础中的基础，先把因子算对 |
| **P0** | 条件表达式引擎（仅 Python 版） | 核心共用逻辑，只实现一遍；Java 选股经 HTTP 调用 |
| **P1** | 选股模块（条件选股 + 排序） | 先把选股跑通 |
| **P1** | 回测模块（单标的 + 固定股票池） | 再把回测跑通（不含调仓选股） |
| **P2** | 回测模块（调仓日重新选股） | 选股和回测联动 |
| **P2** | 参数优化 + Walk-forward | 进阶功能 |

### 8.2 关键技术点

#### 8.2.1 条件引擎只写一遍（Python）

**建议**：
1. 只在 engine（Python）实现一遍——紧邻 akquant.talib，因子求值与条件判断一体，验证方便
2. Java 选股不另写引擎：watcher 经 HTTP 把候选池 + 条件树传给 engine 跑
3. 一套测试用例（JSON 格式）覆盖：基本运算、边界值、NaN 处理、穿越判断
4. 不存在"两套实现对齐"的维护负担（这正是 §1.1 想避免的重复建设）

#### 8.2.2 因子计算一致性

**建议**：
1. 技术面因子的预计算，统一走 Python + akquant.talib，Java 只负责存库
2. 不要用 Java 自己实现技术指标（容易算不对，维护成本高）
3. 定期做一致性校验（CI 里加一个任务，每天抽几只股票对比）

#### 8.2.3 调仓选股的性能

**问题**：全市场 5000 只股票，每个调仓日都重新算因子+选股，慢不慢？

**方案**：
- 月频调仓：慢一点没关系，反正一个月才一次
- 日频调仓：需要预计算（每天收盘后算好存库）
- 第一版先用实时计算（简单，先跑通）
- 性能不够了再上预计算
