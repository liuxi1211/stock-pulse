## 1. 产品定位

把量化投资拆成**三个可独立操作的闭环**，每个闭环都是一条"想法 → 验证 → 复用"的链路：

| 闭环 | 入口 | 产出 |
|---|---|---|
| 因子库 | 指标/因子定义 | 可复用的计算单元，前端可配置参数 |
| 策略管理 | 组合因子构建策略 | 规则树 JSON，版本化管理 |
| 回测中心 | 策略 + 历史数据 | 绩效报告 / 参数优化 / Walk-forward 分析 |
| 选股模块 | 策略规则 + 预计算指标 | 符合条件的股票列表 |

核心价值用一句话总结：**让技术指标可配置，让策略规则可回测，让选股条件可复用。**

## 3. 基础设施约定（跨模块共享）

### 3.1 Java ↔ Python 通信协议

#### 3.1.1 服务地址

| 环境 | 地址 |
|---|---|
| 开发 / 默认 | http://127.0.0.1:8000 |
| 生产可配置 | 通过 application.properties 配置 `python.compute.url` |

#### 3.1.2 统一请求格式

所有 POST 请求遵循以下外层结构：

```json
{
  "task_id": "uuid-generated-by-java",
  "timestamp": 1780000000,
  "payload": {
    "_comment": "具体参数由各模块定义"
  }
}
```

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `task_id` | 是 | Java 生成的任务唯一标识，用于幂等/追踪 |
| `timestamp` | 否 | 请求时间戳（秒），用于日志追溯 |
| `payload` | 是 | 具体业务参数，由各模块定义 |

#### 3.1.3 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "task_id": "uuid-same-as-request",
    "compute_ms": 125,
    "_comment": "具体业务数据由各模块定义"
  }
}
```

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `code` | 是 | 0 = 成功，非 0 = 错误 |
| `message` | 是 | 成功 = "success"，失败 = 错误描述 |
| `data.task_id` | 是 | 回传请求中的 task_id，便于追踪 |
| `data.compute_ms` | 否 | 本次计算耗时（毫秒），用于性能监控 |
| `data.*` | 子模块定义 | 具体业务结果 |

> **非 0 code 的统一约定**：code = 1xx 为参数错误，2xx 为计算引擎错误，3xx 为数据错误。具体子码在各模块中定义。

#### 3.1.4 本模块涉及的接口清单

| 接口 | 说明 |
|---|---|
| `POST /api/compute/backtest` | 执行一次回测 |
| `POST /api/compute/grid-search` | 执行参数网格搜索 |
| `POST /api/compute/walk-forward` | 执行 Walk-forward 分析 |
| `GET /api/compute/health` | 健康检查 |

## 4. 跨模块关键决策（开发前必须拍板）

以下决策点影响多个模块的表结构和代码逻辑，在 Phase 1 开始前必须确认：

| 决策项 | 结论 | 影响模块 |
|--------|------|----------|
| 止损止盈的判断方式 | **止损用 `low`（日内最低），止盈用 `high`（日内最高），条件卖出用 `close`（收盘），`max_holding_days` 不判断价格直接平仓** | 策略管理、回测中心 |
| `rebalance_frequency_days` 的归属 | **从策略 JSON 的 `position_sizing` 中移除，仅保留在回测配置的 `rebalance` 中**（这是回测参数，不是策略的固定属性） | 策略管理、回测中心 |
| `quant_strategy_saved` 表是否保留 | **移除**。用 `quant_strategy` 表的 DRAFT 状态策略统一承载选股草稿场景 | 策略管理 |
| 因子结果 key 的命名规范 | **统一规则**：单输出因子 = `factorKey_paramValue`（如 `MA_5`、`RSI_14`）；多输出因子 = `factorKey_outputLabel`（如 `MACD_DIF`、`MACD_HIST`）。详见因子库 PRD | 因子库、策略管理、选股 |
| INDEX_300 / INDEX_500 回测是否放行 | **放行**。300 只 × 750 交易日 ≈ 1.35 MB，HTTP 可承载 | 回测中心 |

## 7. 需求概述

回测中心回答三个层次的问题：

| 层次 | 回答的问题 | 对应功能 |
|---|---|---|
| L1 | 这个策略在最近 N 年历史上表现如何？ | 单次回测 → 绩效报告 |
| L2 | 策略参数怎么选最好？ | 参数优化（网格搜索） |
| L3 | 这组"最优参数"会不会是过拟合出来的？ | Walk-forward 分析（滚动验证） |

三个层次按建议的使用顺序排列：先单次回测"感觉一下"，再参数优化"找一组好参数"，最后 Walk-forward"确认不过拟合"。

## 8. 回测配置参数

### 8.1 单次回测配置

```json
{
  "strategy_id": 42,
  "strategy_version": 3,

  "universe": {
    "type": "CUSTOM",
    "codes": ["000001.SZ", "000002.SZ", "600519.SH"]
  },

  "date_range": {
    "start": "2024-01-01",
    "end":   "2026-12-31"
  },

  "portfolio": {
    "initial_cash": 1000000,
    "commission_pct": 0.0003,
    "slippage_pct": 0.001
  },

  "rebalance": {
    "frequency_days": 5,
    "max_positions": 10,
    "max_single_position_pct": 0.10
  },

  "benchmark": "INDEX_000300"
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `strategy_id` | int | 是 | 策略 ID |
| `strategy_version` | int | 否 | 版本号；不传则用最新 ACTIVE 版本 |
| `universe.type` | string | 是 | `ALL` / `INDEX_300` / `INDEX_500` / `CUSTOM` |
| `universe.codes` | array | 条件 | `type=CUSTOM` 时必填 |
| `date_range.start` / `end` | string | 是 | ISO date，闭区间 |
| `portfolio.initial_cash` | number | 是 | 初始资金（人民币） |
| `portfolio.commission_pct` | number | 否 | 佣金率，默认 0.0003（万三） |
| `portfolio.slippage_pct` | number | 否 | 滑点，默认 0.001（0.1%） |
| `rebalance.frequency_days` | int | 否 | 调仓频率（交易日），默认 5 |
| `rebalance.max_positions` | int | 否 | 最大持仓数，默认读取策略配置；这里传入可覆盖 |
| `rebalance.max_single_position_pct` | number | 否 | 同上 |
| `benchmark` | string | 否 | `INDEX_000300`（沪深300）或省略；用于计算相对收益 |

### 8.2 参数优化配置

在单次回测基础上增加 `param_grid` 字段：

```json
{
  /* ... 单次回测所有字段 ... */

  "param_grid": [
    {
      "path": "buy_rules.conditions[0].left.params.timeperiod",
      "values": [3, 5, 8, 10]
    },
    {
      "path": "buy_rules.conditions[0].right.params.timeperiod",
      "values": [15, 20, 30, 60]
    },
    {
      "path": "sell_rules.conditions[1].params.percent",
      "values": [0.05, 0.08, 0.12]
    }
  ],

  "optimization": {
    "metric": "sharpe_ratio",
    "max_jobs": 4
  }
}
```

| 字段 | 说明 |
|---|---|
| `param_grid` | 各参数的候选值。笛卡尔积产生总组合数 |
| `optimization.metric` | 评估指标：`sharpe_ratio` / `total_return_pct` / `win_rate` / `calmar_ratio` |
| `optimization.max_jobs` | 并行线程数（Python 端实际串行执行以避免 GIL 问题，此参数仅为预留） |

> **组合数建议**：不要超过 1000 组。超过时提示用户减少参数或缩小范围。

### 8.3 Walk-forward 配置

```json
{
  /* ... 单次回测基础字段 ... */

  "walk_forward": {
    "in_sample_days": 252,
    "out_of_sample_days": 63,
    "step_days": 63,
    "optimization_metric": "sharpe_ratio",
    "param_grid": [ /* 同参数优化 */ ]
  }
}
```

| 字段 | 说明 |
|---|---|
| `in_sample_days` | 训练窗口大小（交易日） |
| `out_of_sample_days` | 验证窗口大小 |
| `step_days` | 滚动步长 |
| 说明 | Walk-forward 的流程：每次用 in_sample 窗口内的数据做参数优化，选最优参数在紧接着的 out_of_sample 窗口回测；然后两个窗口整体向前滚动 step_days |

> **防过拟合的核心**：Walk-forward 的每一段验证期都**不参与**该段的参数选择。最终把所有验证期的结果拼接起来，就是"类样本外"表现。

## 9. Java 数据库设计

### 9.1 quant_backtest — 回测记录

```sql
CREATE TABLE quant_backtest (
    id               INTEGER PRIMARY KEY AUTOINCREMENT,
    strategy_id      INTEGER NOT NULL,
    strategy_version INTEGER NOT NULL,
    task_id          TEXT NOT NULL UNIQUE,             -- 对应 Python 计算的 task_id
    mode             TEXT NOT NULL DEFAULT 'SINGLE',   -- SINGLE / GRID_SEARCH / WALK_FORWARD

    start_date       DATE NOT NULL,
    end_date         DATE NOT NULL,
    initial_cash     REAL NOT NULL,

    universe_type    TEXT NOT NULL,
    universe_codes   TEXT,                             -- JSON: CUSTOM 时的代码列表

    param_grid       TEXT,                             -- JSON: GRID_SEARCH / WF 时的参数网格
    param_applied    TEXT,                             -- JSON: 实际使用的参数（单回测时为策略默认值）

    status           TEXT NOT NULL DEFAULT 'RUNNING',  -- RUNNING / SUCCESS / FAILED
    error_message    TEXT,

    -- 指标（单回测时填充；GRID_SEARCH 填最优组的指标）
    total_return_pct  REAL,
    sharpe_ratio      REAL,
    max_drawdown_pct  REAL,
    win_rate          REAL,
    trades_count      INTEGER,

    created_at       TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    completed_at     TIMESTAMP,
    FOREIGN KEY (strategy_id) REFERENCES quant_strategy(id)
);

CREATE INDEX idx_bt_strategy ON quant_backtest(strategy_id, strategy_version);
CREATE INDEX idx_bt_mode     ON quant_backtest(mode);
CREATE INDEX idx_bt_status   ON quant_backtest(status);
```

### 9.2 quant_backtest_report — 回测详细数据（单次回测）

```sql
CREATE TABLE quant_backtest_report (
    backtest_id     INTEGER PRIMARY KEY,

    equity_curve_json  TEXT NOT NULL,  -- JSON: [{date, value}, ...]
    trades_json        TEXT NOT NULL,  -- JSON: [{date, code, action, price, qty, pnl, reason}, ...]
    positions_json     TEXT,           -- JSON: [{date, code, shares, cost, value, pnl_pct}]
    metrics_json       TEXT,           -- JSON: 全量指标（除已在主表的核心指标外）
    benchmark_json     TEXT,           -- JSON: [{date, value}, ...] 基准净值曲线（可选）

    FOREIGN KEY (backtest_id) REFERENCES quant_backtest(id) ON DELETE CASCADE
);
```

### 9.3 quant_backtest_grid_result — 参数优化结果

```sql
CREATE TABLE quant_backtest_grid_result (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    backtest_id    INTEGER NOT NULL,

    param_values   TEXT NOT NULL,     -- JSON: {path: value}
    metric_value   REAL NOT NULL,     -- 该组参数的目标指标值

    total_return_pct REAL,
    sharpe_ratio     REAL,
    max_drawdown_pct REAL,
    trades_count     INTEGER,

    FOREIGN KEY (backtest_id) REFERENCES quant_backtest(id) ON DELETE CASCADE
);

CREATE INDEX idx_grid_bt     ON quant_backtest_grid_result(backtest_id);
CREATE INDEX idx_grid_metric ON quant_backtest_grid_result(metric_value DESC);
```

### 9.4 quant_backtest_wf_result — Walk-forward 分段结果

```sql
CREATE TABLE quant_backtest_wf_result (
    id               INTEGER PRIMARY KEY AUTOINCREMENT,
    backtest_id      INTEGER NOT NULL,

    segment_index    INTEGER NOT NULL,
    in_sample_start  DATE NOT NULL,
    in_sample_end    DATE NOT NULL,
    out_sample_start DATE NOT NULL,
    out_sample_end   DATE NOT NULL,

    best_params      TEXT,            -- JSON: 该段训练期选的最优参数
    in_sample_metrics  TEXT,          -- JSON: 训练期指标
    out_sample_metrics TEXT,          -- JSON: 验证期指标

    FOREIGN KEY (backtest_id) REFERENCES quant_backtest(id) ON DELETE CASCADE
);

CREATE INDEX idx_wf_bt ON quant_backtest_wf_result(backtest_id, segment_index);
```

## 10. Python 回测执行接口

### 10.1 POST /api/compute/backtest

**请求体**（遵循统一格式）：

```json
{
  "task_id": "uuid",
  "payload": {
    "strategy_json": { /* 完整的策略 JSON */ },
    "stocks_data": {
      "000001.SZ": [ {date, open, high, low, close, volume}, ... ],
      "600519.SH": [ ... ]
    },
    "portfolio": { "initial_cash": 1000000, "commission_pct": 0.0003, "slippage_pct": 0.001 },
    "rebalance": { "frequency_days": 5, "max_positions": 10, "max_single_position_pct": 0.10 },
    "benchmark_data": [ {date, value}, ... ]
  }
}
```

**关键设计说明**：

- `stocks_data` 是 Java 从 daily_quote 表查询后传入的全量历史数据。Python 不直连 DB。
- `strategy_json` 包含完整的 `buy_rules` / `sell_rules` / `position_sizing`。
- Python 端需要将规则树 JSON → akquant Strategy 动态类。

**特殊卖出节点的解析与执行（核心补充）**：

`sell_rules.conditions` 中可能混用 compare 节点和特殊节点（`stop_loss` / `take_profit` / `max_holding_days`）。Python 端 RuleParser 的执行逻辑为：

| 节点类型 | 判断方式 | 成交价格 | 前置约束 |
|----------|---------|---------|---------|
| `stop_loss` | 当日 `low <= entry_price × (1 − percent)` | 当日 `close` | ① `position_sizing.min_holding_days` 未达时跳过；② `entry_price` 是该仓位的加权平均买入价 |
| `take_profit` | 当日 `high >= entry_price × (1 + percent)` | 当日 `close` | 同上 |
| `max_holding_days` | 持仓 bar 计数 >= `days` | 当日 `close` | 无视 `min_holding_days`（最长持仓天数是硬约束） |
| `compare`（普通条件卖出） | 用当日 `close` 计算因子值，递归评估条件树 | 当日 `close` | `min_holding_days` 未达时跳过 |

**优先级**：同一根 bar 上多个卖出条件同时满足时，按 `stop_loss → take_profit → max_holding_days → compare` 顺序判断，第一个触发的即执行卖出，其余不再评估。

**`rebalance.frequency_days`（调仓频率）**：这是回测层的参数（不是策略 JSON 的一部分）。含义是「每隔 N 个交易日重新评估一次买入机会」。实际中，每根 bar 都会评估卖出条件，但买入只在调仓日重新扫描候选池，避免过度交易。

**`min_holding_days` 的实现**：为每个持仓记录一个 `holding_bars` 计数器，每过一根 bar +1。卖出评估时，若 `holding_bars < min_holding_days`，跳过普通条件卖出、止损和止盈（`max_holding_days` 例外——这是最长持仓的硬约束，不受最少天数限制）。

## 11. Java 端回测调度流程

```
用户在前端点击「开始回测」
    ↓
Java 1. 从 quant_strategy 表读取策略 JSON
     2. 读取 daily_quote 中指定股票 + 日期范围的数据
     3. 写入 quant_backtest 表一条 RUNNING 记录
     4. 异步调用 Python /api/compute/backtest（HTTP 超时时间 5 分钟）
     5. 收到响应后，写入 quant_backtest_report 表
        （写入时：JSON 数据较大，按需裁剪；或者存文件系统，此处只存路径）
     6. 更新 quant_backtest.status = SUCCESS
     7. 前端轮询或 SSE 推送状态变化
    ↓
用户在前端查看报告
    ↓
Java 从 quant_backtest + quant_backtest_report 读取数据
返回给前端渲染（图表用 ECharts，表格用 DataTables）
```

**关于大数据量处理**：

- 全 A 股 3 年日线 = 约 3000 只 × 750 交易日 ≈ 225 万条 bar
- 单只股票的日线 bar JSON 体积约 50-80 bytes（含 date/open/high/low/close/volume）
- 全量 JSON 体积 ≈ 225 万 × 60 bytes ≈ **135 MB** → **单次 HTTP 请求携带 135 MB 不合理**

**处理方案**：V2.1 阶段默认限制回测选股范围为 **CUSTOM（最多 50 只）**。50 只 × 750 交易日 = 37500 bar ≈ 2.25 MB，完全合理。

全市场回测场景需要：
- Java 与 Python 建立直接 SQLite 访问（打破"Python 不直连 DB"的约束），或者
- 引入文件模式（Java 先写 Parquet 文件，Python 读取）

这两个方案都放入 V2.2 扩展。**V2.1 明确不支持全市场回测。**
