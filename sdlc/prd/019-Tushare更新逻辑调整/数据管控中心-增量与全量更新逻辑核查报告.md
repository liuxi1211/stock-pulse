## 三、问题汇总

### 3.1 增量/全量逻辑完整性问题

| 严重程度 | 问题描述 | 影响表 | 建议 |
|----------|----------|--------|------|
| 🔴 高 | **真正实现增量的表只有 2 张**（daily_quote, adj_factor），其余 23 张表要么无增量要么增量=全量 | 23 张表 | 按优先级逐步实现真正的增量逻辑 |
| 🟠 中 | **INDEX_DAILY 增量逻辑错误**：增量仅拉取当天，非从最新日期开始 | index_daily | 修复为从 MAX(trade_date)+1 开始 |
| 🟠 中 | **日频表（daily_basic/moneyflow/top_list 等）无全量能力**：全量重建也只拉取当天，历史数据缺失 | 6 张日频表 | 实现按日期范围循环补数的全量逻辑 |
| 🟡 低 | **财务报表增量窗口过大**：用固定 1 年窗口代替精准增量，浪费 API 额度 | 6 张财务表 | 优化为从最新 ann_date / end_date 开始 |
| 🟡 低 | **TRADE_CAL 增量=全量**：都从 30 年前开始拉取 | trade_cal | 增量改为从 MAX(cal_date) 开始 |
| 🟡 低 | **INDEX_WEIGHT 增量=全量**：都从 5 年前开始拉取 | index_weight | 增量改为从最新 trade_date 开始 |

### 3.2 性能问题

| 严重程度 | 问题描述 | 影响范围 | 建议 |
|----------|----------|----------|------|
| 🔴 高 | **deleteBatchByKeys 使用 row-value IN 语法，SQLite 不兼容** | 所有使用 deleteBatchByKeys 的表（daily_quote, adj_factor, income 等） | 改为 OR 连接条件，如 `(ts_code='A' AND trade_date='B') OR ...` |
| 🟠 中 | **delete+insert 无事务保护**：delete 成功但 insert 失败会导致数据丢失 | 所有表的 saveBatch 方法 | 添加 `@Transactional` 保证原子性 |
| 🟠 中 | **全量重建无回滚机制**：先 truncate 再拉取，中途失败表就空了 | 所有表 | 改用"临时表 + RENAME"或"先插后删"策略 |
| 🟠 中 | **财务报表全量重建极慢**：5000+ 只股票 × 30 年数据，且串行执行 | 6 张财务表 | 增加并发度，或使用 Tushare 批量接口 |
| 🟡 低 | **缺少显式索引检查**：需确认各表主键/唯一索引是否正确 | 所有表 | 核查建表 SQL，确保业务键有唯一索引 |
| 🟡 低 | **部分表使用 MyBatis-Plus selectOne + orderBy + LIMIT 取最新**，不如 MAX() 高效 | daily_quote, adj_factor 的单股查询 | 统一用 MAX() 聚合查询 |

### 3.3 逻辑错误

| 严重程度 | 问题描述 | 位置 | 建议 |
|----------|----------|------|------|
| 🔴 高 | **INDEX_DAILY 增量起始日期错误**：`start = isFull ? fullStart : today`，增量只拉当天 | [DataInitServiceImpl.java:315](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/DataInitServiceImpl.java#L315-L315) | 改为先查 MAX(trade_date)，从 +1 天开始 |
| 🟠 中 | **SW_INDUSTRY 全量重建 truncate 了 sw_industry_member，但 fetchAndSaveAllMembers 是 upsert 语义** | [DataInitServiceImpl.java:512-514](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/DataInitServiceImpl.java#L512-L514) | 语义一致，无实际问题，但需确认 sw_industry_member 的 truncate 是否必要 |
| 🟡 低 | **Income 等财务表的去重逻辑在 Java 层做**：按业务键去重保留 ann_date 最新的，依赖 Tushare 返回顺序 | [IncomeServiceImpl.java:60-68](file:///d:/lcProject/stock-pulse/stock-watcher/src/main/java/com/arthur/stock/service/impl/IncomeServiceImpl.java#L60-L68) | 可接受，但建议在 SQL 层也加唯一索引兜底 |

### 3.4 事务与一致性

| 场景 | 事务状态 | 风险 |
|------|----------|------|
| 单批次 delete+insert | ❌ 无事务（大部分表） | delete 成功 insert 失败 → 数据丢失 |
| 单批次 delete+insert | ✅ 有事务（margin/hk_hold/block_trade/top_list/moneyflow/trade_cal） | 安全 |
| 全量重建（truncate + 全量拉取） | ❌ 无事务 | 中途失败 → 表数据为空 |
| 并发多只股票写入 | ❌ 无跨股票事务 | 部分成功部分失败 → 数据不一致 |

## 四、执行流程详解

### 4.1 增量更新流程（incrementalUpdate）

```
调用 incrementalUpdate(tableCode, operator)
    │
    ├─ 参数校验（InitStep.fromCode）
    ├─ 尝试获取任务锁（TaskProgressCache.tryAcquireLock）
    ├─ 创建 DataPullLog（状态=RUNNING）
    ├─ 初始化任务进度
    └─ 异步执行 doIncrementalUpdate(step, taskId)
            │
            ├─ updateTaskRunning("增量拉取: xxx")
            ├─ executeSingleStep(step, taskId, isFull=false)  ← 核心执行
            │       │
            │       └─ 按 step 分发到各 Service 的 fetchAndSaveXxx
            │
            ├─ [取消检查] isCancelled → 标记 CANCELLED
            ├─ finishPullLog（SUCCESS / FAILED）
            ├─ runQualityCheck(step)  ← 自动触发质量检测
            └─ updateTaskSuccess / updateTaskFailed
```

### 4.2 全量重建流程（fullRebuild）

```
调用 fullRebuild(tableCode, operator)
    │
    ├─ 参数校验、获取锁、创建日志（同增量）
    └─ 异步执行 doFullRebuild(step, taskId)
            │
            ├─ updateTaskRunning("重建表: xxx")
            ├─ rebuildTable(step)  ← 先清空表
            │       └─ truncateTable(tableName)
            │              ├─ MySQL: TRUNCATE TABLE xxx
            │              └─ SQLite: DELETE FROM xxx + 重置自增
            │
            ├─ updateTaskRunning("全量拉取: xxx")
            ├─ executeSingleStep(step, taskId, isFull=true)  ← 核心执行
            │
            ├─ [取消检查]
            ├─ finishPullLog
            ├─ runQualityCheck(step)
            └─ updateTaskSuccess / updateTaskFailed
```

### 4.3 逐股票并发执行（executePerStockStep）

```
executePerStockStep(step, taskId, task)
    │
    ├─ 获取在市股票列表（stockBasicService.queryLocal）
    ├─ 计算并发度：min(20, max(4, total/50))  ← 4~20 自适应
    ├─ 创建虚拟线程池（newFixedThreadPool + VirtualThread factory）
    │
    ├─ 为每只股票创建 CompletableFuture
    │       ├─ [取消检查] 已取消则跳过
    │       ├─ task.execute(tsCode)  ← 调用具体 Service
    │       └─ 成功/失败计数
    │
    ├─ allOf().join() 等待全部完成
    └─ 返回 StepStats（total / success / fail）
```

## 六、数据库操作模式

### 6.1 写入模式：Delete + Insert（幂等 Upsert）

所有表均采用"先按业务键删除，再批量插入"的模式，保证幂等性：
